# 功能规格：新增 Text 排版组件

**功能分支**：`20260323-add-text-component`  
**创建时间**：2026-03-23  
**状态**：草稿  
**输入**：用户描述："接下来实现一下 Text 组件，主要是处理文本渲染的。然后支持几个能力，通过props 来展开说说：第一个是variant，定义排版样式类型，默认 plain，正常渲染，然后是span标签，其次支持标题，h1-h5的标题选择，通过字体大小和行高区分，用div标签渲染，然后也支持caption，字体会小一点，然后颜色会稍微浅一点，加点透明度吧，然后是body，跟plain一样正常大小渲染，但是是div标签。第二个是支持斜体、字重、下划线、划线这四个props，你帮我起一下名，别等会又用中文了，然后是boolean，可以叠加使用。然后再支持lineClamp这个prop，只是设置最大行数，超出则显示省略，没有设置这无限长。然后默认的节点props也支持上，然后透传给渲染节点即可。"
**语言要求**：本文件及同级 `/specs/` 文档正文必须使用简体中文；代码标识符、命令、文件路径、协议字段和第三方 API 名称可保留原文。

## Clarifications

### Session 2026-03-24

- Q: `Text` 新增的 `color` / `background` 是否允许任意自定义色值？ → A: 不允许任意自定义色值；两者都直接消费共享色卡 token，并通过预定义 union 暴露给消费方。
- Q: `Text` 的 `color` / `background` 应该暴露什么粒度的颜色选项？ → A: 参考 Tailwind 色卡，维护 26 个颜色家族、每个家族 11 个色阶的共享色卡 token；`color` 与 `background` 对外都只暴露 26 个颜色家族名的 union，由系统再根据用途映射到较深或较浅的实际色阶。
- Q: `Text` 的 `color` / `background` 在浅色与深色主题下如何映射具体色阶？ → A: 按主题自动切换色阶映射；浅色主题使用较深文字色与较浅背景色，深色主题改为较浅文字色与较深背景色，继续保持同一颜色家族语义。

## 用户场景与测试（必填）

### 用户故事 1 - 用统一入口渲染常见文本层级（优先级：P1）

作为消费组件库的应用开发者，我希望通过一个 `Text` 组件就能覆盖普通文本、正文块、说明文字和五级标题样式，这样我在页面里排版时不需要为每一种文本层级寻找不同组件或重复写样式。

**为什么是这个优先级**：这是 Text 组件存在的核心价值。如果基础层级模型不清晰，后续的装饰样式和截断能力都失去复用意义。

**独立测试**：仅使用 `Text` 的公开文档或预览，即可分别创建 `plain`、`body`、`caption`、`h1`、`h2`、`h3`、`h4`、`h5` 八种代表性文本，并确认默认值、视觉层级和默认渲染节点都可预测。

**验收场景**：

1. **假如** 开发者没有显式传入 `variant`，**当** 渲染一段普通文本，**那么** 组件会按 `plain` 的默认样式显示，并使用默认的行内文本节点。
2. **假如** 开发者需要渲染正文段落或说明文字，**当** 分别选择 `body` 或 `caption`，**那么** 两者都以块级文本方式呈现，其中 `caption` 比默认正文更小且视觉强调更弱。
3. **假如** 开发者需要建立页面中的标题层级，**当** 选择 `h1` 到 `h5` 中任一等级，**那么** 组件会默认渲染为对应的原生标题节点，并按照等级差异提供可区分的字体大小与行高。

---

### 用户故事 2 - 组合文本装饰与强调样式（优先级：P2）

作为页面开发者，我希望同一段文本可以通过简单的布尔字段叠加斜体、加粗、下划线和删除线，也能通过受控的 `color` 和 `background` 字段切换文字色与高亮底色，这样我不需要再通过额外 class 或手写样式来表达基础强调关系。

**为什么是这个优先级**：文本强调是高频需求，但它建立在基础文本层级已经可用之上，因此优先级次于基础 `variant` 模型。

**独立测试**：分别验证 `italic`、`bold`、`underline`、`strikethrough`、`color`、`background` 的单独使用和组合使用，确认每个字段都能独立生效，且不会阻断其他字段的效果。

**验收场景**：

1. **假如** 开发者只需要强调语气，**当** 为任意 `variant` 的文本启用 `italic` 或 `bold`，**那么** 组件会在保留原有文本层级的前提下增加对应的强调效果。
2. **假如** 开发者需要同时表达强调与修订，**当** 同时启用 `bold`、`underline` 和 `strikethrough` 中的多个字段，**那么** 这些效果可以叠加显示，而不是互相覆盖或失效。
3. **假如** 开发者需要强调关键词或做局部高亮，**当** 为任意 `variant` 设置 `color`、`background` 或两者同时设置，**那么** 组件会使用预定义共享色卡中的合法值切换文字色和背景色，而不是接受任意自定义颜色字符串。

---

### 用户故事 3 - 控制长文本和透传节点能力（优先级：P3）

作为组件使用方，我希望 Text 既能限制长文本的显示行数，也能继续接受常见节点属性并原样透传到最终渲染节点，这样我既能处理列表摘要，也不会失去 `className`、`id`、`aria-*`、`data-*` 等基础接入能力。

**为什么是这个优先级**：长文本截断和节点透传直接决定 Text 组件能否进入真实页面，但它们建立在基础排版与样式能力已经定义清楚的前提上。

**独立测试**：创建包含超长文本的 `Text` 示例，分别验证未设置 `lineClamp` 和设置正整数 `lineClamp` 的行为；同时透传若干常见节点属性并确认它们落到最终渲染节点上。

**验收场景**：

1. **假如** 开发者未设置 `lineClamp`，**当** 渲染多行长文本，**那么** 文本会保持自然延展，不会被默认截断。
2. **假如** 开发者将 `lineClamp` 设置为正整数，**当** 文本内容超过该行数，**那么** 组件只显示指定的最大行数，并以省略形式提示仍有未显示内容。
3. **假如** 开发者为 Text 传入 `className`、`id`、`aria-label` 或 `data-*` 等节点属性，**当** 组件渲染后，**那么** 这些属性会继续存在于最终渲染节点上。

### 边界情况

- 当 `lineClamp` 未设置时，长文本必须保持自然换行和完整显示，不得默认截断。
- 当 `lineClamp` 传入非正整数时，组件不得产生“显示为空”或不可读的异常结果，且必须保持可预测行为。
- 当 `italic`、`bold`、`underline`、`strikethrough` 同时启用时，文本样式必须可叠加，不得因为组合导致其中某一项静默失效。
- 当 `variant="caption"` 与多个强调字段组合使用时，说明文字的较低视觉强调仍需保持可读，不得因为样式叠加失去辨识度。
- 当 `color`、`background` 与装饰字段叠加使用时，文本必须仍然保持可读，且不得因为接受任意自由输入而出现未文档化的颜色来源。
- 当开发者为 `color` 或 `background` 传入某个颜色家族时，组件必须从该家族的既定色阶中自动选择实际文字色或背景色，而不是要求消费方直接指定 `50` 到 `950` 的具体色阶。
- 当主题从浅色切换到深色，或从深色切换到浅色时，同一颜色家族的 `color` / `background` 必须自动切换到适合当前主题的深浅色阶，并保持高亮语义连续。
- 当 `variant` 为 `h1` 到 `h5` 时，组件必须同时保持可区分的视觉标题层级与对应原生标题节点，不得出现样式层级与节点语义错位。
- 当仅使用键盘或辅助技术浏览页面时，透传到最终节点的 `aria-*`、`tabIndex` 等属性必须继续生效。
- 当文本处于不同主题或 token 变体下时，`caption` 的弱化强调和其他层级的可读性必须保持一致规则，而不是依赖局部例外。

## 需求（必填）

### 功能需求

- **FR-001**：组件库必须提供统一的文本渲染入口 `Text`，并定义与其对应的公开 props 契约。
- **FR-002**：`Text` 必须支持以下 `variant`：`plain`、`body`、`caption`、`h1`、`h2`、`h3`、`h4`、`h5`。
- **FR-003**：`variant` 的默认值必须为 `plain`。
- **FR-004**：`plain` 必须作为普通行内文本样式，并默认渲染为 `span` 节点。
- **FR-005**：`body` 必须保持与 `plain` 相同的基础字号层级，但默认渲染为 `div` 节点，以表达块级正文内容。
- **FR-006**：`caption` 必须默认渲染为 `div` 节点，并在视觉上比 `plain` / `body` 更小且强调更弱。
- **FR-007**：`h1` 到 `h5` 必须提供五级可区分的视觉标题层级，并通过字体大小和行高差异体现等级关系；这些 `variant` 默认渲染为对应的原生 `h1` 到 `h5` 节点。
- **FR-008**：系统必须明确说明 `h1` 到 `h5` 默认同时承担视觉标题层级与原生标题节点语义；本期不额外提供 `as` 等根节点覆盖能力。
- **FR-009**：`Text` 必须支持四个可独立启用且可叠加的布尔样式字段：`italic`、`bold`、`underline`、`strikethrough`。
- **FR-010**：启用任一布尔样式字段时，系统不得改变当前 `variant` 的层级定义、默认渲染节点或其他已启用布尔样式字段的效果。
- **FR-011**：`Text` 必须支持 `color` 与 `background` 两个公开 props，分别用于切换文字颜色与高亮背景颜色。
- **FR-012**：`color` 与 `background` 都必须基于共享色卡 token 暴露预定义 union，不得接受任意自定义颜色字符串、十六进制色值或运行时自由输入。
- **FR-013**：系统必须将 `color` 与 `background` 的公开取值限定为 26 个颜色家族名的 union，而不是直接暴露 `50` 到 `950` 的色阶编号；该 union 包括 `red`、`orange`、`amber`、`yellow`、`lime`、`green`、`emerald`、`teal`、`cyan`、`sky`、`blue`、`indigo`、`violet`、`purple`、`fuchsia`、`pink`、`rose`、`slate`、`gray`、`zinc`、`neutral`、`stone`、`taupe`、`mauve`、`mist`、`olive`。
- **FR-014**：系统必须在内部维护一套与 Tailwind 色卡结构对齐的共享色卡 token，至少覆盖 26 个颜色家族、每个家族 11 个色阶，用于支撑 `color` 与 `background` 的映射。
- **FR-015**：系统必须明确记录 `color` 与 `background` 的合法取值来源、命名方式和面向消费方的公开说明。
- **FR-016**：系统必须基于当前主题为 `color` 与 `background` 自动选择实际色阶：浅色主题使用较深文字色与较浅背景色，深色主题使用较浅文字色与较深背景色。
- **FR-017**：同一颜色家族在不同主题下的自动映射必须保持颜色语义连续，不得在主题切换后落入无关颜色家族或失去高亮意图。
- **FR-018**：`Text` 必须支持 `lineClamp`，用于定义文本允许显示的最大行数。
- **FR-019**：当 `lineClamp` 设置为正整数且内容超出上限时，组件必须只显示指定行数，并对超出内容提供省略提示。
- **FR-020**：当 `lineClamp` 未设置时，组件必须允许文本按自然长度完整显示。
- **FR-021**：系统必须对非正整数的 `lineClamp` 输入提供一致且文档化的处理方式，并确保不会生成不可读结果。
- **FR-022**：`Text` 必须接受并透传常见节点属性到最终渲染节点，包括但不限于 `className`、`style`、`id`、`title`、`role`、`tabIndex`、`aria-*`、`data-*` 和事件处理字段。
- **FR-023**：透传的节点属性必须作用于实际渲染节点，而不是中间包装层。
- **FR-024**：系统必须为 `Text` 定义清晰的公开 API 说明，包括各 `variant` 的预期用途、默认节点、默认值以及布尔样式字段、`color` / `background` 与 `lineClamp` 的使用边界。
- **FR-025**：系统必须提供可评审的预览覆盖，至少包含全部 `variant`、四个布尔样式字段、`color` / `background` 的单独与组合状态、未截断与截断状态，以及透传节点属性的代表性示例。
- **FR-026**：系统必须满足明确的无障碍要求，包括文本内容可读、节点透传后的辅助技术属性有效，以及标题 `variant` 的原生 heading 语义与视觉层级保持一致。
- **FR-027**：系统必须识别是复用现有排版与颜色 token，还是需要新增文本层级 token，并明确记录 `caption` 的弱化强调来源。
- **FR-028**：系统必须为 `Text` 的 `color` 与 `background` 明确复用或新增共享色卡 token，并保证这些 token 可同时用于组件实现、预览和文档说明。

### 无障碍与 UI 契约（UI 工作必填）

- **用户角色 / 参与方**：消费组件库的应用开发者、定义排版层级的设计师、需要阅读内容的终端用户、仅键盘用户、依赖辅助技术的用户、维护组件规范的库维护者。
- **交互模型**：以静态文本渲染和节点属性透传为主，支持通过公开字段切换视觉层级、叠加强调样式，并对长文本进行最大行数限制。
- **需要覆盖的状态**：`plain`、`body`、`caption`、`h1`、`h2`、`h3`、`h4`、`h5`、`italic`、`bold`、`underline`、`strikethrough`、`color`、`background`、多字段组合、未设置 `lineClamp`、设置 `lineClamp`、长文本、省略显示、透传 `aria-*` / `data-*` / `className` 的代表性场景。
- **主题 / token 影响**：优先复用现有字号、行高、颜色和文字强调规则；若现有规则不足以支撑 `caption`、五级标题差异或 `color` / `background` 的共享色卡映射，必须明确记录新增或调整的排版与色卡 token 需求。

### 关键实体（如功能涉及数据则填写）

- **Text 公开契约**：消费方可见的组件名称、默认值、支持的 `variant`、布尔样式字段、`lineClamp` 规则和节点属性透传边界。
- **Text Variant**：定义文本的视觉层级与默认渲染节点，包括 `plain`、`body`、`caption`、`h1`、`h2`、`h3`、`h4`、`h5`。
- **文本装饰集合**：由 `italic`、`bold`、`underline`、`strikethrough`、`color`、`background` 组成的可叠加强调能力集合。
- **共享色卡 token**：供 `Text` 的 `color` 与 `background` 复用的预定义颜色家族和色阶集合，以 union 方式对消费方暴露；颜色家族包括 `red`、`orange`、`amber`、`yellow`、`lime`、`green`、`emerald`、`teal`、`cyan`、`sky`、`blue`、`indigo`、`violet`、`purple`、`fuchsia`、`pink`、`rose`、`slate`、`gray`、`zinc`、`neutral`、`stone`、`taupe`、`mauve`、`mist`、`olive`。
- **截断规则**：由 `lineClamp` 决定的最大可见行数和超出后的省略表现。
- **节点透传属性**：消费方向最终渲染节点附加的标识、无障碍、样式和事件属性集合。

## 依赖与假设

### 依赖

- 本功能依赖组件库已有的基础排版和颜色规则，能够支撑普通正文、弱化说明文字和五级视觉标题层级。
- 本功能依赖存在一个可供设计师和开发者共同评审的预览入口，用于确认所有 `variant`、样式组合和长文本状态。
- 本功能依赖组件库现有的公开入口机制，能够将新增 `Text` 能力纳入统一的消费面。

### 假设

- 本期不引入 `as` 或其他多态根节点能力；因此 `h1` 到 `h5` 统一默认采用对应原生标题节点，`plain` 维持 `span`，`body` / `caption` 维持 `div`。
- 本期 `caption` 的“更浅”表现默认理解为相对 `plain` / `body` 更弱的视觉强调，可通过透明度或等价的弱化方式实现，但必须保持可读。
- 本期四个布尔样式字段统一命名为 `italic`、`bold`、`underline`、`strikethrough`，且均为可叠加布尔能力。
- 本期 `color` 与 `background` 统一采用共享色卡 token 的预定义 union，而不是接受任意自由输入的颜色值。
- 本期 `color` 与 `background` 对外都只暴露 26 个颜色家族名；`50` 到 `950` 的具体色阶由系统基于用途决定，不直接暴露给消费方。
- 本期 `color` 与 `background` 的具体色阶会随主题自动切换：浅色主题偏“深字浅底”，深色主题偏“浅字深底”。
- 本期 `lineClamp` 仅负责限制最大显示行数和超出提示，不扩展到“展开 / 收起”等额外交互。
- 本期节点透传能力以最终渲染节点为准，不额外引入新的包装节点来承接常见属性。

## 成功标准（必填）

### 可度量结果

- **SC-001**：100% 的公开文本层级都可以通过单一 `Text` 入口完成，并能在规格说明或预览中找到对应示例。
- **SC-002**：90% 以上的消费方能够在 30 秒内根据说明判断某段内容应使用 `plain`、`body`、`caption` 还是 `h1` 到 `h5` 中的某个层级。
- **SC-003**：100% 的四个布尔样式字段都具备单独示例与组合示例，且组合示例中不存在某一字段静默失效的情况。
- **SC-004**：100% 的长文本评审样例在未设置 `lineClamp` 时完整显示，在设置有效 `lineClamp` 时按指定行数截断并给出省略提示。
- **SC-005**：100% 的代表性节点属性透传样例都能在最终渲染节点上观察到预期属性，不需要额外包裹层才能完成接入。
- **SC-006**：100% 的视觉标题层级示例都能在首次评审中被清晰区分，不会出现相邻等级难以辨认的情况。
- **SC-007**：100% 的 `caption` 示例在不同主题或 token 变体下都保持“弱于正文但仍清晰可读”的评审结论。
- **SC-008**：100% 的 `color` / `background` 公开示例都仅使用 26 个颜色家族名中的合法取值，不接受额外自由颜色输入。
