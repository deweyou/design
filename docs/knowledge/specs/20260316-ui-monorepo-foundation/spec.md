# 功能规格：UI Monorepo 基础建设

**功能分支**： `20260316-ui-monorepo-foundation`  
**创建时间**： 2026-03-16  
**状态**： 草稿  
**输入**： 用户描述： "定义一个基于 Vite+ 的 React + TypeScript + Less UI 组件库 v1 monorepo 架构，包括 utils、hooks、styles、components 四类 packages，website 与 storybook 两个 apps，以及一个以 TS token 为源、可输出 CSS 主题文件的受控主题系统；组件样式使用 CSS Modules，并采用显式、文档驱动的全局样式导入方式。"

## 用户场景与测试（必填）

### 用户故事 1 - 明确 package 与 app 边界（优先级：P1）

作为 UI 库维护者，我希望 monorepo 能清晰定义各个 package 与 app 的职责边界，从而后续在组件、样式、hooks 和文档上的工作都能在不重构项目结构的前提下继续推进。

**为什么是这个优先级**：这是后续所有组件、主题和文档工作的基础。如果没有统一边界，仓库很快就会漂移，后续功能也会更难组织和发布。

**独立测试**：通过评审仓库结构、package 与 app 职责以及依赖方向文档，确认每一类主要关注点都有唯一明确的归属。

**验收场景**：

1. **假如** 维护者查看 v1 架构，**当** 其检查已定义的 packages 和 apps 时，**那么** 能明确判断 utilities、hooks、styles、components、website 内容与 Storybook 工作分别应该放在哪里。
2. **假如** 维护者准备新增功能，**当** 其根据职责对工作进行分类时，**那么** 能在不移动既有基础设施的前提下判断出正确的 package 或 app。

---

### 用户故事 2 - 建立受控主题系统（优先级：P2）

作为消费该组件库的应用开发者，我希望库能够提供稳定的默认设计语言，并暴露一小块清晰定义的主题覆盖面，从而我能切换明暗主题、调整品牌色，而不需要重新设计整套组件系统。

**为什么是这个优先级**：该库本身是有设计立场的，因此主题边界必须尽早确定，才能保证后续组件 API、文档和视觉决策保持一致。

**独立测试**：通过查看主题模型文档，确认只有被批准的公开主题色 token 可供用户配置，且其他设计 token 默认保持内部私有。

**验收场景**：

1. **假如** 消费者不做任何自定义主题覆盖，**当** 其导入文档中说明的全局样式时，**那么** 默认设计语言会生效，且不需要额外配置。
2. **假如** 消费者希望做品牌适配，**当** 其覆盖文档中说明的公开主题色面时，**那么** 组件的品牌相关颜色会变化，而组件的形状、间距、字号体系和动效语言仍保持库的默认规则。

---

### 用户故事 3 - 分离公开文档与内部开发工具（优先级：P3）

作为 UI 库维护者，我希望 website 和 Storybook 的职责保持分离，从而对外网站专注于文档与精选示例，而 Storybook 则继续作为内部状态覆盖与实现验证工具。

**为什么是这个优先级**：如果职责不清晰，同一份示例和文档会在两个地方重复维护，既增加成本，也会造成指导不一致。

**独立测试**：通过查看每个 app 的文档定位，确认公开文档、精选示例和探索式状态验证分别被放到了正确的表面。

**验收场景**：

1. **假如** 维护者准备面向用户的说明内容，**当** 其决定发布位置时，**那么** 官方文档、安装指引、设计理念和精选 demo 都应归属于 website。
2. **假如** 维护者需要检查边界状态或不稳定状态，**当** 其决定使用哪个表面时，**那么** 探索型 stories、全量状态矩阵和内部验证都应归属于 Storybook。

### 边界情况

- 当消费者导入组件但没有导入必需的全局样式时，应如何表现？
- 当消费者将公开品牌色覆盖为与默认文本或背景对比度过低的值时，系统应如何表现？
- 仅使用键盘的用户在 website 中浏览精选 demo、并在明暗主题间切换时，会发生什么？
- 当未定义任何自定义主题，仅依赖默认内部 token 映射时，库应如何表现？

## 需求（必填）

### 功能需求

- **FR-001**：系统必须将 monorepo 重新组织为 `packages` 与 `apps` 两类关注点，并分别为 utilities、可复用 React hooks、styles、components、website 和 Storybook 提供明确归属。
- **FR-002**：系统必须将 `utils` 定义为框架无关工具逻辑的归属位置，并为可复用 React hooks 单独设立 `hooks` package。
- **FR-003**：系统必须将 `styles` 定义为设计 token 源、主题定义、reset / base 全局样式，以及面向消费者的主题输出的归属位置。
- **FR-004**：系统必须将 `components` 定义为可复用 UI 组件及其组件本地样式的归属位置。
- **FR-005**：系统必须将 website 定义为对外安装文档、设计原则、主题文档、精选组件示例和有限交互 playground 的公开表面。
- **FR-006**：系统必须将 Storybook 定义为内部组件状态覆盖、边界状态验证、探索性 stories 与实现导向评审的内部表面。
- **FR-007**：系统必须要求消费者显式导入文档中说明的全局样式入口，而不是依赖组件 package 静默注入全局样式。
- **FR-008**：系统必须提供无需额外主题覆盖即可工作的默认设计语言，只要消费者已导入文档化的全局样式。
- **FR-009**：系统必须将明亮主题和暗色主题作为一等主题模式支持。
- **FR-010**：系统必须定义一个小而明确的公开主题面，仅包含用于品牌表达和交互强调的一组关键颜色 token。
- **FR-011**：系统必须默认将所有非公开 token 视为内部实现，包括结构、间距、圆角、字号体系、动效和其他设计语言 token。
- **FR-012**：系统必须识别并文档化初始公开主题色面，包括品牌背景、品牌交互态、品牌面上的文字色、焦点强调和链接强调。
- **FR-013**：系统必须产出主题样式文件，包括独立的亮色主题文件、暗色主题文件，以及一个用于标准消费者接入的默认主题入口。
- **FR-013a**：styles package 必须从 `dist/` 发布 CSS 和 Less 主题资产，而不是依赖源码目录导出。
- **FR-014**：系统必须定义组件本地样式与全局主题 / reset 样式分离，并由每个组件自行管理其本地样式。
- **FR-015**：系统必须要求组件本地样式使用 CSS Modules，使组件实现样式默认保持作用域隔离。
- **FR-015a**：系统必须按组件目录组织组件源码，每个组件目录下放置共置的 `index.tsx` 与 `index.module.less`。
- **FR-016**：系统必须定义一致的组件自定义契约，其中包括供消费者覆盖的根级 `className` 入口。
- **FR-017**：系统必须允许组件暴露有限数量的组件级样式变量，以支持局部视觉调整，但不能因此打开整套设计系统的完全重塑能力。
- **FR-018**：系统必须明确 CSS Module 的内部类名属于实现细节，不构成公开自定义契约的一部分。
- **FR-019**：系统必须定义每个受影响 package 的公开 API 面，包括命名、package 用途和预期依赖方向。
- **FR-019a**：所有可复用 workspace packages 必须发布在 `@deweyou-ui/*` scope 下，所有内部 app 或 package 消费方都必须引用这些带 scope 的包名，而不是无 scope 别名。
- **FR-020**：系统必须文档化依赖方向，确保 utilities 和 styles 作为底层基础存在，hooks 独立于通用 utilities，components 只消费低层 packages，而不是反向依赖。
- **FR-021**：系统必须在 website、Storybook 示例和该基础层所覆盖的组件状态中满足明确的无障碍要求，包括语义结构、焦点处理、键盘交互和辅助技术标签。
- **FR-022**：系统必须说明是否引入、复用，或明确不需要新的设计 token、主题 hook 或视觉 primitives。
- **FR-023**：系统必须在 website 中提供主要组件示例、主题行为以及代表性边界 / 禁用状态的预览覆盖，为后续组件建设打基础。

### 无障碍与 UI 契约（必填）

- **用户角色 / 参与方**：UI 库维护者、消费 packages 的应用开发者、使用键盘评审 demo 的用户、与 website 示例交互的屏幕阅读器用户、验证主题面的设计师
- **交互模型**：显式导入全局样式、切换明暗主题、通过 package API 消费组件、根级 `className` 覆盖、有限组件级视觉覆盖
- **需要覆盖的状态**：默认、hover、focus-visible、active、disabled、亮色主题、暗色主题、未覆盖主题时的默认态、消费者提供品牌色覆盖
- **主题 / token 影响**：建立受控公开主题色面，同时保留更广泛的设计 token 为内部实现，并定义主题输出和默认主题入口

### 关键实体

- **Package 边界**：monorepo 中一个被文档化的职责区域，包含其用途、预期内容和允许的依赖方向。
- **App 表面**：monorepo 中一个可运行的体验，用于服务特定受众，例如公开文档或内部组件评审。
- **公开主题 Token**：可由消费者覆盖的颜色 token，用于品牌表达适配，而不改变组件系统的核心设计语言。
- **内部设计 Token**：用于维持库的设计语言和默认值，但不属于面向消费者的自定义契约。
- **组件自定义契约**：消费者可稳定依赖的组件样式入口，包括根级 `className` 和被明确允许的组件级样式变量。

## 假设

- 消费者愿意在初始接入组件库时导入文档中说明的全局样式入口。
- 初始公开主题面刻意保持较小并聚焦颜色，因为该库有明确设计立场，并不打算在 v1 暴露整套设计语言的完全重塑能力。
- 即便 website 成为对外主要文档与 demo 表面，Storybook 仍作为内部评审面保有价值。

## 成功标准（必填）

### 可度量结果

- **SC-001**：维护者能够将初始 monorepo 中 100% 的关注点归类到唯一的 package 或 app 职责中，且不存在重叠。
- **SC-002**：消费者能够只通过文档中的全局样式导入与 package 使用说明，一次性完成初始组件库接入，而不依赖未文档化行为。
- **SC-003**：消费者能够在不修改组件结构或依赖未文档化内部实现的情况下，切换明暗主题并应用被批准的品牌色覆盖。
- **SC-004**：初始公开主题面的 100% 内容都已文档化，包括其用途以及公开 / 内部自定义边界。
- **SC-005**：官方 website 文档和内部 Storybook 使用方式可分别评审，而无需为同一组件使用流程维护两套冗长说明。
