# Phase 0 研究：包构建与发布产物治理

## 决策 1：将 Vite+ 统一构建约定设为默认，按产物需求识别少数例外包

**Decision**：把 Vite+ 的统一构建约定作为 `packages/` 下拟发布包的默认方案。`packages/hooks` 与 `packages/utils` 以默认约定为目标；`packages/components`、`packages/icons`、`packages/styles` 仅在默认约定无法满足公开产物要求时保留专用构建行为。

**Rationale**：

- 仓库治理已经明确“非必要不保留包级专用构建配置”，因此规划必须先判断默认约定能否覆盖，再决定是否保留例外。
- 当前 `packages/utils/tsdown.config.ts` 与 `packages/styles/tsdown.config.ts` 只承载最小包装逻辑；其中 `utils` 的配置通过 `vite.config.ts` 间接引用，说明它更适合作为默认约定的一部分而不是独立配置面。
- `packages/components` 需要同时处理根入口兼容、组件子路径导出和组件样式交付，`packages/icons` 需要生成大量图标子路径入口，`packages/styles` 需要复制 CSS/Less 资产并重写发布清单；这三类需求都超出普通单入口包的默认范围。

**Alternatives considered**：

- 保留当前所有包级配置不动：会继续增加维护面，违背最新治理原则。
- 强制所有包完全共用同一套零例外配置：无法覆盖多入口、静态资产复制和组件样式交付的现实需求。

## 决策 2：组件入口契约同时保证根入口兼容和单组件样式自带

**Decision**：`@deweyou-ui/components` 的公开消费面同时保留根入口和每个公开组件的独立子路径入口；单组件入口必须自带完成使用所需的样式契约，不再要求消费者额外导入组件专属样式入口。

**Rationale**：

- 规格要求单组件可直接导入且无需手动补样式，同时又要求保留现有根入口兼容消费方式，因此入口契约必须双轨并存。
- 当前组件源代码已经以 `src/<component>/index.tsx` 和 `src/index.ts` 组织，天然具备为公开组件生成独立入口的边界。
- 将样式交付纳入入口契约，可以把“组件是否可直接用”从文档提醒变成构建结果的一部分，减少消费端心智负担。

**Alternatives considered**：

- 只保留根入口并继续要求手动导入样式：不能满足单组件消费目标。
- 仅提供子路径入口并移除根入口：会制造不必要的兼容性破坏。

## 决策 3：所有运行时依赖 React 的发布包统一改为宿主安装契约

**Decision**：凡是运行时依赖 `react` 或 `react-dom` 的拟发布包，都将这两项依赖视为宿主负责安装的依赖契约；仅开发期需要的类型、测试和构建依赖保留在开发依赖中。

**Rationale**：

- 组件库包需要与宿主应用共享同一 React 运行时，否则容易出现重复运行时和安装责任不清的问题。
- 现有 `packages/components`、`packages/hooks`、`packages/icons` 都直接依赖 React 运行时；将其改为宿主安装契约后，外部消费者的安装责任更清晰。
- 该分类规则可被统一应用到未来新增包，降低发布前人工判断成本。

**Alternatives considered**：

- 继续把 React 运行时放在普通依赖中：会让包自身隐式携带运行时，削弱宿主安装契约。
- 对不同包采用不同 React 依赖策略：会破坏仓库级一致性。

## 决策 4：发布清单必须在对外产物中解析内部占位版本

**Decision**：发布产物中的 `package.json` 必须把 `workspace:*`、`catalog:` 和内部包引用解析为对外可安装的明确 semver 范围；发布前检查以最终清单而不是源码清单为准。

**Rationale**：

- 工作区占位版本只适用于 monorepo 内协作，不能直接暴露给 npm 消费者。
- `packages/styles/scripts/copy-assets.mjs` 已经展示出仓库允许在构建后重写发布清单，这说明“构建产物清单”而非源码清单才是最终契约。
- 如果不把版本解析定义为发布前门禁，后续所有包的 npm 发布都将存在安装失败风险。

**Alternatives considered**：

- 直接发布源码中的 `workspace:*` 和 `catalog:`：外部宿主无法解析。
- 要求维护者手工改写每个包的版本号：容易出错，也不具备规模化治理能力。

## 决策 5：本次 Phase 1 契约文档拆为组件入口契约和发布清单契约

**Decision**：在 `contracts/` 下分别维护组件入口契约和发布清单契约，用来约束“用户如何导入组件”和“npm 消费者会看到什么依赖与导出”。

**Rationale**：

- 该功能的公开接口并不是 HTTP API，而是包导出面和包清单本身。
- 把消费契约和发布契约分开后，可以分别支持组件侧验证与发版侧审查。
- 这两类契约都能直接映射到任务拆分和验收测试。

**Alternatives considered**：

- 不写契约文档，只在 plan 中描述：后续任务和验收边界会变得模糊。
- 把所有规则塞进单一文档：会让消费侧和发布侧的职责混在一起。
