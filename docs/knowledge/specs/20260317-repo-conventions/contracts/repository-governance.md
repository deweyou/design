# 契约：仓库治理

## 目的

定义仓库规范如何被传达给贡献者、评审者和自动化系统，以及这些规则如何被执行。

## 规则与执行方式

| 规则                                                     | 主要执行方式                   | 辅助执行方式                        | 备注                                          |
| -------------------------------------------------------- | ------------------------------ | ----------------------------------- | --------------------------------------------- |
| 函数默认使用箭头函数                                     | lint 或静态规则检查            | 根级与 package `AGENTS.md`          | 允许有文档化例外                              |
| React 组件统一使用 TSX，而不是 `createElement`           | lint 或 restricted-syntax 检查 | package `AGENTS.md`、demo 示例清理  | 若有例外必须说明工具限制                      |
| 文件与目录使用小写连字符命名                             | 路径检查或评审规则             | `AGENTS.md` 指导                    | 历史文件允许渐进迁移                          |
| utilities、hooks、components 在 `src` 下使用独立单元目录 | 路径级校验                     | package `AGENTS.md`、code review    | 迁移时需避免误伤公开导入面                    |
| 测试与单元源码共置                                       | 路径检查、测试用例或评审门禁   | `AGENTS.md`                         | 顶层 `tests/` 仅保留跨单元或跨 workspace 场景 |
| 组件变更必须保留预览与无障碍评审覆盖                     | 预览更新要求和评审门禁         | `vp test`、`website` 验证、宪章检查 | 适用于新增或实质修改的组件                    |

## 贡献者预期

- 新增或实质修改的 utilities、hooks 和 components 必须遵循仓库规范。
- 如果历史产物尚未符合规范，贡献者必须明确此次是顺手对齐还是延后处理。
- 如涉及目录迁移，必须同时检查是否影响 package 的公开导入面。
- 贡献者不能以 demo 代码、临时脚本或局部工具限制为理由，引入与规范冲突的新模式，除非已有文档化例外。

## 评审者预期

- 评审时必须检查变更是否符合函数风格、组件源码格式、命名模式和测试位置规则。
- 如果目录结构调整影响可见组件行为，评审者还必须确认相关预览、无障碍覆盖和文档已同步更新。
- 对于历史代码的延后处理决定，评审者必须确认其理由成立，且不会让新代码继续扩散旧模式。

## 自动化预期

- `vp check` 应覆盖所有可以可靠自动判定的规则。
- `vp test` 应包含用于验证仓库结构和规范回归的测试。
- 自动化规则必须尽可能精确，避免对历史代码或非受治理区域产生大量误报。
