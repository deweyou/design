# 功能规格：按钮 Hover 反馈整理

**功能分支**：`20260323-button-hover-motion`  
**创建时间**：2026-03-23  
**状态**：已实现  
**输入**：按钮 hover 反馈整理：`link` 保留从左到右的下划线显现，`outlined` 取消环绕描边动画，改为默认低色度 border 并在 hover 时平滑过渡到与文字一致的颜色，同时移除 `animated` 公开属性。  
**语言要求**：本文件及同级 `/specs/` 文档正文使用简体中文；代码标识符、命令、文件路径、协议字段和第三方 API 名称可保留原文。

## Clarifications

### Session 2026-03-23

- Q: hover 反馈是否通过公开属性按需开启？ → A: 不提供公开开关，默认行为直接内建到对应 variant。
- Q: `outlined` 是否保留环绕式描边动画？ → A: 不保留，改为真实 border 的颜色过渡。

## 用户场景与测试

### 用户故事 1 - Link 下划线显现反馈（优先级：P1）

作为正在评审文本按钮交互的产品或设计人员，我希望 `link` 按钮在 hover 时通过一条从左到右显现的下划线表达交互状态，这样我能快速区分它与普通文本或静态标签。

**独立测试**：在 `link` 按钮上触发 hover，确认下划线从左向右显现，退出 hover 后可重置；当按钮同时包含 icon 和 text 时，下划线必须覆盖整条可见内容行。

**验收场景**：

1. **假如** 一个 `link` 按钮处于默认状态，**当** 指针移入交互区域，**那么** 下划线必须从左到右连续显现。
2. **假如** 用户快速移入又移出 `link` 按钮，**当** 再次 hover，**那么** 下划线必须从左侧重新开始，不得残留或叠粗。
3. **假如** 一个 `link` 按钮同时展示 icon 和 text，**当** hover 生效，**那么** 下划线必须覆盖整条内容行，而不是只贴在文字片段下方。

### 用户故事 2 - Outlined 边框颜色过渡（优先级：P2）

作为正在评审强调型按钮反馈的产品或设计人员，我希望 `outlined` 按钮默认使用更克制的 border 颜色，并在 hover 时平滑过渡到与文字一致的颜色，这样交互变化更稳定，也不会出现额外描边层的错位问题。

**独立测试**：在 `outlined` 按钮上检查默认与 hover 状态，确认只有真实 border 颜色发生过渡，不出现额外边框轨迹、内嵌新边框或圆角错位。

**验收场景**：

1. **假如** 一个 `outlined` 按钮处于默认状态，**当** 用户尚未 hover，**那么** border 颜色必须比文字颜色更低色度、更克制。
2. **假如** 指针移入一个 `outlined` 按钮，**当** hover 生效，**那么** border 颜色必须平滑过渡到与当前文字一致的颜色，而不是瞬时闪变。

### 用户故事 3 - 一致且可访问的状态反馈（优先级：P3）

作为组件消费者和可访问性评审者，我希望 `link` 与 `outlined` 的 hover 反馈在不同颜色、尺寸、形状和主题下都保持一致，并且不影响键盘焦点、禁用态和低动态偏好场景，这样该能力才能作为稳定的公共组件行为推广。

**独立测试**：在不同颜色、尺寸、形状及 disabled / focus-visible / reduced-motion 条件下检查状态反馈，确认 hover 反馈不会替代焦点或可用性提示。

## 边界情况

- `link` 在快速重复 hover 时不得出现双重下划线或结束瞬间变粗。
- `link` 在 icon + text 组合下不得只对文字部分显示下划线。
- `outlined` 不得通过额外装饰层模拟边框，避免与真实 border 产生间距或圆角错位。
- disabled 按钮不得展示 hover 反馈。
- focus-visible 必须继续作为键盘用户的主要状态信号。
- reduced-motion 场景下，反馈仍需可识别，但不能依赖显著连续运动。

## 需求

### 功能需求

- **FR-001**：系统必须为 `link` 按钮提供默认的 hover 下划线显现反馈，且显现方向为从左到右。
- **FR-002**：`link` 下划线反馈必须在 hover 退出后可重置，并在下一次 hover 时重新从左侧开始。
- **FR-003**：`link` 下划线反馈不得造成 hover 结束瞬间的线条加粗或与原生 `text-decoration` 叠加。
- **FR-004**：当 `link` 按钮同时包含 icon 和可见文字时，下划线必须覆盖整条内容行。
- **FR-005**：系统必须为 `outlined` 按钮提供默认的 border 颜色过渡反馈，而不是额外的描边动画层。
- **FR-006**：`outlined` 默认 border 颜色必须比当前文字颜色更低色度，并在 hover 时平滑过渡到与文字一致的颜色。
- **FR-007**：本次能力不得新增或保留 `animated` 一类的公开开关属性。
- **FR-008**：`filled`、`ghost`、`IconButton` 和 `Button.Icon` 的既有交互模型不得被本次调整破坏。
- **FR-009**：disabled、focus-visible 和 reduced-motion 场景必须保持可识别且不误导的状态反馈。
- **FR-010**：Storybook、website、README 和测试材料必须反映新的默认行为，不得继续描述已删除的 `animated` 开关或 `outlined` 环绕描边动画。

### 无障碍与 UI 契约

- **用户角色 / 参与方**：鼠标或触控板用户、仅键盘用户、屏幕阅读器用户、产品设计评审者、消费 `Button` 能力的应用开发者。
- **交互模型**：`link` 使用默认下划线显现；`outlined` 使用真实 border 的颜色过渡；点击与键盘激活语义保持不变。
- **需要覆盖的状态**：默认、hover、hover 退出后重置、focus-visible、disabled、reduced-motion。
- **主题 / token 影响**：`outlined` 默认 border 保持低色度；hover 后与文字颜色对齐；`link` 下划线跟随当前文字颜色。

### 假设与依赖

- 本次范围仅涉及 `Button` 的 `link` 与 `outlined` 两个文本 variant。
- 本次不引入新的公开属性，不改变按钮点击语义、尺寸体系或 shape 规则。
- 评审与回归验证继续复用现有 Storybook、website 和单测入口。

## 成功标准

- **SC-001**：在 100% 的 `link` 预览样例中，下划线都能从左到右显现，并在重复 hover 时保持无残影、无变粗。
- **SC-002**：在 100% 的 `link` icon + text 样例中，下划线都覆盖整条内容行。
- **SC-003**：在 100% 的 `outlined` 预览样例中，border 都只通过真实边框颜色变化表达 hover，不出现额外描边层错位。
- **SC-004**：在所有已定义的颜色、尺寸、形状组合中，`outlined` 的默认 border 都明显低于文字饱和度，hover 后都能顺畅过渡到文字颜色。
- **SC-005**：在 100% 的键盘与 disabled 验收场景中，用户都能在不依赖 hover 的前提下识别按钮状态并完成操作判断。
