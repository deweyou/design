# 功能规格：引入 Ark UI 作为组件库基础层

**功能分支**：`20260327-ark-ui-integration`
**创建时间**：2026-03-27
**状态**：草稿
**输入**：用户描述："现在要做一个大改动，因为我昨天发现了一个 ark.ui：https://ark-ui.com，它是通过了react基础组件能力，然后可以直接去实现样式。我就想基于它去封装我自己的 react 组件库，而不要自己重新实现了。目前我们实现了 button、popover、text 三个组件，button 和 text 目前应该是不用动的，但是 popover 是可以重新基于 ark.ui 实现了，然后保留我们的设计风格。然后后续的新增组件需要基于 ark-ui 去开发，这是后面的迭代范式，看看怎么补充到规则中"
**语言要求**：本文件及同级 `/specs/` 文档正文必须使用简体中文；代码标识符、命令、文件路径、协议字段和第三方 API 名称可保留原文。

## 背景

Ark UI 是一套无样式（headless）的 React 组件原语库，内置经过验证的无障碍行为、键盘交互和焦点管理，使用者只需专注于样式层的实现。本特性的核心目标是：**将 Ark UI 确立为本组件库中复杂交互组件的开发基础层**，避免重复实现已有成熟解法的行为逻辑，同时保留库的设计风格和公开 API 契约。

本次范围：

- 将已有的 `popover` 组件迁移至 Ark UI 基础层，保留现有设计风格
- `button` 和 `text` 组件**不作任何改动**
- 在项目规范中明确：新增组件原则上必须基于 Ark UI 构建

## 用户场景与测试（必填）

### 用户故事 1 - Popover 消费方无感知迁移（优先级：P1）

作为一名正在使用 `popover` 组件的开发者，我希望迁移后的 popover 在视觉和交互上与之前完全一致，不需要修改任何调用侧代码。

**为什么是这个优先级**：这是整个迁移的核心前提。如果 API 或视觉发生破坏性变化，消费方需要同步改动，迁移价值将大打折扣；且这是其他故事能正常进行的基础。

**独立测试**：可以在 `website` 或 Storybook 中打开现有 popover 示例，确认视觉、交互和无障碍行为与重构前完全一致，无需修改任何演示代码。

**验收场景**：

1. **假如** 消费方使用重构前的 `<Popover />` 调用代码，**当** 切换到迁移后版本，**那么** 无需更改任何 props 或引用，组件即可正常渲染并工作
2. **假如** 用户点击触发器打开 popover，**当** 点击外部区域，**那么** popover 正确关闭，与迁移前行为一致
3. **假如** 用户使用键盘打开 popover，**当** 按下 `Escape`，**那么** popover 关闭并将焦点归还触发元素
4. **假如** popover 处于打开状态，**当** 使用 `Tab` 键，**那么** 焦点在 popover 内部正确循环，不会意外跳出

### 用户故事 2 - 开发者使用新范式新增组件（优先级：P2）

作为一名向本组件库贡献新组件的开发者，我希望有明确的规范告诉我：凡是 Ark UI 已有对应原语的交互型组件，都应当基于 Ark UI 构建，而不是从头实现行为逻辑，这样我只需专注于样式和设计 token 的对接。

**为什么是这个优先级**：迁移范式的价值不仅在于当前的 popover 重构，更在于为后续组件建立可持续的开发标准。若规范不明确，后续贡献者会重新陷入自行实现行为的状态。

**独立测试**：查阅更新后的项目规范文件（AGENTS.md 及宪章），确认能找到"新增交互组件应基于 Ark UI 开发"的明确条款，并附有判断准则（何时可以例外）。

**验收场景**：

1. **假如** 开发者查阅 AGENTS.md，**当** 搜索"新增组件"或"Ark UI"相关条目，**那么** 能找到明确说明：交互型新组件优先使用 Ark UI 原语，仅在 Ark UI 无对应覆盖时才考虑自行实现
2. **假如** 开发者查阅宪章（constitution.md），**当** 查看组件开发原则，**那么** 能看到 Ark UI 作为组件基础层的定位和使用边界已被正式记录

### 用户故事 3 - 无障碍能力通过 Ark UI 自动覆盖（优先级：P2）

作为一名关注无障碍合规的使用者，我希望迁移后的 `popover` 能通过 Ark UI 提供的内置能力满足键盘导航、屏幕阅读器标签和焦点管理要求，而不需要库开发者手工实现这些逻辑。

**为什么是这个优先级**：当前自行实现的无障碍逻辑存在遗漏风险；Ark UI 的内置无障碍能力是本次引入的核心收益之一，必须作为独立验证点被覆盖。

**独立测试**：可以单独使用辅助技术（如 VoiceOver 或屏幕阅读器）测试重构后的 popover，确认 ARIA 属性、角色和焦点管理均符合预期，无需其他组件配合。

**验收场景**：

1. **假如** 屏幕阅读器用户将焦点移动到 popover 触发器，**当** 触发器被激活，**那么** 阅读器能正确播报 popover 内容区域已展开
2. **假如** popover 内有可聚焦元素，**当** popover 打开，**那么** 焦点自动移入 popover 内部
3. **假如** popover 关闭，**当** 焦点离开，**那么** 焦点返回触发器，不会丢失

## 需求（必填）

### 功能需求

- **FR-001**：系统必须将 Ark UI 作为 `packages/components` 的正式依赖引入，并在版本约束上与现有依赖体系兼容
- **FR-002**：`popover` 组件必须基于 Ark UI 对应原语重新实现，保留现有公开 API（props、受控/非受控模式、事件回调）不发生破坏性变更
- **FR-003**：重构后的 `popover` 视觉风格必须与迁移前保持一致，包括颜色、间距、圆角、动效等，均通过现有设计 token 驱动
- **FR-004**：`button` 和 `text` 组件在本次变更中不得被修改
- **FR-005**：项目规范文件（AGENTS.md 及 constitution.md）必须新增明确条款：交互型新组件优先基于 Ark UI 原语开发，并注明何时可以例外（即 Ark UI 无对应覆盖时）
- **FR-006**：系统必须为 `popover` 定义公开 API 面，包括 props 命名、variants，以及受控/非受控行为，且与重构前保持兼容
- **FR-007**：重构后的 `popover` 必须满足无障碍要求，包括语义化结构、焦点进入/退出/归还、键盘交互（`Escape` 关闭、`Tab` 焦点循环）和辅助技术标签（ARIA role、aria-expanded、aria-controls 等），通过 Ark UI 内置能力覆盖
- **FR-008**：本次变更不引入新的设计 token；样式层必须复用现有 token，不得硬编码一次性值
- **FR-009**：`website` 或 Storybook 中的 `popover` 预览必须覆盖主要状态（默认关闭、打开、受控模式）及边界状态（无可聚焦子元素、内容超长），且与重构前呈现一致

### 无障碍与 UI 契约

- **用户角色 / 参与方**：仅键盘用户、屏幕阅读器用户、消费 `@deweyou-ui/components` 的应用开发者
- **交互模型**：点击触发器打开/关闭、`Escape` 关闭、`Tab` 焦点循环、点击外部区域关闭、焦点进入与归还
- **需要覆盖的状态**：关闭（默认）、打开、受控打开、受控关闭、内部无可聚焦元素时的焦点行为
- **主题 / token 影响**：复用现有语义色 token 及间距/圆角 token，无新增 token

### 假设

- Ark UI 提供了与 `popover` 行为对等的原语（浮层定位、触发/内容分离、焦点管理），可以满足现有设计需求
- 现有 `popover` 的公开 API 足够简洁，迁移时无需引入 breaking change
- 包体积增长在业务可接受范围内（Ark UI 按需引入，不全量打包）
- `button` 和 `text` 的行为逻辑简单，不需要 headless 层介入，维持现状合理
- 开发者在实现阶段已安装 Ark UI MCP Server（`@ark-ui/mcp`），可在 Claude Code 中直接查阅组件 API 和文档，无需手动查阅网页文档
