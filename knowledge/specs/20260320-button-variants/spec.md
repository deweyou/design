# 功能规格：重构 Button 组件基础能力

**功能分支**：`20260320-button-variants`  
**创建时间**：2026-03-20  
**状态**：草稿  
**输入**：用户描述："开始来重新实现 Button 组件吧 我看现在的demo有几个问题：buttonCustomizationContract 是什么东西？暴露的组件和类型应该是Button 和ButtonProps，没必要写那么长。 然后我本期想实现的button能力是：支持几种button 类型：outlined、filled、text、icon；支持几种 size：large、medium、small、extra-large、extra-small；支持几种按钮形状：直角、小圆角、圆形；button类型和按钮形状可能会有 gap，可以对一下。"
**语言要求**：本文件及同级 `/specs/` 文档正文必须使用简体中文；代码标识符、命令、文件路径、协议字段和第三方 API 名称可保留原文。

## 需求澄清

### 会话 2026-03-20

- 问题：按钮的公开主配置字段应命名为 `type` 还是 `variant`？  
  回答：继续使用 `variant`。原生 `<button>` 已存在 `type="button" | "submit" | "reset"` 语义，视觉类型不应与原生字段重名。
- 问题：按钮轮廓字段应命名为 `radius` 还是 `shape`？  
  回答：使用 `shape`。本期表达的是离散轮廓类型，而不是可调数值圆角大小。
- 问题：公开 `shape` 支持哪些值？  
  回答：支持 `rect`、`rounded`、`pill` 三种值。
- 问题：所有 `variant` 都支持 `shape` 吗？  
  回答：不是。`shape` 只对 `filled` 与 `outlined` 生效；`ghost` 与 `link` 不支持 `shape`，传入时必须给出明确错误，而不是静默忽略。
- 问题：`icon` 是否继续作为独立 `variant`？  
  回答：不继续。icon 只是内容形态，不是按钮类型；是否只有图标由内容和无障碍规则约束，而不是由 `variant` 决定。
- 问题：`danger`、`success`、`loading`、`selected`、`disabled` 应如何建模？  
  回答：它们应作为独立语义字段或状态字段存在，不应被折叠进 `variant`。本期必须覆盖 `disabled`；`color` 在本期直接落地为独立字段，`loading`、`selected` 等后续扩展也不得并入 `variant`。
- 问题：本期支持哪些按钮类型？  
  回答：公开 `variant` 收敛为 `filled`、`outlined`、`ghost`、`link` 四类；其中 `link` 是纯文本 hover 下划线，`ghost` 是纯文本 hover 背景高亮。

### 会话 2026-03-21

- 问题：是否公开 `color` 字段，而不是只把语义色留作未来方向？  
  回答：是。`color` 在本期直接公开，支持 `neutral` 与 `primary`。
- 问题：`color` 的默认值是什么？  
  回答：默认值为 `neutral`。Button 在未显式配置时保持黑白灰中性色表现。
- 问题：`primary` 具体控制哪些反馈？  
  回答：当 `color="primary"` 时，`filled` 的背景、`outlined` 的边框、`ghost` 的 hover 背景、`link` 的文本与下划线都切换到主题色体系。
- 问题：`filled` 在默认色和主题色下的文字颜色如何处理？  
  回答：默认 `neutral` 的 `filled` 走黑白对比；`primary` 的 `filled` 使用主题前景色 token，保证在主题色背景上可读。
- 问题：本期是否需要同步调整页面和预览面的整体底色氛围？  
  回答：需要。website 与 Storybook 的页面底色统一收敛为浅色白底、深夜模式黑底，避免页面氛围干扰按钮的中性色与主题色判断。
- 问题：`ghost` 与 `link` 的内边距是否继续与 `filled` / `outlined` 对齐？  
  回答：不继续。`ghost` 与 `link` 作为轻量文本动作，应使用更紧凑的文本流内边距，不再强行对齐实体按钮的盒子密度。
- 问题：`ghost` 的 hover 圆角是否按内容模式区分？  
  回答：不区分。所有 `ghost` 默认统一使用 `0.4rem` 圆角，与当前 `rounded` 语义对齐，但不通过 `shape` 公开。
- 问题：纯图标按钮是否继续使用自动推断的专用布局模式？  
  回答：不使用。无可见文本只作为无障碍约束条件存在，不再自动触发方形按钮或其他布局特例。
- 问题：按钮内默认 icon 是否需要随按钮字号一起缩放？  
  回答：需要。未显式传入 icon `size` 时，icon 默认继承周围字号；icon 渲染器还需要通过稳定的方形 wrapper 与视口补方降低非正方形源数据带来的占位偏差。
- 问题：`filled` 与 `outlined` 是否继续依赖固定 `min-height` 维持高度？  
  回答：不继续。本期实体按钮高度改由字号、行高和统一四边内边距共同驱动，不再额外使用固定最小高度。
- 问题：常规按钮遇到超长文案时应如何处理？  
  回答：`filled`、`outlined` 与 `ghost` 默认保持单行，并在受限宽度下使用省略号截断；`link` 继续保留文本流中的可换行行为。

## 用户场景与测试（必填）

### 用户故事 1 - 用统一 API 使用标准按钮（优先级：P1）

作为消费该组件库的应用开发者，我希望只需要理解 `Button`、`ButtonProps` 以及清晰的 `variant` / `color` / `shape` 能力模型，就能选出合适的按钮视觉样式，而不需要额外理解冗长别名或辅助契约对象。

**为什么是这个优先级**：这是本期最直接的价值。如果公开命名和使用入口不清晰，后续所有视觉能力都会继续放大接入成本和认知负担。

**独立测试**：仅使用公开文档中的 `Button` 与 `ButtonProps`，分别完成 `filled`、`outlined`、`ghost`、`link` 的代表性示例，并确认默认 `color="neutral"`、显式 `color="primary"`、`filled` / `outlined` 的 `shape` 以及 `ghost` / `link` 的非 shape 组合都能被直接理解和接入。

**验收场景**：

1. **假如** 开发者第一次接入组件库中的按钮，**当** 其查看按钮文档或预览入口，**那么** 能看到 `Button` 与 `ButtonProps` 作为主要公开 API，而不是更长的别名或额外契约对象。
2. **假如** 开发者需要创建一个标准操作按钮，**当** 其通过统一的 `variant` 模型选择 `filled`、`outlined`、`ghost` 或 `link`，并根据是否需要主题强调选择 `neutral` 或 `primary`，且在需要时为 `filled` / `outlined` 配置 `shape` 时，**那么** 能得到可预测的视觉结果。

---

### 用户故事 2 - 在支持矩阵内选择合适的按钮样式（优先级：P2）

作为设计师或组件开发者，我希望组件库明确列出按钮 `variant`、`color`、尺寸和 `shape` 的支持边界，从而在评审和开发时能快速判断哪些组合可以直接使用，哪些组合当前不在支持范围内。

**为什么是这个优先级**：这直接决定该组件是否可被稳定复用。如果边界不清晰，团队会在 demo、设计稿和实际接入中反复猜测组件能力定义。

**独立测试**：查看按钮预览或规格说明，逐一验证 `filled`、`outlined`、`ghost`、`link` 在 `neutral` / `primary` 下的支持范围、五种尺寸、三种 `shape` 值，以及 `ghost` / `link` 不支持 `shape` 的明确说明。

**验收场景**：

1. **假如** 设计师需要为主操作选择 `filled` 按钮样式，**当** 其查看支持矩阵时，**那么** 能明确知道 `filled` 支持哪些尺寸、`shape` 与 `color` 模式，并可直接看到对应示例。
2. **假如** 开发者正在比较 `link` 与 `ghost` 两种轻量按钮形态，**当** 其查看按钮文档或预览时，**那么** 能快速理解“hover 下划线”与“hover 背景色”两种反馈分别适用于什么场景。
3. **假如** 开发者尝试为 `ghost` 或 `link` 配置 `shape`，**当** 其查看按钮文档、类型定义或运行时反馈时，**那么** 能明确知道该组合当前不受支持，而不是默默失效。

---

### 用户故事 3 - 在边界与无障碍场景下保持可预测（优先级：P3）

作为仅键盘用户、屏幕阅读器用户和组件维护者，我希望按钮在禁用、聚焦、纯图标内容与不支持组合等边界情况下都保持可预测行为，从而避免视觉能力升级后引入可用性回归。

**为什么是这个优先级**：按钮是高频交互入口。即使样式模型清晰，如果在无障碍或边界场景下表现不一致，组件仍然不适合被广泛复用。

**独立测试**：验证所有支持 `variant` 的默认、hover、focus-visible、active、disabled 状态，并检查任意没有可见文本的按钮是否仍具备可访问名称；同时确认对 `ghost` / `link` 传入 `shape` 不会静默失效。

**验收场景**：

1. **假如** 用户通过键盘导航到按钮，**当** 焦点进入任意支持的按钮 `variant` 时，**那么** 按钮会显示清晰的聚焦反馈，并可通过键盘完成激活或识别禁用状态。
2. **假如** 某个按钮只有图标而没有可见文本，**当** 屏幕阅读器用户访问该按钮时，**那么** 仍能获得该按钮动作的清晰可访问名称。
3. **假如** 开发者为 `ghost` 或 `link` 传入了 `shape`，**当** 组件进行渲染或预览展示时，**那么** 能得到明确的不可用说明，而不是静默忽略该字段。
4. **假如** 开发者把默认 icon 放入不同 `size` 的 Button 中，**当** 未显式配置 icon `size` 时，**那么** icon 会跟随按钮字号缩放，而不是停留在固定像素尺寸。

### 边界情况

- 当按钮只有图标而没有可见文本时，系统如何确保其仍具备可访问名称？
- 当开发者需要在 `link` 与 `ghost` 之间选择时，系统如何帮助其快速判断“hover 下划线”和“hover 背景色”两种反馈各自的适用场景？
- 当按钮文案较长但尺寸为 `extra-small` 或 `small` 时，组件如何保持可读性与点击目标的稳定性？
- 当按钮处于 `disabled` 状态时，鼠标、键盘和辅助技术用户会看到怎样一致的非交互反馈？
- 当开发者为 `ghost` 或 `link` 传入 `shape` 时，系统如何明确告知该组合不受支持？
- 当开发者需要在默认中性色与主题强调色之间选择时，系统如何帮助其判断何时保持 `neutral`、何时切换到 `primary`？
- 当后续需要引入 `danger`、`success`、`loading` 或 `selected` 时，如何保证这些能力以独立字段存在，而不是继续膨胀 `variant`？
- 当 `ghost` 或 `link` 被插入正文或密集工具栏时，系统如何避免它们因实体按钮级别的 padding 而显得突兀？
- 当 icon 源数据的 `viewBox` 不是正方形时，系统如何降低图标在 Button 内容区中的占位偏差？

## 需求（必填）

### 功能需求

- **FR-001**：组件库必须将该组件的主要公开名称统一为 `Button`，并将主要 props 类型统一为 `ButtonProps`。
- **FR-002**：标准按钮使用流程必须仅依赖 `Button` 与 `ButtonProps` 这组公开 API；消费者在完成常规使用时不得被要求理解额外的冗长别名或辅助契约对象。
- **FR-003**：系统必须以 `variant` 为中心组织按钮的公开视觉能力，并保持 `Button` 与 `ButtonProps` 作为主要公开名称。
- **FR-004**：按钮必须支持以下 `variant`：`filled`、`outlined`、`ghost`、`link`。
- **FR-005**：按钮必须支持以下 `color` 模式：`neutral`、`primary`。
- **FR-006**：按钮必须支持五种尺寸：`extra-small`、`small`、`medium`、`large`、`extra-large`。
- **FR-007**：按钮必须支持以下 `shape` 选项并对外使用一致命名：`rect`、`rounded`、`pill`。
- **FR-008**：系统必须定义并文档化 `variant` / `color` / `shape` 的支持矩阵；本期默认支持矩阵如下：所有 `variant` 默认 `color="neutral"`，显式 `color="primary"` 时切换到主题强调色；`filled`、`outlined` 支持 `rect`、`rounded`、`pill`，`ghost` 与 `link` 不支持 `shape`。
- **FR-009**：对于不在支持矩阵内的组合，系统必须给出明确且文档化的不可用说明，不得静默忽略或静默回退。
- **FR-010**：系统必须提供文档化默认值，使开发者在未显式配置时仍能得到可预测结果；默认 `variant` 为 `filled`，默认 `color` 为 `neutral`，默认尺寸为 `medium`，默认 `shape` 为 `rounded`。
- **FR-011**：系统必须将 `ghost` 与 `link` 的差异明确限定为轻量交互反馈模式差异，其中 `link` 使用下划线反馈，`ghost` 使用背景色反馈。
- **FR-012**：系统必须将 `loading`、`selected`、未来的语义色等能力视为独立字段方向，而不是新的 `variant` 值；本期至少要明确这一建模原则，并已将 `color` 作为独立字段落地。
- **FR-013**：系统必须覆盖按钮的关键交互状态，包括默认、hover、focus-visible、active 和 `disabled`，并保证这些状态在所有支持 `variant` 中具有一致且可辨识的反馈。
- **FR-014**：系统必须满足明确的无障碍要求，包括按钮语义、键盘激活、焦点可见性、`disabled` 非交互行为，以及任意无可见文本按钮的辅助技术命名。
- **FR-015**：系统必须提供可评审的按钮预览，覆盖所有支持的 `variant`、`color`、尺寸、`shape` 组合，以及至少一组 `disabled`、长文案、纯图标内容、`ghost` 和 `link` 的代表性边界示例。
- **FR-016**：系统必须明确按钮视觉能力与现有设计语言之间的关系，说明默认中性色、主题强调色、页面白底 / 黑底基调与状态反馈之间的契约边界。
- **FR-017**：系统必须为贡献者提供可直接判断的能力边界说明，使设计师和开发者无需阅读实现代码，就能判断某个按钮组合是否在本期范围内。
- **FR-018**：系统必须将 `ghost` 与 `link` 视为文本流中的轻量动作，使用明显轻于 `filled` / `outlined` 的内边距密度，而不是继续对齐实体按钮的盒子尺寸。
- **FR-019**：系统必须将 `ghost` 的默认圆角固定为 `0.4rem`，作为该 variant 的内部样式契约，而不是 `shape` 公开能力的一部分。
- **FR-020**：系统不得因按钮内容没有可见文本而自动切换到专用方形布局；无可见文本仅用于触发可访问名称校验和边界预览说明。
- **FR-021**：系统必须保证按钮内未显式配置尺寸的 icon 默认继承周围字号，并通过稳定的方形 wrapper 与非正方形 `viewBox` 补方策略降低图标占位抖动。
- **FR-022**：`filled` 与 `outlined` 的高度必须由字号、行高和统一四边内边距驱动，不再额外依赖固定 `min-height` 维持实体按钮高度。
- **FR-023**：`filled`、`outlined` 与 `ghost` 默认必须保持单行不换行，并在受限宽度下以省略号截断超长文案；`link` 可以继续按照文本流规则换行。

### 无障碍与 UI 契约（UI 工作必填）

- **用户角色 / 参与方**：消费组件库的应用开发者、评审按钮规范的设计师、在页面中组合按钮的组件开发者、仅键盘用户、屏幕阅读器用户、维护组件能力边界的库维护者
- **交互模型**：鼠标点击、键盘聚焦与激活、`disabled` 非交互反馈、无可见文本按钮的可访问命名、通过预览面选择按钮 `variant` / `color` / 尺寸 / `shape` 组合，以及区分 `ghost` 与 `link` 的轻量反馈
- **需要覆盖的状态**：默认、hover、focus-visible、active、`disabled`、长文案、纯图标内容、`ghost`、`link`、`neutral`、`primary`、支持组合、不支持组合说明，以及 icon 跟随字号缩放的代表性场景
- **需要覆盖的状态**：默认、hover、focus-visible、active、`disabled`、长文案、省略截断、纯图标内容、`ghost`、`link`、`neutral`、`primary`、支持组合、不支持组合说明，以及 icon 跟随字号缩放的代表性场景
- **主题 / token 影响**：优先复用现有语义颜色、尺寸、轮廓和焦点反馈定义；Button 默认中性色表现与 `primary` 主题强调色都必须落在现有 token 契约内；website 与 Storybook 的页面底色同步收敛为浅色白底、深夜模式黑底

### 关键实体

- **按钮公开契约**：开发者接入按钮时需要理解的公开名称、默认值和可选能力集合，包括 `Button`、`ButtonProps`、`variant`、`color`、`size`、`shape` 以及相关状态字段。
- **按钮 Variant**：决定按钮视觉优先级和交互反馈的样式类别，包括 `filled`、`outlined`、`ghost`、`link`。
- **按钮 Color 模式**：决定按钮保持黑白灰中性色，还是显式切换到主题强调色的配置轴，包括 `neutral` 与 `primary`。
- **按钮尺寸**：决定按钮点击目标、内边距和内容密度的标准尺寸等级，包括 `extra-small`、`small`、`medium`、`large`、`extra-large`。
- **按钮形状**：决定 `filled` / `outlined` 按钮外轮廓的标准 `shape` 选项，包括 `rect`、`rounded`、`pill`。
- **按钮状态字段**：决定按钮当前语义或状态的扩展配置，例如 `disabled`，以及后续可能引入但不得折叠进 `variant` 的 `loading`、`selected` 等字段。
- **按钮轻量内容密度**：决定 `ghost` 与 `link` 在文本流和密集布局中的内边距、圆角与 icon 间距策略，确保它们保持轻量动作观感。
- **图标渲染约束**：决定按钮内 icon 在未显式配置尺寸时如何继承字号，并通过方形 wrapper 与 `viewBox` 补方保持稳定占位。
- **按钮预览示例**：用于展示按钮在主要状态、边界场景和公开能力范围内表现的评审样例。

## 依赖与假设

### 依赖

- 本功能依赖组件库已有的设计语言能够提供按钮所需的基础颜色、层级、轮廓、间距、文字和焦点反馈规则。
- 本功能依赖存在一个可供设计师和开发者共同评审的按钮预览入口，用于展示主要场景和边界场景。
- 本功能依赖 website 与 Storybook 能共享更新后的浅色白底 / 深夜黑底页面基调，以保证主题方向在消费端预览中一致。

### 假设

- 本期不再将 `icon` 视为独立 `variant`；图标仅作为按钮内容的一部分存在。
- 本期默认将两种轻量按钮形态命名为 `ghost` 与 `link`，便于与常见设计系统语义对齐并降低理解成本。
- 本期公开 `color` 支持 `neutral` 与 `primary` 两种值，并以 `neutral` 为默认值。
- 本期公开 `shape` 只服务于 `filled` 与 `outlined`，`ghost` / `link` 不支持该配置轴。
- 本期范围不要求一次性完整实现 `loading`、`selected` 等扩展字段，但要求公开模型为它们预留独立字段方向。
- 本期默认页面与预览底色从旧暖米色转向浅色白底 / 深夜黑底，以降低背景氛围对按钮颜色判断的干扰。
- 本期 `ghost` 与 `link` 将使用比 `filled` / `outlined` 更轻的文本流内边距，不追求四类按钮视觉盒子完全一致。
- 本期 `filled` 与 `outlined` 也不再依赖固定最小高度，而是回到 padding 驱动的内容盒模型。
- 本期不会为纯图标按钮保留自动方形布局特例；若未来需要标准方形 icon button，应通过显式能力单独建模。
- 本期重点是建立稳定、易理解的基础按钮契约与能力边界，而不是一次性覆盖所有业务化按钮风格。

## 成功标准（必填）

### 可度量结果

- **SC-001**：90% 以上的代表性按钮接入任务可仅通过 `Button` 与 `ButtonProps` 完成，不需要额外查找其他公开命名或辅助契约对象。
- **SC-002**：100% 的公开 `variant`、`color`、尺寸和受支持 `shape` 组合都能在按钮预览中找到对应示例，且 `ghost` / `link` 的差异说明清晰可见。
- **SC-003**：100% 的无可见文本按钮评审示例都具备可访问名称，且所有支持 `variant` 都覆盖键盘聚焦与激活检查。
- **SC-004**：设计师或开发者可以在 30 秒内通过按钮预览或规格说明判断某个 `variant`、`color`、尺寸、`shape` 组合是否受支持。
- **SC-005**：在主操作、次级操作、两类轻量文本操作这四类代表性任务中，90% 以上可以直接映射到文档化按钮 `variant`，并在 15 秒内判断应保持 `neutral` 还是切换到 `primary`。
- **SC-006**：100% 的代表性 icon 示例在未显式传入 icon `size` 时都随周围字号缩放，且非正方形源 `viewBox` 不会再直接引入明显的占位抖动。
- **SC-007**：100% 的 `filled`、`outlined` 与 `ghost` 超长文案示例在受限宽度下保持单行，并通过省略号提示截断，而不是自动换行。
