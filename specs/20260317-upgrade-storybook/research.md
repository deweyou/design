# 研究：升级 Storybook 到最新稳定版

## 决策：目标版本采用当前稳定的 Storybook 10.2 版本线

**理由**：Storybook 官方文档和发布页面当前都把 10.2 视为活动稳定版本线，因此这里的 “latest” 应规划为 10.2，而不是继续停留在 8.6 的补丁更新。现在升级，可以在更多组件和 stories 加入之前先完成大版本迁移。

**备选方案**：

- 保持在 8.6.x，只修版本偏差：拒绝，原因是会把大版本迁移风险留到后续更重的组件工作阶段。
- 只升级到 9.1：拒绝，原因是虽然短期风险更低，但无法满足“升级到最新稳定版”的目标。

## 决策：按 8.6.x -> 9.1 -> 10.2 的迁移路径进行规划

**理由**：Storybook 官方的 10 版本迁移指南明确要求，低于 9 的项目必须先升到 9。因此即使实现最终落在一个分支里，规划上也必须明确这是一条两阶段兼容性迁移路径。

**备选方案**：

- 直接从 8 跳到 10，不做中间版本兼容性评审：拒绝，原因是违背官方迁移顺序，且会让问题定位更困难。
- 把 9 和 10 拆成两个 feature 分支：拒绝，原因是当前用户目标就是尽快升级，而当前项目规模也仍适合一次完成。

## 决策：保持 React + Vite 的框架组合以及当前 app 结构

**理由**：当前 app 已经使用 `@storybook/react-vite`、基于 TypeScript 的 ESM 配置，以及独立的 `apps/storybook` 边界。这与 Storybook 官方支持的 React + Vite 方案一致，也能把迁移范围控制到最小。

**备选方案**：

- 使用其他 framework preset 重新搭建 Storybook：拒绝，原因是会引入不必要的大规模改动，并掩盖真正的升级回归问题。
- 将 Storybook 合并进 `apps/website`：拒绝，原因是仓库规则明确 Storybook 是内部评审面，而不是公开文档主入口。

## 决策：将所有 Storybook 相关依赖对齐到同一版本线，并移除废弃包

**理由**：当前工作区在 `@storybook/react`、`@storybook/react-vite`、`@storybook/test` 和根级 `storybook` 包之间存在版本偏差。Storybook 9 的迁移说明还要求合并或移除若干旧包，并把部分导入迁到 `storybook/*` 命名空间。因此规划中不能只做版本号提升，还必须包含包清单清理。最终实现结果为：`storybook`、`@storybook/react`、`@storybook/react-vite` 和 `@storybook/addon-docs` 统一为 `10.2.19`，并移除了旧的 `@storybook/addon-essentials`、`@storybook/addon-interactions` 与 `@storybook/test`。

**备选方案**：

- 只升级当前产生 peer warning 的包：拒绝，原因是部分对齐仍会留下隐藏的不兼容问题。
- 继续保留已废弃的包，只要还能安装就行：拒绝，原因是 Storybook 10 对废弃项更严格，9 中的 deprecations 很可能在 10 中直接被移除。

## 决策：使用仓库标准的 `vp` 和 catalog 管理依赖，同时把 Storybook 自动迁移指南当作参考

**理由**：虽然 Storybook 官方建议使用 `storybook@latest upgrade`，但本仓库的宪章要求依赖和工作流变更都必须通过 Vite+ 命令和共享 catalog 进行。因此最稳妥的方案是：遵循 Storybook 官方迁移说明与校验思路，但具体依赖修改仍通过仓库标准机制完成。

**备选方案**：

- 直接把 Storybook CLI 升级器作为主工作流：拒绝，原因是这会绕过仓库现有工具纪律，并可能破坏 catalog 管理。
- 完全忽略官方迁移工具：拒绝，原因是自动迁移指南仍然有助于识别包移除、配置变更和已知坑位。

## 决策：把显式样式导入和内部评审范围视为不可妥协的兼容性约束

**理由**：`apps/storybook` 明确只用于内部评审，而且当前 preview 会直接导入 `@deweyou-ui/styles/theme.css`。这些都是本仓库自己的本地契约，而不是 Storybook 默认行为，因此即便 Storybook 提供更丰富的 docs 或 testing 特性，迁移计划也必须保留这些约束。

**备选方案**：

- 在升级过程中把 Storybook 扩展成公开文档面：拒绝，原因是这会改变产品范围，并与 `apps/website` 重复。
- 把全局样式导入移动到组件 package 内：拒绝，原因是这违反仓库显式导入全局样式的规则。

## 决策：主要通过可构建性、story 无障碍性和维护者工作流连续性验证迁移成功

**理由**：当前 Storybook app 规模较小。对这次升级来说，最有意义的成功标准是：维护者可以完成安装、构建、打开和导航，不出现依赖不匹配、story 发现失败或 docs / controls 丢失等问题。

**备选方案**：

- 只看依赖版本是否提升：拒绝，原因是版本对齐本身无法证明评审工作流仍然可用。
- 只依赖人工评审：拒绝，原因是宪章要求存在可重复执行的验证命令和预览覆盖。

## 已实现的验证结果

**决策**：保留 `vp run storybook#dev` 作为标准维护者入口，但将默认开发端口调整为 `6106`。

**理由**：用于验证的环境里，`6006` 已经产生端口冲突。将默认端口改为 `6106` 可以为工作区提供可预测的 Storybook 端口，同时仍保持标准 Vite+ 工作流。

**备选方案**：

- 保持 `6006`：拒绝，原因是升级后进行端到端验证时发生端口冲突。
- 要求维护者每次手动指定端口：拒绝，原因是这会削弱默认工作流的可用性。
