# 研究：重构 Button 组件基础能力

## 决策 1：公开 API 统一收敛为 `Button` 与 `ButtonProps`

- **决策**：将按钮主要公开导出统一为 `Button` 与 `ButtonProps`，并把 `FoundationButton`、`FoundationButtonProps`、`buttonCustomizationContract` 视为待移除的旧公开命名。
- **理由**：当前 story、README 和实现都暴露了过长且不直观的 Foundation/contract 命名，和本期“公开 API 易读、可直接消费”的目标相冲突。统一命名也更符合组件库的长期对外表面。
- **备选方案**：
  - 保留 `FoundationButton` 作为长期主入口：拒绝，原因是继续保留历史前缀会放大接入和文档认知成本。
  - 保留 `buttonCustomizationContract` 作为标准使用的一部分：拒绝，原因是常规按钮接入不应要求消费者理解内部样式覆盖契约对象。

## 决策 2：继续使用 `variant`，不改成 `type`

- **决策**：按钮公开视觉模型继续以 `variant` 为主字段，支持 `filled`、`outlined`、`ghost`、`link`。
- **理由**：原生 `<button>` 已有 `type` 语义。如果再使用 `type` 表示视觉类型，会与原生 `button` 属性冲突，导致 API 可读性和类型系统都变差。
- **备选方案**：
  - 改用 `type`：拒绝，原因是会与原生 `type="button" | "submit" | "reset"` 冲突。
  - 改用 `appearance` 或 `kind`：可行，但当前 `variant` 已足够清晰，且与现有讨论语境一致。

## 决策 3：公开暴露 `shape`，但仅适用于 `filled` / `outlined`

- **决策**：按钮公开 `shape` 字段，支持 `rect`、`rounded`、`pill` 三种值；`shape` 只适用于 `filled` 与 `outlined`，`ghost` 与 `link` 不支持。
- **理由**：用户明确希望这组三档轮廓成为 Button 类型能力的一部分，但不想把它建模为“可调半径”。`shape` 比 `radius` 更准确，也比把所有组合折叠进 `variant` 更清晰。
- **备选方案**：
  - 使用 `radius`：拒绝，原因是容易让人误解为数值圆角大小。
  - 把所有轮廓直接折叠进 `variant`：拒绝，原因是会让 `variant` 枚举膨胀。
  - 完全不暴露轮廓：拒绝，原因是用户明确要求这是可选能力。

## 决策 4：`ghost` 与 `link` 通过交互反馈区分

- **决策**：`link` 表示 hover 以文本下划线反馈为主的轻量按钮，`ghost` 表示 hover 以背景色反馈为主的轻量按钮。
- **理由**：这样能把两者的差异稳定地绑定到用户可感知的交互反馈，而不是再引入新的视觉配置轴。对评审和文档来说也更容易建立清晰示例。
- **备选方案**：
  - 将两者继续统称为 `text`，再额外补一个 hover-style 属性：拒绝，原因是会重新分裂原本想收敛到 `variant` 的模型。
  - 用 tone 或 emphasis 层级区分：拒绝，原因是会把视觉层级和交互反馈混在一起。

## 决策 5：`icon` 视为内容模式，而不是独立 `variant`

- **决策**：图标只是按钮内容的一部分，纯图标按钮不再通过 `variant="icon"` 表达。是否只有图标由内容和无障碍规则控制。
- **理由**：用户已明确指出 icon 是内容集合而不是按钮类型。这样也能让“无可见文本必须提供可访问名称”的规则适用于所有按钮，而不是只作用于某个特定 variant。
- **备选方案**：
  - 保留 `icon` 作为独立 variant：拒绝，原因是会把内容模式误建模为类型。
  - 完全忽略纯图标按钮场景：拒绝，原因是实际使用中仍需要覆盖这类无障碍边界。

## 决策 6：语义和状态字段与 `variant` 解耦

- **决策**：`color`、`loading`、`selected`、`disabled` 等能力应作为独立语义字段或状态字段存在，不得折叠进 `variant`；其中 `color` 在本期直接落地为独立公开字段。
- **理由**：`variant` 用于表达按钮类型，语义色和状态应是另外的维度。否则 `variant` 会不断膨胀成不可维护的枚举；把 `color` 直接做成独立字段，也能让“默认中性色、显式主题强调色”比新增更多 `variant` 更容易理解。
- **备选方案**：
  - 将所有状态和语义都折叠进 `variant`：拒绝，原因是会破坏 API 可扩展性。
  - 在当前阶段完全不讨论后续字段方向：拒绝，原因是现在就需要定清建模原则，避免后续继续堆进 `variant`。

## 决策 7：`color` 默认保持中性色，`primary` 显式 opt-in 主题强调色

- **决策**：`Button` 新增 `color` 字段，支持 `neutral` 与 `primary`；默认值为 `neutral`。只有 `primary` 才会让 `filled` 的背景、`outlined` 的边框、`ghost` 的 hover 背景、`link` 的文本和下划线切换到主题色体系。
- **理由**：用户明确要求按钮默认保持黑白灰，而不是一上来全部染成主题色。把主题色变成显式 opt-in，能降低 UI 噪声，也让主次按钮的强调策略更可控。
- **备选方案**：
  - 保持所有 `filled` 默认就是主题色：拒绝，原因是与用户新的视觉偏好相冲突。
  - 为主题色再创建新的 `variant`：拒绝，原因是会让 `variant` 膨胀，并把颜色维度错误建模为类型维度。

## 决策 8：theme 与 token 方案优先复用现有公共颜色 surface

- **决策**：按钮优先复用 `@deweyou-ui/styles` 已公开的颜色 token，例如 `--ui-color-brand-bg`、`--ui-color-brand-bg-hover`、`--ui-color-brand-bg-active`、`--ui-color-text-on-brand`、`--ui-color-focus-ring` 和 `--ui-color-link`；本期不计划扩展公共 radius、spacing、typography token，只调整现有 theme 值与页面基调。
- **理由**：`packages/styles/README.md` 已明确 v1 仅开放颜色 token 给消费者覆盖，布局、间距、圆角和排版仍由库内部控制。Button 作为基础组件，应先落在这个边界内，而不是直接扩大公共主题面。
- **备选方案**：
  - 立即新增 Button 专属公共 CSS 变量或全局 radius token：拒绝，原因是当前需求不足以证明必须开放更大的主题覆盖面。
  - 在 Button 内部写死一次性颜色值：拒绝，原因是这会破坏 token-first 的主题约束。

## 决策 9：页面与预览底色同步收敛为白底 / 黑底

- **决策**：website 与 Storybook 的页面底色收敛为浅色白底、深夜模式黑底，不再使用带明显情绪色彩的页面氛围背景。
- **理由**：如果页面底色本身带明显色相，会干扰用户判断 Button 的 `neutral` 与 `primary`。白底 / 黑底是最直接的中性载体，能让按钮颜色决策更清晰。
- **备选方案**：
  - 保持原暖色页面底色：拒绝，原因是会持续干扰按钮颜色判断。
  - 使用透明页面背景：可行，但当前白底 / 黑底更可预测，也更方便 Storybook 与 website 保持一致。

## 决策 10：内部评审与公开指导继续分层

- **决策**：`apps/storybook` 负责内部评审矩阵和边界状态；`apps/website` 负责官方使用指导与精选示例。
- **理由**：仓库 README、website 首页文案和宪章都已经把这两个 app 的职责分开。Button 重构延续这一路径，可以避免把内部状态矩阵和公开指导混成一体。
- **备选方案**：
  - 只更新 Storybook：拒绝，原因是违背宪章对 `website` 预览/指导更新的要求。
  - 只更新 website：拒绝，原因是 Storybook 更适合做完整状态矩阵与边界评审。

## 决策 11：`ghost` 与 `link` 使用文本流密度，而不是实体按钮密度

- **决策**：`ghost` 与 `link` 不再强行对齐 `filled` / `outlined` 的实体按钮内边距，而是使用更轻的文本流密度；其中 `ghost` 的默认圆角统一固定为 `0.4rem`。
- **理由**：用户明确指出这两类按钮经常出现在正文和密集工具区。如果继续沿用实体按钮的 padding，它们会在文本流中显得突兀，也会把“轻量动作”误做成“缩小版实体按钮”。
- **备选方案**：
  - 保持四类 `variant` 一致的盒子尺寸：拒绝，原因是会破坏 `ghost` / `link` 作为轻量动作的语义。
  - 为 `ghost` 单独开放 `shape`：拒绝，原因是用户想要的是统一默认圆角，而不是再引入新的公开配置轴。

## 决策 12：无可见文本不再自动触发布局特例

- **决策**：Button 内部只把“无可见文本”视为无障碍校验条件，不再自动把该场景做成方形按钮或其他 icon-only 专用布局。
- **理由**：基于 children 结构推断布局模式有启发式和命名错位问题。用户更倾向于抹平差异，让布局由各 `variant` 自己的 padding 和圆角规则决定，而不是继续隐藏一套 icon-only 特例。
- **备选方案**：
  - 保留自动 icon-only 方形按钮：拒绝，原因是语义不准确，也容易和未来显式 icon button API 冲突。
  - 新增本期公开 `square` 或 `contentMode` 字段：暂缓，原因是当前目标是先把隐式特例去掉，而不是再引入一个新的公开轴。

## 决策 13：icon 默认继承字号，并通过 wrapper 稳定占位

- **决策**：当 icon 未显式传入 `size` 时，默认使用 `1em` 继承周围字号；icon 渲染器统一包裹一个方形 inline wrapper，并在源 `viewBox` 非正方形时补方。
- **理由**：Button `size` 变化后，默认 icon 需要随字号一起变化，否则会出现文字变大而 icon 停留在固定像素大小的问题。wrapper 与 `viewBox` 补方则用于降低行高环境和非正方形源数据带来的占位偏差。
- **备选方案**：
  - 继续使用固定像素默认值：拒绝，原因是会直接破坏 Button 内 icon 的随字号缩放。
  - 在 icon 内部写死 `line-height: 1 !important`：拒绝，原因是问题主要发生在占位容器和父级文本排版环境，强制 `!important` 副作用更大。

## 决策 14：实体按钮高度回到 padding 驱动

- **决策**：`filled` 与 `outlined` 不再依赖固定 `min-height` 维持实体按钮高度，而是由字号、行高和统一四边内边距共同决定。
- **理由**：用户希望按钮尺寸逻辑更直接，也认为此前由 `min-height` 导致的“外盒子比内容高、但 icon 不会随之外扩”的模型不够自然。现有 size token 已经控制了字号和 padding，因此高度仍然可预测。
- **备选方案**：
  - 保留 `min-height` 兜底：拒绝，原因是会继续制造“内容是正方形但外盒子高度被额外托高”的差异。
  - 只对 icon-only 场景移除 `min-height`：暂缓，原因是当前先统一实体按钮的高度策略更简单，也已通过现有回归验证。

## 决策 15：常规按钮默认单行，`link` 保留文本流换行

- **决策**：`filled`、`outlined` 与 `ghost` 默认保持单行不换行；`link` 继续保留文本流中的可换行行为。
- **理由**：用户认为常规按钮换行观感太差，这也符合多数设计系统把按钮视为单行动作标签的处理习惯。`link` 更接近正文里的轻量动作，因此仍应允许在受限宽度下换行。
- **备选方案**：
  - 所有 variant 都允许换行：拒绝，原因是会让实体按钮更像段落容器而不是动作控件。
  - 所有 variant 都强制单行：拒绝，原因是会破坏 `link` 作为文本流动作的语义。

## 决策 16：常规按钮的超长文案使用单行省略

- **决策**：当 `filled`、`outlined` 或 `ghost` 处于受限宽度中时，超长文案默认使用单行省略号，而不是自动换行。
- **理由**：用户希望常规按钮保持单行动作观感。省略号能比多行按钮更稳定地维持按钮节奏，也更符合多数设计系统对工具栏按钮和实体按钮的处理方式。
- **备选方案**：
  - 自动换行：拒绝，原因是视觉上更像文案块，破坏按钮的动作感。
  - 直接裁切不显示省略号：拒绝，原因是用户无法判断内容被截断。

## 决策 17：验证以 `vp` 工作流和 app 构建为门禁

- **决策**：本功能的最低验证集为 `vp check`、`vp test`、`vp run storybook#build` 和 `vp run website#build`。
- **理由**：这是仓库宪章和现有 icon-package plan 的一致做法，既覆盖 package 级自动化检查，也覆盖 Storybook/website 两个消费面。
- **备选方案**：
  - 只跑 package 单测：拒绝，原因是无法覆盖评审面和集成面的回归。
  - 直接使用 `pnpm`、`vite` 或独立 `vitest` 命令：拒绝，原因是违反 Vite+ monorepo 纪律。
