## 用户场景与测试（必填）

### 用户故事 1 - 直接按组件消费（优先级：P1）

作为业务应用开发者，我希望只导入一个公开组件入口就能使用该组件，并自动获得它所需的样式，这样我不需要先导入整包，也不需要额外记住样式入口。

**为什么是这个优先级**：这是最终消费者最直接感知的体验，决定组件库是否具备按需接入和低心智负担的消费方式。

**独立测试**：选取任一已公开组件，仅通过该组件的独立入口接入到示例应用中，验证页面可正常渲染且样式完整，无需额外导入全量样式或根入口。

**验收场景**：

1. **假如** 消费者只需要单个公开组件，**当** 其按该组件的独立入口进行导入，**那么** 组件可以正常渲染且具备完整样式表现。
2. **假如** 消费者未额外导入组件库的全局组件样式入口，**当** 其使用单个公开组件，**那么** 不会因为缺少额外样式导入而出现裸样式或不可用状态。
3. **假如** 消费者仍通过组件库根入口使用该组件，**当** 升级到新产物后，**那么** 现有主入口消费方式仍保持可用。

---

### 用户故事 2 - 统一包级构建约定（优先级：P2）

作为组件库维护者，我希望每个包只保留真正必要的构建配置，并有清晰规则判断何时需要包级专用配置，这样我可以减少重复配置和维护成本。

**为什么是这个优先级**：这会直接影响后续新增包的成本和一致性，但不阻塞最终消费者的最核心使用路径，因此优先级次于消费体验。

**独立测试**：针对现有 `packages` 下的包逐个检查，确认每个包都能根据统一规则判断为“使用默认构建约定”或“保留专用构建配置”，且原因可解释。

**验收场景**：

1. **假如** 某个包只需要遵循仓库默认发包约定，**当** 维护者检查其构建配置时，**那么** 不需要额外保留冗余的包级专用配置文件。
2. **假如** 某个包存在明显不同于默认约定的产物或入口要求，**当** 维护者检查其构建配置时，**那么** 可以保留专用配置并说明其必要性。
3. **假如** 仓库中存在供团队参考的自有 `AGENTS.md`，**当** 维护者审查这些文档时，**那么** 其正文应统一为简体中文且不包含面向当前团队的英文主体说明。

---

### 用户故事 3 - 发布时输出可安装依赖契约（优先级：P3）

作为发版负责人，我希望所有计划发布到 npm 的包在产出清单中都携带可直接被外部宿主安装的依赖信息，包括由宿主提供的 React 运行时和解析后的具体版本号，这样发布后的包可以独立安装和消费。

**为什么是这个优先级**：该能力影响发版可靠性和外部安装体验，但建立在前两项消费与构建约定明确之后。

**独立测试**：对受影响包执行一次发布前清单审查，确认运行时宿主依赖、内部包依赖和外部依赖都已按发布要求表达为最终可安装的依赖契约。

**验收场景**：

1. **假如** 某个待发布包在运行时依赖 `react` 或 `react-dom`，**当** 审查其发布清单时，**那么** 这些依赖会被标记为宿主需安装的依赖，而不是由包自身隐式携带。
2. **假如** 某个待发布包依赖其他工作区包或目录化版本别名，**当** 审查其发布清单时，**那么** 对外呈现的是明确可安装的版本范围，而不是工作区占位符。

## 边界情况

- 当消费者同时混用根入口导入和单组件入口导入时，系统需要保证不会因为导入方式不同而出现缺失样式、重复要求手动补样式或公开入口失效。
- 当某个包不在运行时依赖 `react` 或 `react-dom` 时，系统不应强制其声明无意义的宿主依赖。
- 当某个包存在默认约定无法覆盖的多入口、资产复制或生成型产物需求时，系统需要允许其保留专用构建配置，并将其视为例外而非默认。
- 当发布流程无法把工作区占位符或目录化版本别名解析为明确版本范围时，系统必须阻止该包进入可发布状态。
- 当内部包之间存在相互依赖时，系统需要确保每个即将发布的包都能在外部宿主环境中被独立解析与安装。
- 当仓库中同时存在自有 `AGENTS.md` 和第三方依赖携带的 `AGENTS.md` 时，系统需要只改造仓库自有文档，不得把第三方依赖目录纳入中文化范围。

## 需求（必填）

### 功能需求

- **FR-001**：系统必须为 `@deweyou-ui/components` 中每个对外公开的组件提供稳定的独立消费入口，并保留现有整包入口的兼容消费方式。
- **FR-002**：用户必须能够仅通过单个公开组件入口完成组件接入，而无需额外导入该组件所需的样式资源。
- **FR-003**：系统必须将单组件消费的公开接入要求控制在“安装宿主依赖并导入目标组件入口”这一层级，不得再要求消费者额外导入该组件直接使用之外的整包样式入口或整包代码入口。
- **FR-004**：系统必须将 Vite+ 提供的统一构建约定定义为 `packages/` 下拟发布包的默认方案，只有在公开入口、产物结构或发布契约无法被默认约定满足时，才允许保留包级专用构建配置，并记录明确原因。
- **FR-005**：系统必须将所有在运行时依赖 `react` 或 `react-dom` 的拟发布包定义为“由宿主安装并提供对应运行时”的消费契约。
- **FR-006**：系统必须在每个拟发布包的对外依赖清单中清楚区分宿主依赖、包自身运行时依赖和仅开发期使用的依赖，避免消费者误判安装责任。
- **FR-007**：系统必须确保 `packages/` 下所有拟发布包在发布产物中不暴露 `workspace:*`、`catalog:` 或其他仅适用于仓库内部协作的版本占位表达。
- **FR-008**：系统必须将拟发布包之间的相互依赖解析为可被外部宿主安装的明确版本范围，确保单个包脱离工作区上下文后仍可安装。
- **FR-009**：系统必须为受影响包提供一致的发布前检查标准，至少覆盖公开入口可用性、样式可用性、宿主依赖声明和最终依赖版本表达四类结果。
- **FR-010**：系统必须为维护者提供可执行的判断依据，说明“是否需要专用构建配置”和“发布后依赖如何被最终表达”，使新包接入时无需重复试错。
- **FR-011**：系统必须把“非必要不保留包级专用构建配置”作为仓库级治理规则进行文档化，使后续新增包默认遵循统一约定而不是复制局部构建配置。
- **FR-012**：系统必须将仓库自有的根级、`apps/` 和 `packages/` 下 `AGENTS.md` 正文统一为简体中文，保证团队参考文档与仓库其他治理文档保持一致语言。
- **FR-013**：系统必须把第三方依赖目录中的 `AGENTS.md` 视为外部文件，不得把这些文件纳入本次中文化改造或验收范围。

### 关键实体（如功能涉及数据则填写）

- **公开组件入口**：面向组件库消费者暴露的可导入单元，关键属性包括入口标识、对应组件、是否兼容根入口消费，以及是否自带完成使用所需的样式契约。
- **包级构建约定**：用于判定某个包采用默认构建约定还是专用构建配置的治理规则，关键属性包括适用范围、例外条件、维护责任和发布前验证要求。
- **发布依赖契约**：包对外发布时呈现给宿主的依赖声明，关键属性包括宿主负责安装的依赖、包自身运行时依赖、内部包依赖的最终版本范围，以及是否满足独立安装条件。

## 假设与依赖

- 假设 `packages/` 下现有和未来新增的包都以 npm 发布为目标，`apps/` 下应用继续作为私有消费方存在。
- 假设 `react` 与 `react-dom` 的版本兼容责任由宿主应用承担，组件库只需清楚声明支持范围与安装责任。
- 假设整包入口仍需继续保留，以避免已有消费者因导入路径调整而被迫立即迁移。
- 依赖仓库发布流程能够在发布前生成可审查的包清单，以便验证占位版本是否已经被解析为最终版本范围。
- 假设仓库治理文档会把 Vite+ 统一构建约定设为默认要求，使后续新增包不再以“先复制一个包级构建配置”作为起点。
- 假设本次提到的 `AGENTS.md` 指仓库内由团队维护的指导文档，不包括 `node_modules` 等第三方依赖自带文件。

## 成功标准（必填）

### 可度量结果

- **SC-001**：100% 的公开组件可通过独立入口被消费，并在不额外导入组件样式入口的情况下完成可视与交互呈现。
- **SC-002**：100% 的拟发布包都能依据统一规则被判定为“使用默认构建约定”或“保留专用构建配置”，且维护者可在 5 分钟内说明判定原因。
- **SC-003**：100% 的运行时依赖 `react` 或 `react-dom` 的拟发布包，都会把这两项依赖表达为宿主负责安装的依赖契约。
- **SC-004**：在发布前审查中，100% 的拟发布包对外依赖清单不再出现 `workspace:*`、`catalog:` 或其他仓库内专用版本占位符。
- **SC-005**：至少选取 3 个代表性消费场景完成验证，其中包括单组件导入、整包导入兼容性和独立安装审查，且全部一次通过。
- **SC-006**：100% 的仓库自有 `AGENTS.md` 正文完成简体中文化，且发布范围内不包含第三方依赖目录中的同名文件。
