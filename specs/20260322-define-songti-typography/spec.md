# 功能规格：定义组件库宋体字体系统

**功能分支**：`20260322-define-songti-typography`  
**创建时间**：2026-03-22  
**状态**：草稿  
**输入**：用户描述："我想定一下整个组件的字体。我是想走宋体，基本都是系统自带的吧？同时也拥有不同字粗的字体。同时有比较贴合宋体的英文数字字体吗"  
**语言要求**：本文件及同级 `/specs/` 文档正文必须使用简体中文；代码标识符、命令、文件路径、协议字段和第三方 API 名称可保留原文。

## 澄清记录

### 会话 2026-03-22

- Q: 默认字体策略应仅限系统字体、默认系统字体加可选开源增强，还是直接以开源宋体为统一默认？ → A: 直接以开源宋体为统一默认，系统宋体用于兜底与字体尚未可用时的过渡显示。
- Q: 默认开源宋体应锁定单一发行名，还是锁定同源字体体系？ → A: 锁定思源宋体系作为默认目标，接受 `Source Han Serif SC` / `Noto Serif CJK SC` 这类同源发行。
- Q: 英文和数字默认跟随思源宋体系内置字形，还是额外配单独西文字体？ → A: 英文和数字默认跟随思源宋体系内置字形，不额外配单独西文字体。
- Q: 系统回退应锁定到哪些平台字体家族？ → A: `macOS` 回退锁定为 `Songti SC`、`STSong`；`Windows` 回退锁定为 `SimSun`、`NSimSun`。

## 用户场景与测试（必填）

### 用户故事 1 - 统一中文字体基调（优先级：P1）

作为组件库维护者，我希望整套组件默认采用统一的开源宋体方案，并在字体尚未可用或不可用时自动回退到系统宋体，这样既能获得更统一的中文气质，也能避免大字体资源影响首屏可读性。

**为什么是这个优先级**：这是整个字体方案的核心目标。如果默认中文字体方向没有先统一，后续字重层级和英文数字搭配都无法形成稳定体验。

**独立测试**：通过查看代表性的按钮、表单、提示、标题和数据展示组件，即可验证默认中文文本是否遵循统一字体方向，并在开源字体未就绪时仍由系统宋体保持可读和一致的过渡观感。

**验收场景**：

1. **假如** 组件库使用默认主题，**当** 查看代表性的基础组件预览，**那么** 所有中文文本都应优先呈现统一的开源宋体方向，而不是在不同组件间出现明显割裂的字体气质。
2. **假如** 开源默认字体尚未可用或当前环境无法使用该字体，**当** 打开同一组组件预览，**那么** 文本仍应按平台退回到约定的系统宋体家族，并保持整体中文气质一致。

---

### 用户故事 2 - 建立可复用的字重层级（优先级：P2）

作为设计师和组件使用方，我希望这套字体方案不仅有默认字体方向，还能提供清晰的字重层级，让正文、强调、标题和强强调在组件中有稳定且可复用的表达方式。

**为什么是这个优先级**：在统一字体方向后，第二个直接影响日常设计与开发效率的就是字重规则。没有统一字重层级，组件之间会很快回到各自为政。

**独立测试**：通过查看字体规范和代表性组件预览，即可验证是否存在明确、可区分的字重层级，以及这些层级能否覆盖常见文本用途。

**验收场景**：

1. **假如** 设计师需要为正文、次强调、标题和强强调选择不同层级，**当** 查阅字体规范，**那么** 应能找到清晰的字重分档及其适用场景。
2. **假如** 某个平台可用字重少于目标层级，**当** 查看相同内容层级的组件预览，**那么** 仍应保留足够清晰的主次差异，而不需要逐组件临时修正。

---

### 用户故事 3 - 处理中西文与数字混排（优先级：P3）

作为设计师和组件使用方，我希望英文和数字也有与宋体气质协调的默认搭配，这样在按钮、表单、统计数值、日期和状态文本中，中西文混排不会显得突兀。

**为什么是这个优先级**：组件库里大量场景都会同时出现中文、英文和数字。若只定义中文字体而不定义西文与数字搭配，最终观感仍会不稳定。

**独立测试**：通过查看包含中文、英文、数字、日期、价格和百分比的代表性样例，即可验证混排是否协调、易读，并且无需为常见场景单独找例外字体。

**验收场景**：

1. **假如** 一个组件同时包含中文、英文与数字，**当** 查看按钮、表单标签、统计值和表格字段，**那么** 西文与数字应直接跟随思源宋体系内置字形，与中文字体气质协调且保持稳定可读。
2. **假如** 组件展示日期、价格、百分比或版本号，**当** 在代表性样例中评审，**那么** 数字字形不应因字体搭配而显得明显跳脱或造成误读。

---

### 边界情况

- 当默认开源宋体在某个受支持环境中不可用，或在首屏阶段尚未可用时，必须按平台退回到约定的系统宋体家族。
- 当不同环境提供的可用字重数量不一致时，仍需保证正文与强调层级可辨认。
- 当组件中同时出现中文、英文大写、阿拉伯数字、货币符号、百分号和日期分隔符时，混排规则仍需保持稳定。
- 当文本出现在紧凑按钮、表单控件、表格列和状态标签等小尺寸场景中时，字体方案不能以牺牲辨识度为代价追求风格统一。
- 当遇到代码片段、固定宽度编号或其他需要等宽表现的信息时，必须有明确的例外规则，而不是强制沿用宋体方向。
- 仅使用键盘浏览预览或依赖辅助技术的用户，不应因为字体方案调整而失去焦点可见性、层级辨识或信息可理解性。
- 该字体方案在不同主题或 token 变体下都应保持同一字体方向与层级规则，而不是随主题切换出现另一套字体逻辑。

## 需求（必填）

### 功能需求

- **FR-001**：系统必须为整套组件库定义统一的默认中文字体方向，并以思源宋体系作为默认开源来源，接受 `Source Han Serif SC`、`Noto Serif CJK SC` 等同源发行。
- **FR-002**：系统必须为受支持环境定义清晰的字体回退优先级，确保默认开源字体尚未可用、不可用或加载失败时，中文文本仍保持宋体方向而不是退化到无关风格。
- **FR-013**：系统必须将平台回退家族明确锁定为 `macOS -> Songti SC, STSong`，`Windows -> SimSun, NSimSun`，并以此作为预览和验收基准。
- **FR-003**：系统必须定义可复用的字重层级，至少覆盖正文、次强调、标题和强强调四类语义，并为每类语义提供明确的使用规则。
- **FR-004**：系统必须定义中西文混排契约，覆盖中文、英文、数字、常见标点、日期、价格、百分比和版本号等代表性内容。
- **FR-005**：系统必须为英文与数字指定与宋体气质协调的默认搭配方向，并默认跟随思源宋体系内置字形，避免在按钮、表单、数据展示和状态文本中产生明显违和感。
- **FR-006**：系统必须明确本次字体方案覆盖的组件范围，至少包括内容文本、标题、操作控件、表单字段与数据展示；任何不在范围内的类别都必须显式说明。
- **FR-007**：系统必须明确哪些场景属于字体例外规则，包括代码、固定宽度信息或其他不适合使用宋体方向的内容，并说明这些例外如何与整体视觉保持协调。
- **FR-008**：系统必须满足明确的无障碍要求，包括紧凑尺寸下的可读性、仅依靠字重也能形成层级、键盘焦点不受影响，以及辅助技术用户不会因字体调整失去关键信息。
- **FR-009**：系统必须识别是复用现有全局字体角色，还是需要新增专门的全局字体角色，并保证命名、用途和适用范围在受影响 package 中一致。
- **FR-010**：系统必须给出候选字体可接受性的判断标准，至少覆盖系统可得性、字重覆盖度、中西文混排协调性和组件界面可读性四个方面。
- **FR-011**：系统必须在 `website` 中提供代表性预览，覆盖主要组件、主要字重层级、混排场景、例外场景和边界场景，以支持设计评审和使用方确认。
- **FR-012**：系统必须将默认开源候选范围收敛到思源宋体系，不将其他无关开源宋体与其并列为默认候选，除非后续另行评审替换。

### 无障碍与 UI 契约（UI 工作必填）

- **用户角色 / 参与方**：组件库维护者、设计师、消费组件的产品开发者、仅键盘用户、依赖屏幕阅读器或放大阅读的用户。
- **交互模型**：以静态阅读与预览评审为主，用户需要在不同组件、不同主题和不同文本场景中对比字体基调、字重层级和混排效果。
- **需要覆盖的状态**：默认正文、次强调、标题、强强调、禁用态文本、紧凑控件文本、混排文本、数据密集文本、例外等宽文本。
- **主题 / token 影响**：本次工作需要统一全局字体契约；颜色和其他视觉基础元素不应承担字体层级职责，但现有主题切换后仍需维持同一字体方向和例外边界。

### 关键实体（如功能涉及数据则填写）

- **字体角色**：定义组件库中的全局字体职责，包括以思源宋体系为目标的默认中文字体角色、跟随同一体系内置字形的英文数字角色，以及不纳入宋体方向的例外角色。
- **字重层级**：定义正文、次强调、标题和强强调等语义层级，以及它们在不同组件类别中的期望差异。
- **字体适用范围**：定义哪些组件类别默认跟随宋体方向，哪些内容类型需要按例外规则处理。
- **混排样例集**：用于评审中西文、数字、日期、价格、百分比和版本号等代表性文本场景的样例集合。
- **平台回退家族**：定义默认开源字体未就绪或不可用时，各平台必须使用的系统宋体家族集合。

## 依赖与假设

### 依赖

- 本功能依赖组件库能够提供一套基于思源宋体系的开源可商用默认宋体资源，并同时依赖 `macOS` 具备 `Songti SC` / `STSong`、`Windows` 具备 `SimSun` / `NSimSun`，作为未就绪或不可用时的回退。
- 本功能依赖组件库已经具备集中管理全局字体角色的能力，便于整套组件共享同一套字体契约。
- 本功能依赖存在可供设计师与开发者共同评审的预览入口，以展示代表性组件与混排场景。

### 假设

- 本期优先解决组件库默认主题中的中文字体方向问题，不扩展到营销页面或独立品牌活动页的专属排版风格。
- 本期默认优先保证中文气质统一，其次再通过系统宋体回退控制字体资源尚未可用时的阅读稳定性，而不是追求所有环境下逐像素一致的字形表现。
- 英文与数字的默认搭配必须优先跟随思源宋体系内置字形，并在回退到系统字体时保持可接受的一致性。
- 代码片段、等宽编号和其他需要固定宽度的内容继续按例外规则处理，不纳入宋体方向的默认覆盖范围。
- 若不同环境的精确字重可用性不同，只要用户可感知的层级关系稳定且可复用，即视为满足本期目标。

## 成功标准（必填）

### 可度量结果

- **SC-001**：100% 的代表性预览组件在默认主题下都采用统一的默认开源宋体方向，不需要为单个组件单独指定另一套默认中文字体。
- **SC-002**：100% 的受支持环境评审样例都能在默认开源字体未就绪或不可用时回退到 `macOS` 的 `Songti SC` / `STSong` 或 `Windows` 的 `SimSun` / `NSimSun`，不出现明显空白、不可读或退化为不协调字体风格的情况。
- **SC-003**：90% 以上的设计师或开发者能在 30 秒内根据规格说明或预览，判断正文、次强调、标题和强强调各自应使用的字重层级。
- **SC-004**：100% 的代表性混排样例都覆盖中文、英文与数字共同出现的场景，并在首次评审中达到可接受结论，不需要临时追加单场景字体例外。
- **SC-007**：100% 的代表性混排样例在默认方案下都不需要额外切换单独西文字体，即可完成中文、英文与数字的一致排版。
- **SC-005**：默认接入组件库的使用方不需要执行任何额外字体安装步骤，即可获得规定的中文字体方向和英文数字搭配。
- **SC-006**：100% 的范围内组件类别与例外类别都在规格说明和预览中被明确标注，使设计师和开发者能在 1 分钟内判断某类文本是否应跟随宋体方向。
