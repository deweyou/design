# 研究：定义组件库宋体字体系统

## 决策 1：默认字体能力落在 `@deweyou-ui/styles`，而不是 `website` 私有实现

- **决策**：默认宋体方案、平台回退顺序、字重层级和全局字体角色都以 `packages/styles` 为事实来源；`apps/storybook` 与 `apps/website` 只负责展示和评审。
- **理由**：宪章要求所有可复用能力必须先落在 package 中。字体系统属于整套组件共享的基础能力，不能只在 `website` 里单独声明，否则下游消费者无法复用。
- **备选方案**：
  - 只在 `apps/website` 中加载字体并定义样式：拒绝，原因是会让组件库正式消费面和 demo 面脱节。
  - 让各组件自行声明字体：拒绝，原因是会破坏全局一致性并增加维护成本。

## 决策 2：继续复用现有 `body` / `display` / `mono` 字体角色，不扩公开 typography token surface

- **决策**：继续沿用现有 `--ui-font-body`、`--ui-font-display`、`--ui-font-mono` 角色；本期不新增新的公开 typography token，也不把 typography 加入 `publicThemeTokens`。
- **理由**：`packages/styles/README.md` 已明确 v1 对消费者公开的主题覆盖面仅限颜色 token，typography 仍属于库内控制范围。本期目标是重设库默认字体系统，而不是扩大消费者可覆盖的公开主题契约。
- **备选方案**：
  - 新增公开 `--ui-font-serif` 或更细颗粒的字体 token：拒绝，原因是会扩大当前公开主题面，并增加未来 semver 负担。
  - 直接删除 `display` 角色：拒绝，原因是现有 `base.css` 仍以 `display` 角色承接标题字体职责，保留角色更有利于渐进演进。

## 决策 3：默认开源字体目标锁定为思源宋体系

- **决策**：默认开源宋体目标锁定为思源宋体系，接受 `Source Han Serif SC` 与 `Noto Serif CJK SC` 这类同源发行，不并列引入其他无关宋体家族。
- **理由**：规格澄清已经确认应优先用开源宋体作为统一默认，而思源宋体系在中文覆盖、字重数量和中西文一体设计上最符合本期目标。
- **备选方案**：
  - 只锁定单一发行名：可行，但没有必要在计划阶段把同源发行人为拆开。
  - 允许任意开源宋体并列候选：拒绝，原因是会导致视觉结果和验收基准漂移。

## 决策 4：`body` 与 `display` 统一指向同一套宋体方向，`mono` 保持例外

- **决策**：`--ui-font-body` 与 `--ui-font-display` 都切换到思源宋体系主栈；`--ui-font-mono` 继续保持现有等宽栈，用于代码、编号和其他例外信息。
- **理由**：用户希望“整个组件”的默认字体都走宋体方向。如果 `body` 保持宋体而 `display` 仍保留几何无衬线，标题和正文会立即出现风格割裂。`mono` 则属于规格已确认的例外类型，不应被强制并入宋体方向。
- **备选方案**：
  - 仅替换 `body`，保留现有 `display`：拒绝，原因是会让标题继续违背整体字体方向。
  - 让 `mono` 也使用宋体：拒绝，原因是会削弱代码和固定宽度信息的可读性。

## 决策 5：英文与数字跟随思源宋体系内置字形，不额外配西文字体

- **决策**：默认混排策略采用思源宋体系内置的英文与数字字形，不额外引入第二套西文字体或数字字体。
- **理由**：规格已明确希望英文、数字与中文“融合”，并且已澄清不希望为了默认方案再切换单独西文字体。统一家族能减少字体切换、降低版面节奏断裂，并简化回退策略。
- **备选方案**：
  - 中文用宋体，英文数字另配衬线西文字体：拒绝，原因是默认方案复杂度更高，且容易产生字面风格不一致。
  - 仅在数字密集场景单独配数字字体：暂不采用，原因是本期目标先建立统一默认，而不是引入场景特化字体。

## 决策 6：默认开源字体未就绪时，按平台退回系统宋体家族并保证立即可读

- **决策**：当默认开源字体尚未可用、加载失败或当前环境无法使用时，`macOS` 按 `Songti SC`、`STSong` 回退，`Windows` 按 `SimSun`、`NSimSun` 回退；首屏必须先显示系统回退字体，而不是等待开源字体导致空白文本。
- **理由**：这是规格澄清后的明确要求，也符合“中文字体包体较大，先用系统字体拉起可读内容”的目标。平台锁定后，评审和测试可以用同一套预期检查。
- **备选方案**：
  - 只写“系统衬线字体”不锁具体家族：拒绝，原因是验收基准不够稳定。
  - 以系统字体为默认，开源字体为可选增强：拒绝，原因是与已确认的产品方向不符。

## 决策 7：字体资源由 `@deweyou-ui/styles` 自带并随 package 分发，不依赖第三方 CDN

- **决策**：默认开源字体资源归属 `packages/styles`，并通过该 package 的构建产物一并分发；不把默认字体依赖放到第三方 CDN，也不只放在 `website` 私有静态资源目录。
- **理由**：这符合 package-first 原则，也能保证 Storybook、website 与任何下游消费者使用相同的默认字体来源。`packages/styles/scripts/copy-assets.mjs` 已经负责复制 CSS 和 Less 产物，本期可以在同一链路中纳入字体资产目录。
- **备选方案**：
  - 从第三方 CDN 动态拉取默认字体：拒绝，原因是会增加运行时外部依赖和不确定性。
  - 只把字体文件放在 `apps/website`：拒绝，原因是 Storybook 与 package 消费面无法共享同一资产来源。

## 决策 8：字重层级按语义建模，允许平台回退时退到最接近可用值

- **决策**：字重系统按“正文、次强调、标题、强强调”四个语义层级建模，默认开源字体优先提供完整层级；当平台回退家族可用字重较少时，允许退到最接近可用值，但必须通过示例评审证明层级仍可辨认。
- **理由**：规格已经要求四档层级，同时也承认不同平台的实际可用字重数量并不完全一致。语义层级比绑定某几个绝对数值更稳，也更符合组件库长期演进方向。
- **备选方案**：
  - 只定义两个层级（normal / bold）：拒绝，原因是不满足规格对四类语义的要求。
  - 强制所有平台精确匹配完全相同的字重值：拒绝，原因是系统回退字体的现实能力不足以保证这一点。

## 决策 9：验证门禁以 `packages/styles` 测试、Storybook 评审和 website 指导面共同构成

- **决策**：最低验证集为 `vp check`、`vp test`、`vp run storybook#build`、`vp run website#build`；并在 Storybook 与 website 中补充能观察中文、英文、数字、标题、正文、紧凑控件和例外等宽文本的评审样例。
- **理由**：字体系统变更属于共享基础样式更新，只跑 package 测试不足以覆盖真实消费面。宪章也要求自动化验证与人工评审面同时存在。
- **备选方案**：
  - 只更新 `packages/styles` 单测：拒绝，原因是无法覆盖实际页面观感和混排示例。
  - 只更新 website：拒绝，原因是缺少内部状态矩阵和回归检查入口。

## 决策 10：对外影响属于视觉契约更新，而不是新增公开 API

- **决策**：本期不新增 `@deweyou-ui/styles` 的导出路径和公开 props；主要影响是 `theme.css`、`theme-light.css`、`theme-dark.css` 与基础样式中的默认字体栈从现有无衬线方向切换到宋体方向，需要在发布说明中明确这是一项默认视觉契约更新。
- **理由**：当前公开 API 的关键在于包入口与主题覆盖面，而不是具体默认字体值。保持导出面稳定有利于控制实施范围，但视觉变化依然需要被评审和记录。
- **备选方案**：
  - 将其定义为新增公开 API：拒绝，原因是本期没有新增消费者调用入口。
  - 完全不记录对外影响：拒绝，原因是默认字体变化会直接影响所有消费者的视觉结果。
