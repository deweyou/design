# 研究：UI Monorepo 基础建设

## 决策：将可复用 React hooks 单独拆到 `packages/hooks`

**理由**：可复用 hooks 与框架无关 utilities 具有不同的依赖特征。单独的 hooks package 可以让 `packages/utils` 保持 React-free，明确依赖方向，也避免最底层 package 变成一个什么都往里塞的杂物桶。

**备选方案**：

- 把 hooks 放进 `packages/utils`：拒绝，原因是会混合 React 特定逻辑与框架无关逻辑。
- 将 `utils` 重命名为 `shared`：拒绝，原因是会削弱 package 边界，并鼓励无关代码不断堆积。

## 决策：将 Storybook 保持为 `apps/storybook` 下的 app 表面

**理由**：Storybook 是一个消费可复用 packages 的可运行环境，而不是一个本身对外发布的 package。把它视作 app，能保持 monorepo 模型一致，同时清晰区分公开文档与内部评审工具。

**备选方案**：

- 将 Storybook 放在 `packages/components` 内：拒绝，原因是 Storybook 不是可复用 package API 或发布表面的一部分。
- 只保留 website，不再使用 Storybook：拒绝，原因是对于 v1 而言，内部状态验证和快速组件评审仍然受益于专门工具表面。

## 决策：将 `apps/website` 设为公开文档与精选 demo 表面

**理由**：该项目希望输出一个有明确设计立场的组件系统。相比直接把 Storybook 作为公开入口，独立 website 更适合作为文档、设计说明、精选示例和有限 playground 的对外叙事表面。

**备选方案**：

- 直接把 Storybook 作为公开文档表面：拒绝，原因是它更倾向于展示实现驱动的穷举示例，而不是面向用户的精选指导。
- 长期将 Storybook 嵌入为公开网站体验：拒绝，原因是这会造成信息架构割裂，并削弱品牌与文档的一致性。

## 决策：使用 TypeScript 对象作为设计 token 的事实来源

**理由**：TypeScript token 源更易组织、校验、文档化和转换成其他输出，同时也更适合未来的文档场景，并且能与生成出的 CSS 主题文件兼容。

**备选方案**：

- 以 Less 变量作为主 token 源：拒绝，原因是运行时主题切换、结构化 token 文档和未来工具支持都会更困难。
- 以原始 CSS 文件作为主 token 源：拒绝，原因是 CSS 文件不适合作为结构化文档和转换工作流的源头。

## 决策：生成独立的 `theme-light.css`、`theme-dark.css` 和默认 `theme.css`

**理由**：这样可以保持主题模型显式，同时为未来按需拆分加载预留空间，并为消费者保留一个简单的默认导入路径。它也支持运行时切换亮色 / 暗色主题，而不要求消费者理解复杂的主题加载机制。

**备选方案**：

- 只生成一个合并主题文件：拒绝，原因是后续演进到更灵活加载方式时会更不干净。
- 一开始就要求消费者手动导入亮色与暗色主题文件：拒绝，原因是会提高 v1 的接入复杂度。

## 决策：要求消费者显式导入全局样式

**理由**：显式导入能保持 package 边界清晰，避免意外副作用，也更利于在文档和消费方应用中区分“全局设计基础”与“组件本地样式”。

**备选方案**：

- 由 components package 自动注入全局样式：拒绝，原因是这会隐藏一个全局依赖，并模糊 packages 间的职责归属。
- 在每个组件样式入口里直接导入完整主题文件：拒绝，原因是这会重复处理全局关注点，并增加未来主题演进复杂度。

## 决策：组件本地样式使用 CSS Modules，全局样式层保留在 `packages/styles`

**理由**：CSS Modules 为每个组件提供作用域隔离的实现样式，同时严格区分“组件本地展示”与“共享全局主题 primitives”。这正好支持目标自定义模型：根级 `className`、有限组件级 CSS 变量，以及内部类名不对外公开。

**备选方案**：

- 全部使用全局 Less 文件：拒绝，原因是会提高样式冲突风险，并削弱组件边界。
- 将内部组件类名直接作为公开样式 API：拒绝，原因是这会让消费者和实现细节紧耦合。

## 决策：v1 的公开主题面保持小而且只聚焦颜色

**理由**：这个库本身是有设计立场的。公开自定义应支持品牌适配和主题切换，但不能开放到让消费者重塑整套间距、尺寸、圆角、字号体系或动效语言。

**备选方案**：

- 一开始就暴露大量 token 自定义能力：拒绝，原因是这会削弱预设设计语言，并显著增加维护复杂度。
- 只暴露单个 `primary-color`：拒绝，原因是品牌表达、焦点态和链接样式需要一组相关语义色，而不是一个被过度复用的单一颜色。

## 决策：按组件目录组织源码，并优先使用标准 TSX 与直接 `classnames`

**理由**：每个组件一个目录的结构，在组件逐渐增加样式、测试、stories 和辅助文件后更容易扩展。标准 TSX 是 React 的常规编写方式，也比 `React.createElement` 更便于维护者阅读。直接使用 `classnames` 则避免了为简单类名拼接维护一个额外包裹工具。

**备选方案**：

- 将组件文件平铺在 `packages/components/src` 下：拒绝，原因是随着 package 增长会迅速变得嘈杂。
- 保留本地 `cx` helper：拒绝，原因是 `classnames` 已经足够解决问题，也更易理解。
- 继续在组件代码中使用 `React.createElement`：拒绝，原因是那只是一个临时 workaround，而不是期望的长期规范。
