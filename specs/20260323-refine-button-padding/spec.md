# 功能规格：优化 Button 间距平衡

**功能分支**：`20260323-refine-button-padding`  
**创建时间**：2026-03-23  
**状态**：草稿  
**输入**：用户描述："我想重新优化一下 button 的padding 问题,之前是让上下左右padding对齐,为了确保 icon only情况下是个正方形.但是现在导致了纯文本 button 看起来不协调, 因为line-height + 上下padding的缘故, 视觉上会上下大左右小的感觉."  
**语言要求**：本文件及同级 `/specs/` 文档正文必须使用简体中文；代码标识符、命令、文件路径、协议字段和第三方 API 名称可保留原文。

## Clarifications

### Session 2026-03-23

- Q: icon-only 场景应继续通过内容自动判断，还是改为显式能力建模？ → A: 为 `Button` 提供 `icon` prop；当 `icon` 有值且内容为空时进入 icon button 模式，同时导出 `IconButton` 组件。
- Q: `IconButton` 是否还需要挂载为 `Button.Icon` 入口？ → A: 保留 `IconButton` 作为主导出，同时让 `Button.Icon` 指向同一个组件。

## 用户场景与测试（必填）

### 用户故事 1 - 让纯文本按钮恢复视觉协调（优先级：P1）

作为设计师和组件使用者，我希望只包含可见文本的按钮在各尺寸下看起来横向与纵向留白均衡，而不是为了迁就仅图标按钮的方形外观，导致常规文本按钮出现“上下显大、左右显小”的视觉失衡。

**为什么是这个优先级**：纯文本按钮是最常见的使用场景。如果这个场景本身不协调，Button 的主视觉质量和组件可信度都会直接下降。

**独立测试**：查看所有代表性尺寸的纯文本按钮示例，确认在默认、hover、focus-visible 和 disabled 状态下，按钮仍保持清晰层级，同时不再出现明显的垂直留白偏重现象。

**验收场景**：

1. **假如** 一个 `filled` 或 `outlined` Button 仅包含单行可见文本，**当** 设计师查看各尺寸预览时，**那么** 按钮的上下与左右留白会呈现协调的视觉密度，而不是显得上下更厚重。
2. **假如** 同一按钮在默认、hover、focus-visible 和 disabled 状态之间切换，**当** 用户观察其内容位置与整体体量时，**那么** 不会感到文本在垂直方向被额外挤压或包裹。

---

### 用户故事 2 - 保持 IconButton 的方形触达（优先级：P2）

作为组件使用者，我希望没有可见文本、只有图标的按钮可以通过明确的 `IconButton` 场景获得方形外观和稳定点击区域，同时也能通过 `Button.Icon` 发现这个能力，而不是依赖组件去猜测内容结构，这样工具栏、操作组和紧凑操作位不会因为间距优化而失去一致性。

**为什么是这个优先级**：仅图标按钮是此前引入统一四边内边距的原因，必须保住这一场景的可用性，否则会把问题从文本按钮转移到工具型按钮。

**独立测试**：查看所有代表性尺寸的 `IconButton` / `Button.Icon` 示例，确认按钮在不同主题和交互状态下仍保持方形轮廓、图标居中和清晰的可访问名称，并验证两个入口的行为一致。

**验收场景**：

1. **假如** 使用者通过 `IconButton`、`Button.Icon` 或等价的 icon button 模式创建只有图标的按钮，**当** 用户查看各尺寸预览时，**那么** 按钮仍保持稳定的方形外观，不会因为文本按钮间距优化而变成横向或纵向拉长。
2. **假如** `IconButton` 或 `Button.Icon` 处于 hover、focus-visible 或 disabled 状态，**当** 用户进行鼠标或键盘操作时，**那么** 图标仍保持居中，按钮的触达区域和状态反馈保持稳定。

---

### 用户故事 3 - 带图标文本按钮不再被误判为方形模式（优先级：P3）

作为应用开发者，我希望带图标和文本的按钮通过明确的 `icon` 能力沿用文本优先的视觉密度，而不是被迫继承 `IconButton` 的方形约束，这样我可以在不额外调参的情况下获得自然的按钮比例。

**为什么是这个优先级**：图标加文本是高频组合。如果系统只区分“等边内边距”与“非等边内边距”，很容易让混合内容按钮出现过宽、过高或图文距离失衡的问题。

**独立测试**：分别检查前置图标加文本、后置图标加文本、较短文案和较长文案示例，确认它们与纯文本按钮属于同一视觉密度体系，同时不影响 icon-only 的独立表现。

**验收场景**：

1. **假如** Button 同时包含图标和可见文本，**当** 用户比较其与同尺寸纯文本按钮的体量时，**那么** 二者应表现为同一层级的内容按钮，而不是一个接近方形、一个接近文本流。
2. **假如** Button 在不同尺寸下展示图标加文本内容，**当** 设计师对比各尺寸层级时，**那么** 图标与文本的相对间距、按钮整体比例和点击目标都会保持一致的递进关系。
3. **假如** 仅使用键盘或辅助技术访问带图标的按钮，**当** 焦点进入按钮或读取按钮名称时，**那么** 间距调整不会影响原有语义、焦点可见性和可访问命名。

### 边界情况

- 当按钮只包含一个汉字、两个汉字或极短英文词时，系统如何避免按钮在视觉上显得过高过厚？
- 当按钮文案较长但仍需保持单行时，系统如何让文本内容与左右留白保持协调，而不是只压缩文字区域？
- 当按钮同时包含 `icon` 和文本内容时，系统如何保证它仍属于内容型按钮，而不是进入 `IconButton` 模式？
- 当按钮只有 `icon` 而没有可见文本时，系统如何同时满足 `IconButton` 的方形外观与可访问名称要求？
- 当按钮切换不同尺寸、不同颜色模式或不同形状时，系统如何保持新的间距规则具有一致的层级感？
- 当用户通过键盘聚焦或在 disabled 状态下查看按钮时，系统如何保证间距调整不会引入内容偏移、焦点遮挡或状态识别困难？

## 需求（必填）

### 功能需求

- **FR-001**：系统必须提供显式的图标承载能力，使按钮不再依赖对任意 `children` 内容结构的猜测来判断是否进入 icon button 模式。
- **FR-002**：系统必须为 `Button` 提供 `icon` prop；当 `icon` 有值且内容为空时，系统必须进入 icon button 模式，并允许使用者通过导出的 `IconButton` 组件以同等规则接入该模式。
- **FR-003**：纯文本 Button 在所有受支持尺寸下都必须呈现稳定的横向与纵向视觉密度，使其看起来是内容驱动的操作按钮，而不是被额外撑高的图形容器。
- **FR-004**：`IconButton` 在所有受支持尺寸下都必须保持方形外观、图标居中和稳定的点击目标。
- **FR-005**：带 `icon` 且同时包含可见文本的 Button 必须归入与纯文本 Button 一致的内容型密度体系，而不是自动继承 `IconButton` 的方形规则。
- **FR-006**：本次优化不得要求消费方新增手工调节四边间距的接入方式；开发者在现有 Button 使用路径下应能自然获得正确的内容密度。
- **FR-007**：本次优化可以在现有 Button 能力面上新增 `icon`、`IconButton` 与 `Button.Icon` 入口，但不得通过新增 `variant="icon"` 或继续依赖隐式内容推断来表达 icon button 语义。
- **FR-008**：`Button.Icon` 必须指向与 `IconButton` 相同的能力与行为契约；两个入口在尺寸、状态、无障碍要求和视觉结果上不得出现分叉。
- **FR-009**：系统必须在默认、hover、focus-visible、active 和 disabled 状态下保持按钮内容的稳定对齐，不得因状态切换出现明显的尺寸跳动或内容偏移。
- **FR-010**：系统必须继续满足 `IconButton` 的无障碍要求；任意没有可见文本、仅通过 `icon` 呈现动作的按钮都必须具备可访问名称。
- **FR-011**：系统必须识别并记录本次优化是否需要新增、复用或调整现有间距与尺寸 token；如果不需要新增 token，也必须明确复用策略。
- **FR-012**：系统必须在 `website` 中提供可评审的示例，覆盖纯文本 Button、带 `icon` 的文本 Button、`IconButton` 三类内容场景，以及主要尺寸、主要视觉类型和关键交互状态。
- **FR-013**：系统必须确保不同尺寸之间仍保有清晰层级；较大尺寸不能因为文本间距收缩而与较小尺寸看起来接近，较小尺寸也不能因为方形约束而显得过重。
- **FR-014**：系统必须明确本次优化的适用边界，说明纯文本 Button、带 `icon` 的文本 Button 与 `IconButton` 分别遵循哪类密度规则，以便设计师和开发者在评审时快速判断预期结果。

### 无障碍与 UI 契约（UI 工作必填）

- **用户角色 / 参与方**：消费组件库的应用开发者、评审按钮密度的设计师、组件维护者、仅键盘用户、屏幕阅读器用户
- **交互模型**：鼠标点击、键盘聚焦与激活、通过 `icon` 与内容是否为空决定是否进入 `IconButton` 模式、通过 `Button.Icon` 发现同一能力、在 `website` 中对比纯文本 Button / 带 `icon` 的文本 Button / `IconButton` 三类按钮
- **需要覆盖的状态**：纯文本、带 `icon` 的文本按钮、`IconButton`、`Button.Icon` 入口一致性、默认、hover、focus-visible、active、disabled、短文案、长文案、不同尺寸、不同颜色模式、不同形状
- **主题 / token 影响**：优先复用现有尺寸、间距、字体和焦点反馈契约；若现有 token 无法同时满足文本按钮平衡与 `IconButton` 方形约束，则必须明确新增或调整的必要性与影响范围

### 关键实体（如功能涉及数据则填写）

- **按钮图标入口**：描述 Button 如何通过 `icon` prop 承载图标内容，以及该入口如何与文本内容是否存在共同决定按钮密度模式。
- **按钮间距配置**：描述同一尺寸下按钮应呈现的横向与纵向留白关系，用于区分内容型按钮与方形图标按钮。
- **IconButton 场景**：描述当 `icon` 有值且内容为空时应进入的方形按钮模式，以及与导出 `IconButton` 组件和 `Button.Icon` 同义入口的一致契约。
- **按钮尺寸层级**：描述从 `extra-small` 到 `extra-large` 的体量递进关系，确保新的间距规则不会破坏既有层级认知。
- **按钮评审样例**：描述需要在 `website` 中展示的代表性按钮集合，用于设计评审、无障碍检查和后续回归验证。

## 依赖与假设

### 依赖

- 本功能依赖现有 Button 已支持的 `variant`、`size`、`shape`、颜色模式和原生按钮交互语义可继续复用。
- 本功能依赖存在可供设计师与开发者共同评审的 `website` 预览入口，用于对比不同内容场景下的按钮密度。
- 本功能依赖当前设计语言已经具备基础尺寸、字体、颜色和焦点反馈规则，可供本次间距优化复用或微调。

### 假设

- 本次优化的重点是现有 Button 的视觉密度与 icon button 入口建模，不通过新增 `variant="icon"` 来表达图标按钮。
- “纯文本 Button”指不带 `icon` 且包含可见文本的 Button；“带 `icon` 的文本 Button”归为内容型按钮，而不是方形图标按钮。
- 当 `icon` 有值且内容为空时进入 `IconButton` 模式，这一方形约束不再主导所有带文本按钮的四边留白。
- 本次优化允许在现有公开能力上新增 `icon`、`IconButton` 与 `Button.Icon` 入口，但不要求消费方新增手工间距调节方式。
- 纯文本 Button、带 `icon` 的文本 Button 和 `IconButton` 三类场景都需要在主要尺寸与主要交互状态下提供明确评审样例。

## 成功标准（必填）

### 可度量结果

- **SC-001**：100% 的纯文本 Button 代表性样例在设计评审中不再被标记为“上下显大、左右显小”的视觉失衡。
- **SC-002**：100% 的 `IconButton` 代表性样例在所有受支持尺寸下保持方形外观，且图标保持居中。
- **SC-003**：100% 的带 `icon` 的文本 Button 代表性样例在与同尺寸纯文本 Button 对比时，被判定为同一内容密度层级，而不是 `IconButton` 层级。
- **SC-004**：100% 的代表性按钮样例可以继续通过现有 Button 使用方式完成接入，无需新增消费方手工间距参数才能获得预期间距。
- **SC-005**：100% 的 `IconButton` 评审样例都具备可访问名称，且纯文本 Button、带 `icon` 的文本 Button、`IconButton` 三类样例都覆盖键盘聚焦检查。
- **SC-006**：覆盖纯文本 Button、带 `icon` 的文本 Button、`IconButton` 三类内容场景、主要尺寸、主要视觉类型和关键交互状态的评审样例完整度达到 100%。
- **SC-007**：100% 的 `IconButton` 与 `Button.Icon` 对照样例在尺寸、状态反馈、无障碍命名要求和视觉结果上保持一致，不出现单独分叉。
