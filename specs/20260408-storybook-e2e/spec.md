## 用户场景与测试（必填）

### 用户故事 1 - 组件交互行为回归测试（优先级：P1）

作为组件库开发者，我希望能在 Storybook 中自动化验证现有组件的交互行为，
以便在修改代码后快速发现功能回归。

**为什么是这个优先级**：交互行为回归是组件库最常见的问题来源，手动 QA
成本高且容易遗漏，自动化是最低成本的保障手段。

**独立测试**：可通过运行测试命令完整验证，交付物是可执行的测试套件
和通过率报告。

**验收场景**：

1. **假如** Button 组件已有各变体的 Story，**当** 运行 E2E 测试，**那么**
   所有 Story 均能正常渲染，点击交互触发预期行为，禁用态不可交互。
2. **假如** Popover 组件已有 Story，**当** 运行 E2E 测试，**那么**
   触发元素点击后浮层出现，Escape 键关闭浮层，点击外部关闭浮层。
3. **假如** Menu 组件已有 Story，**当** 运行 E2E 测试，**那么**
   菜单可打开/关闭，方向键可导航菜单项，禁用项不可选中，子菜单可展开。
4. **假如** Tabs 组件已有 Story，**当** 运行 E2E 测试，**那么**
   点击 Tab 可切换面板，键盘左右方向键可在 Tab 间导航，受控模式行为正确。
5. **假如** Typography / Color / Icon 等展示类组件已有 Story，**当** 运行
   E2E 测试，**那么** 所有 Story 均能正常渲染，无 JavaScript 错误。

---

### 用户故事 2 - 无障碍行为自动化验证（优先级：P2）

作为组件库开发者，我希望 E2E 测试能覆盖键盘导航和焦点管理，
以便确保无障碍行为符合宪章原则 II 的要求。

**为什么是这个优先级**：无障碍验证依赖手动测试成本极高，自动化覆盖可以
捕获大部分回归，但需要先建立 Story 级别的测试基础设施（依赖 US1）。

**独立测试**：可单独验证键盘交互测试用例通过与否。

**验收场景**：

1. **假如** Popover 组件支持键盘触发，**当** E2E 测试模拟 Tab 聚焦触发器
   并按 Enter，**那么** 浮层正确展开，焦点进入浮层内容区。
2. **假如** Button 组件处于禁用态，**当** E2E 测试尝试键盘激活，**那么**
   按钮不响应 Enter/Space，且 `aria-disabled` 正确设置。
3. **假如** Menu 组件已打开，**当** E2E 测试模拟方向键导航，**那么**
   焦点在菜单项之间正确移动，禁用项被跳过。
4. **假如** Tabs 组件渲染，**当** E2E 测试模拟左右方向键，**那么**
   活跃 Tab 在可用 Tab 间正确切换，`aria-selected` 属性同步更新。

---

### 用户故事 3 - 集成到 Vite+ 工作流（优先级：P3）

作为团队成员，我希望 E2E 测试能通过 `vp` 命令运行，与现有工作流一致，
以便不需要记忆额外命令。

**为什么是这个优先级**：工作流集成是体验问题，不影响测试本身的价值，
但对团队长期采用率很重要。

**独立测试**：可通过执行 `vp run storybook#test`（或类似命令）验证集成效果。

**验收场景**：

1. **假如** Storybook 已在后台运行，**当** 执行 E2E 测试命令，**那么**
   测试自动连接到 Storybook 并输出结果。
2. **假如** 有测试失败，**当** 命令执行完毕，**那么** 退出码非零，
   可被 CI 正确识别为失败。

---

### 边界情况

- 当 Storybook 未启动时，测试应给出明确错误提示而非静默挂起。
- 当 Story 中包含异步数据加载时，测试应等待加载完成后再断言。
- 仅键盘用户交互的测试应在无鼠标环境下可通过。
- CI 环境中 Playwright 浏览器依赖缺失时，应给出明确安装提示。

## 需求（必填）

### 功能需求

- **FR-001**：系统必须为 Button 组件的主要 Story 提供可执行的 E2E 测试，
  覆盖渲染正确性、点击交互和禁用态行为。
- **FR-002**：系统必须为 Popover 组件提供 E2E 测试，覆盖开启、关闭
  （Escape 键 / 点击外部）和焦点管理。
- **FR-003**：系统必须为 Menu 组件提供 E2E 测试，覆盖菜单开启/关闭、
  方向键导航、禁用项跳过，以及子菜单展开。
- **FR-004**：系统必须为 Tabs 组件提供 E2E 测试，覆盖 Tab 切换（点击和
  键盘方向键）、`aria-selected` 属性同步和受控模式行为。
- **FR-005**：系统必须为 Typography、Color、Icon 等展示类组件的全部 Story
  提供渲染正确性（smoke test），确保无 JavaScript 错误。
- **FR-006**：测试必须能在 CI 环境中无头（headless）运行，退出码正确
  反映通过（0）或失败（非 0）状态。
- **FR-007**：`@storybook/test-runner` 必须集成进 `apps/storybook` package，
  不引入对 `packages/` 下组件代码的侵入性修改。
- **FR-008**（公开 API）：本功能不涉及公开组件 API 变更；测试代码仅位于
  `apps/storybook` 内，不向 `packages/` 暴露任何接口。
- **FR-009**（无障碍）：E2E 测试必须覆盖 Button、Popover、Menu、Tabs 的键盘
  交互和焦点管理，并验证关键 ARIA 属性在交互前后的正确性。
- **FR-010**（token）：本功能不引入新的设计 token 或视觉基础元素；测试仅
  消费现有 Storybook stories，不修改组件样式或 token。

### 无障碍与 UI 契约

- **用户角色 / 参与方**：组件库开发者（本地运行测试）、CI 系统（自动执行）。
- **交互模型**：测试模拟鼠标点击、键盘输入（Tab / Enter / Escape）、焦点转移。
- **需要覆盖的状态**：默认渲染、交互后状态变化、disabled 态、浮层开启/关闭、
  Tab 激活/非激活、菜单项聚焦/禁用。
- **主题 / token 影响**：无 token 变更。

## 成功标准（必填）

### 可度量结果

- **SC-001**：Button、Popover、Menu、Tabs、Typography、Color、Icon 所有现有
  Story 测试通过率达到 100%。
- **SC-002**：E2E 测试套件可在 CI 环境中完整运行，总耗时不超过 5 分钟。
- **SC-003**：键盘交互测试覆盖 Button、Popover、Menu、Tabs 的核心交互路径，
  通过率 100%。
- **SC-004**：测试失败时 CI 退出码非零，开发者可从输出中明确定位失败的
  Story 和断言。
- **SC-005**：测试基础设施建立后，新增组件只需新增对应测试文件，无需修改
  测试运行配置。

## 假设

- 现有 `apps/storybook` 中已有 Button、Popover、Menu、Tabs、Typography、Color、Icon 的 Story 文件。
- CI 环境可安装 Playwright 所需的浏览器依赖（Chromium）。
- `@storybook/test-runner` 与当前 Storybook 版本兼容（需在 plan 阶段确认版本）。
- 不需要视觉回归测试（截图对比）；本期仅覆盖行为和渲染正确性。
