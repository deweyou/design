# 研究：新增 Text 排版组件

## 决策 1：`Text` 采用固定 `variant -> 默认节点` 映射，不引入额外 `as` 能力

- **决策**：`plain` 默认渲染 `span`；`body`、`caption` 默认渲染 `div`；`h1`、`h2`、`h3`、`h4`、`h5` 默认渲染对应原生标题节点。本期不新增 `as` 或其他多态根节点切换能力。
- **理由**：标题 `variant` 在实际使用里通常既承担视觉层级也承担文档结构职责。把 `h1`-`h5` 直接映射到原生 heading，能让默认写法更顺手，同时仍保留固定映射带来的可预测性。
- **备选方案**：
  - 新增 `as`：拒绝，原因是超出本期范围，也会让 `variant` 的默认语义失去约束力。
  - 让所有 `variant` 都退回 `div`：拒绝，原因是会让标题层级的默认语义缺失，增加日常使用负担。

## 决策 2：四个布尔样式字段命名为 `italic`、`bold`、`underline`、`strikethrough`

- **决策**：公开布尔字段统一命名为 `italic`、`bold`、`underline`、`strikethrough`，并允许任意组合叠加。
- **理由**：这组命名直观、与现有设计系统常见术语一致，并且都能清晰表达“这是视觉样式开关而不是新的文本类型”。
- **备选方案**：
  - 使用 `fontWeight` 一类非布尔字段：拒绝，原因是本期需求明确要求使用可叠加的布尔能力。
  - 使用 `lineThrough`：拒绝，原因是命名一致性和行业通用性弱于 `strikethrough`。

## 决策 3：标题 `variant` 默认同时承担视觉层级与原生 heading 语义

- **决策**：`h1`-`h5` 在 `Text` 中默认直接输出对应原生标题节点，不再要求消费方通过 `role="heading"` 与 `aria-level` 手动补语义。
- **理由**：标题组件最常见的使用方式就是同时表达标题样式和标题语义。把这件事做成默认行为，能减少消费方重复劳动，也能让 Storybook 与 website 的示例更接近日常页面写法。
- **备选方案**：
  - 继续通过 `div` + `role="heading"` / `aria-level` 组合补语义：拒绝，原因是默认体验更繁琐，且在多数页面里属于重复劳动。
  - 额外渲染隐藏标题节点：拒绝，原因是会制造不可见冗余结构，并破坏节点透传预期。

## 决策 4：`lineClamp` 只在正整数下生效，其他输入视为未设置

- **决策**：`lineClamp` 的有效输入为正整数。`undefined`、`0`、负数、非整数、`NaN` 或无穷值都按“未设置”处理，即文本保持自然长度显示。
- **理由**：规格要求非正整数输入必须有一致且文档化的处理方式，同时不能导致“显示为空”或不可读结果。把无效值归一为“不开启截断”比抛错或渲染异常更适合作为公开组件的默认容错策略。
- **备选方案**：
  - 对无效值直接抛错：拒绝，原因是会让普通内容渲染因为边缘输入而中断。
  - 将 `0` 视为隐藏全部文本：拒绝，原因是与“不可读结果必须避免”的规格要求冲突。

## 决策 5：共享排版尺寸与行高进入 `packages/styles`，而不是在 `Text` 内硬编码

- **决策**：现有 `packages/styles` 中的字体角色、字重和 `text-muted` 颜色继续复用；同时新增一组共享排版尺寸和行高变量，用于 `body`、`caption` 和 `h1`-`h5` 的视觉层级，再由 `Text` 消费。
- **理由**：`Text` 是公开排版组件，标题与说明文字的尺寸差异属于可复用视觉决策。宪章要求这类决策由共享 token / theme primitive 表达，不能长期停留在组件内的孤立常量。
- **备选方案**：
  - 在 `packages/components/src/text/index.module.less` 中直接写死全部字号和行高：拒绝，原因是与 token-first 原则冲突。
  - 完全复用浏览器默认 `h1`-`h5` 样式：拒绝，原因是设计层级无法通过默认 UA 样式稳定表达。

## 决策 6：`caption` 弱化强调复用 `text-muted` 语义色，不新建专属颜色体系

- **决策**：`caption` 的“更浅”表现默认复用现有 `--ui-color-text-muted` 语义色，并允许通过透明度或等价弱化方式完成视觉调整，但不单独引入新的 caption 专属颜色 token。
- **理由**：仓库已有成熟的弱化文本语义色 surface，可直接覆盖浅色 / 深色主题。当前需求的核心是说明文字低于正文的视觉强调，而不是创建第三套语义颜色层级。
- **备选方案**：
  - 为 `caption` 新增专用颜色 token：暂不采用，原因是当前没有独立语义需求，新增 token 会扩大维护面。
  - 仅依赖缩小字号而不调整颜色：拒绝，原因是与用户明确提出“颜色会稍微浅一点”不符。

## 决策 7：验证同时覆盖 package、Storybook、website 和 README

- **决策**：最小验证集为 `vp check`、`vp test`、`vp run components#build`、`vp run storybook#build`、`vp run website#build`，并同步更新 `packages/components/README.md`、`apps/storybook/src/stories/Typography.stories.tsx` 与 `apps/website/src/main.tsx`。
- **理由**：Text 既是 package 公开 API，又是用户可见排版能力。只做单测无法覆盖消费面示例，只更新展示面又无法稳定验证 props 契约和边界输入。
- **备选方案**：
  - 只补 package 单测：拒绝，原因是无法发现预览面与公开契约脱节。
  - 只更新 website：拒绝，原因是缺少 Storybook 的内部状态矩阵，不符合宪章的测试与预览双门禁。

## 决策 8：`color` 与 `background` 公开暴露 26 个颜色家族名，而不是色阶编号

- **决策**：`Text` 的 `color` 与 `background` 对外只暴露颜色家族名 union，不直接暴露 `50` 到 `950` 的色阶编号。颜色家族对齐 Tailwind colors 的 26 个家族：`red`、`orange`、`amber`、`yellow`、`lime`、`green`、`emerald`、`teal`、`cyan`、`sky`、`blue`、`indigo`、`violet`、`purple`、`fuchsia`、`pink`、`rose`、`slate`、`gray`、`zinc`、`neutral`、`stone`、`taupe`、`mauve`、`mist`、`olive`。
- **理由**：消费方真正关心的是高亮语义落在哪个颜色家族，而不是逐一理解色阶编号。把色阶收敛在内部映射里，可以降低 API 复杂度，也有利于后续统一主题切换策略。
- **备选方案**：
  - 直接暴露 `red-500` 一类完整色阶 token：拒绝，原因是公开面过宽，且把主题适配责任推给消费方。
  - 只提供少量语义色而不提供色卡家族：拒绝，原因是无法满足“直接用这套色卡”的要求。

## 决策 9：在 `packages/styles` 中维护 26 色族 x 11 色阶共享色卡 token

- **决策**：在 `packages/styles` 中新增一套与 Tailwind colors 结构对齐的共享色卡 token，覆盖 26 个颜色家族、每个家族 11 个色阶，供 `Text` 的 `color` / `background` 消费。
- **理由**：这是一类明确会复用的视觉 primitive，必须进入共享 token 层，而不是停留在组件私有样式或 Storybook 示例里。集中维护也方便后续其他组件复用同一色卡体系。
- **备选方案**：
  - 在 `Text` 内部私有维护色卡对象：拒绝，原因是违反 token-first 原则，也阻断其他组件复用。
  - 运行时从第三方库直接读取色卡：拒绝，原因是公开样式事实来源应收敛在本仓库的 styles package。

## 决策 10：`color` / `background` 的实际色阶按主题自动映射

- **决策**：浅色主题下，`color` 统一映射到 `800` 色阶、`background` 统一映射到 `100` 色阶；深色主题下，`color` 统一映射到 `200` 色阶、`background` 统一映射到 `900` 色阶，但始终保持同一颜色家族语义。
- **理由**：高亮文本的首要目标是可读性。把实际色阶交给主题映射，而不是固定死在单一深浅方案上，能让浅色 / 深色模式都维持稳定对比度和相同高亮意图。
- **备选方案**：
  - 不分主题固定色阶：拒绝，原因是深色模式下容易得到过暗文字或过浅背景。
  - 深色模式直接退回默认色：拒绝，原因是会让高亮语义在主题切换后丢失。

## 决策 11：共享色卡以 HSL 生成 26 色族 x 11 色阶，并输出稳定的 `Text` 语义变量

- **决策**：`packages/styles` 中的共享色卡以 HSL 生成 26 个颜色家族、每个家族 11 个色阶；`Text` 运行时不直接消费 `50`-`950` 编号，而是通过 `--ui-text-color-<family>` 与 `--ui-text-background-<family>` 这组稳定语义变量读取当前主题映射结果。
- **理由**：对组件来说，真正稳定的契约是“某个颜色家族在当前主题下对应的文字色和背景色”。把色阶收敛为语义变量后，组件无需感知具体 shade 编号，Storybook、website 和后续其他组件也能复用同一入口。
- **备选方案**：
  - 让组件直接拼接 `--ui-palette-red-800` 一类 shade 变量：拒绝，原因是会把主题映射细节暴露到组件实现里。
  - 只保留共享色卡而不输出 `Text` 专用语义变量：拒绝，原因是消费面仍然需要重复做主题映射选择。
