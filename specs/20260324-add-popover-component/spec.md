**功能分支**： `20260324-add-popover-component`  
**创建时间**： 2026-03-24  
**状态**： 草稿  
**输入**： 用户描述： "我想实现 popover 的组件"
**语言要求**：本文件及同级 `/specs/` 文档正文必须使用简体中文；代码标识符、命令、文件路径、协议字段和第三方 API 名称可保留原文。

## 澄清记录

### 会话 2026-03-24

- Q: Popover 需要哪些视觉基础能力？ → A: 默认提供 border、shadow 和指向触发目标的小箭头；圆角支持与 Button 对齐的 `rect` 与 `rounded` 两档；内容样式支持 `card`、`loose`、`pure` 三种模式，默认 `card`；显示与关闭都需要动画。
- Q: Popover 需要哪些触发、定位与状态控制能力？ → A: 支持 `click`、`hover`、`context-menu`、`focus` 触发，并允许配置多种触发方式；支持上、下、左、右、左上、左下、右上、右下位置；空间不足时需要自动回退；支持 `visible`、`onVisibleChange` 和 `offset`。
- Q: Popover 需要哪些内容与透传能力？ → A: `content` 为必填；支持 `disabled`；支持基础 DOM 属性透传如 `className`、`style`、`onClick`；并支持 `overlayClassName` 与 `overlayStyle` 透传到浮层外容器。
- Q: 默认触发方式应该是什么？ → A: 默认使用 `click`，`hover`、`focus`、`context-menu` 作为显式可配置触发方式。
- Q: 非受控模式是否需要显式初始可见状态入口？ → A: 需要，提供 `defaultVisible`，并与 `visible` / `onVisibleChange` 组成标准受控/非受控契约。
- Q: 面板内部点击后是否应默认自动关闭？ → A: 不应默认自动关闭；面板内部点击默认保持打开，关闭主要由外部点击、显式关闭动作或受控状态变化触发。
- Q: 同一页面多个 Popover 默认是否互斥？ → A: 默认不互斥；每个 Popover 独立管理自己的开关状态，是否互斥由上层业务自行编排。
- Q: Popover 打开后是否需要焦点陷阱？ → A: 不需要；Popover 保持非模态，不做焦点陷阱，但必须保证键盘用户可以进入内容区域、离开内容区域，并在关闭后返回触发元素。
- Q: 还需要补充哪些定位与挂载相关 props？ → A: 增加 `boundaryPadding` 和 `popupPortalContainer`；`boundaryPadding` 用于定义 Popover 与边界框之间的安全边距，默认值为 `16`；`popupPortalContainer` 用于指定浮层挂载的父节点。

## 用户场景与测试（必填）

### 用户故事 1 - 展示锚定浮层内容（优先级：P1）

作为使用该 UI 库的应用开发者，我希望在一个明确的触发元素旁展示补充信息或轻量操作面板，从而在不打断当前页面流程的前提下，为用户提供更多上下文和下一步动作。

**为什么是这个优先级**：这是 Popover 的核心价值。如果无法稳定地把内容锚定到触发元素附近并在需要时收起，该组件就无法支撑后续帮助说明、快速操作和轻量配置等常见场景。

**独立测试**：在一个示例页面中放置触发元素和补充内容，验证用户能够打开、查看并关闭 Popover，而不需要跳转页面或进入阻断式流程。

**验收场景**：

1. **假如** 页面上存在一个可交互的触发元素，**当** 用户激活该触发元素时，**那么** 补充内容会在其附近打开，并保持明确的视觉关联。
2. **假如** Popover 已打开，**当** 用户执行关闭动作或与其外部区域交互时，**那么** Popover 会被收起，且页面原有上下文继续可用。

---

### 用户故事 2 - 适配不同触发、定位与开关方式（优先级：P2）

作为组件消费者，我希望能够根据页面场景决定 Popover 的触发方式、首选展示位置、偏移量、默认开关状态以及由组件自身管理还是由使用方显式管理开关，从而在不同业务界面中复用同一套稳定契约，而不需要额外封装一次性行为。

**为什么是这个优先级**：只有当 Popover 的位置、触发和开关行为足够可预测，它才适合作为通用基础组件复用到筛选器、说明卡片、快捷操作和轻量表单等场景。

**独立测试**：在至少两个不同布局中分别验证多触发方式、默认开关和显式开关管理场景，确认 Popover 的展示位置、打开状态、关闭回调和偏移规则都符合文档化规则。

**验收场景**：

1. **假如** 使用方为 Popover 指定了首选展示方向和偏移量，**当** 用户打开 Popover 时，**那么** Popover 会优先按该方向展示，并在空间不足时采用文档化的兜底方式保持内容可见。
2. **假如** 使用方显式管理 Popover 的 `visible` 状态，**当** 状态发生变化时，**那么** Popover 的可见性与 `onVisibleChange` 反馈会与外部状态保持一致。
3. **假如** 使用方指定了 `popupPortalContainer` 或调整了 `boundaryPadding`，**当** Popover 打开并接近边界时，**那么** 浮层会挂载到指定父节点，并基于安全边距规则执行位置回退。

---

### 用户故事 3 - 满足无障碍与评审覆盖（优先级：P3）

作为仅键盘用户、屏幕阅读器用户以及组件维护者，我希望 Popover 的打开、内容访问和关闭过程都清晰可感知，并且主要状态与边界状态有可评审的预览覆盖，从而降低交互回归和无障碍缺陷进入发布面的风险。

**为什么是这个优先级**：Popover 很容易在焦点管理、读屏语义和边界布局上出现隐性回归。提前把这些行为纳入组件契约和预览覆盖，可以显著降低后续组件扩张时的维护成本。

**独立测试**：仅使用键盘完成打开、进入内容、关闭并返回触发元素的完整流程，同时查看预览面中主要状态和边界状态是否都可访问、可评审。

**验收场景**：

1. **假如** 仅键盘用户把焦点移动到触发元素上，**当** 其执行打开操作后，**那么** Popover 的出现和内容关系是可感知的，且用户能够继续进入内容区域完成交互。
2. **假如** 维护者打开组件预览面，**当** 其检查默认、长内容、边缘位置、多触发和交互内容等示例时，**那么** 能看到完整且一致的状态覆盖，而不需要阅读实现代码猜测行为。

### 边界情况

- 当触发元素靠近视口边缘，首选展示方向放不下完整内容时，Popover 会如何保持内容可见？
- 当 Popover 内容高度或宽度超过可用空间时，系统如何避免内容被完全裁切或脱离触发元素语义？
- 当 Popover 打开后，触发元素变为不可用、被移除或其所在内容被重新渲染时，系统如何避免残留孤立浮层？
- 当同一页面存在多个可打开的 Popover 时，系统如何在默认互不影响的前提下，仍让用户明确感知当前正在交互的是哪一个？
- 当同时配置了多种触发方式时，系统如何避免一次打开后又被另一种触发方式立即错误关闭？
- 当触发元素处于 `disabled` 状态时，系统如何阻止 Popover 打开并保持页面语义一致？
- 仅使用键盘的用户在打开 Popover 后，如何进入内容、退出内容并返回原始触发点？
- 当键盘用户离开已打开的 Popover 内容区域时，系统如何在不使用焦点陷阱的前提下保持交互可预测？
- 当使用方把 Popover 挂载到自定义父节点时，系统如何继续保证边界回退、层级和关闭行为可预测？
- 该功能在不同主题、颜色 token 和密度变体下如何保持层级感、可读性与触发关联性？

## 需求（必填）

### 功能需求

- **FR-001**：系统必须提供一个用于展示锚定式补充内容的 Popover 组件，使用户无需离开当前页面上下文即可查看额外信息或执行轻量操作。
- **FR-002**：系统必须支持通过明确的触发元素打开和关闭 Popover，并使重复触发、外部区域交互、关闭命令和显式状态更新都得到可预测的收起结果。
- **FR-003**：系统必须支持 `click`、`hover`、`context-menu` 与 `focus` 作为文档化触发方式，并允许使用方配置多种触发方式组合；其中默认触发方式为 `click`。
- **FR-004**：系统必须同时支持“由组件自身管理开关状态”和“由使用方显式管理开关状态”两种模式，并将 `defaultVisible`、`visible` 与 `onVisibleChange` 作为公开状态契约的一部分，清晰说明默认行为、状态变化时机和职责边界。
- **FR-005**：系统必须支持上、下、左、右、左上、左下、右上、右下这些文档化展示位置，并支持通过 `offset` 调整浮层与触发目标之间的间距。
- **FR-005a**：系统必须支持 `boundaryPadding` 作为公开定位约束，用于定义浮层与边界框之间的安全边距，默认值为 `16`。
- **FR-005b**：系统必须支持 `popupPortalContainer` 作为公开挂载契约，用于指定浮层挂载的父节点，并说明该设置对层级、边界检测和关闭行为的影响。
- **FR-006**：系统必须在首选位置空间不足时采用保持内容可感知、可访问的兜底展示策略，而不是简单裁切或脱离触发目标。
- **FR-007**：系统必须保持 Popover 与其触发元素之间的明确视觉和语义关联，包含指向触发目标的小箭头，避免用户在页面上丢失“该浮层属于哪里”的判断。
- **FR-008**：系统必须支持在 Popover 中承载说明文本、链接、按钮以及轻量表单控件等补充内容，并将 `content` 作为必填公开输入，同时为过长内容提供可预期的展示边界。
- **FR-009**：系统必须为每个受影响的 package 定义公开组件契约，包括组件命名、触发区域与内容区域的职责、展示位置选项、触发方式组合、`mode` 取值，以及开关状态由谁管理。
- **FR-010**：系统必须支持 `card`、`loose`、`pure` 三种内容样式模式，并以 `card` 作为默认值，用于区分标准边距、紧凑边距和无内边距内容面板。
- **FR-011**：系统必须提供默认带 border、shadow 的浮层视觉，并支持与 Button 一致的 `rect` 与 `rounded` 圆角选项，不提供大圆角模式。
- **FR-012**：系统必须支持显示与关闭动画，并确保动画不会削弱内容可读性或交互可预测性。
- **FR-013**：系统必须支持基础 DOM 属性透传，包括 `className`、`style`、`onClick` 等常见属性，并额外支持 `overlayClassName` 与 `overlayStyle` 透传到浮层外容器。
- **FR-014**：系统必须支持 `disabled` 状态；当触发元素或组件被标记为不可用时，不得打开 Popover，并且不得产生误导性的打开反馈。
- **FR-014a**：系统必须默认在 Popover 面板内部点击时保持打开状态；关闭主要由外部区域交互、显式关闭动作或受控状态更新触发。
- **FR-014b**：系统必须默认将每个 Popover 视为独立实例管理，不得隐式关闭同页其他 Popover；如需互斥行为，必须由上层场景显式控制。
- **FR-015**：系统必须满足明确的无障碍要求，包括正确的语义关系、可感知的打开状态、键盘打开与关闭、顺序导航、焦点返回以及辅助技术可识别的标签与描述。
- **FR-015a**：系统必须保持非模态键盘行为；打开后不得默认使用焦点陷阱，但必须保证键盘用户能够进入 Popover 内容、离开内容，并在关闭后返回到触发元素。
- **FR-016**：系统必须默认保持非阻断式体验，使页面其余内容仍然可见；同时必须清晰说明哪些场景适合使用 Popover，哪些场景应改用更强约束的模式。
- **FR-017**：系统必须识别是否需要新增设计 token、主题钩子或视觉基础元素；若现有表面、边框、阴影、圆角、间距和层级契约已足够，则必须优先复用现有能力。
- **FR-018**：系统必须在公开预览面中覆盖默认状态、打开状态、不同展示位置、长内容、交互内容、多触发方式、显式状态管理、`disabled`、边界位置以及关闭行为等主要与边界场景。
- **FR-019**：系统必须在触发元素不可用、被禁用、被移除或对应内容刷新时，确保 Popover 能以文档化方式关闭或不再展示，避免遗留孤立浮层。
- **FR-020**：系统必须说明 Popover 与 Tooltip、Dropdown、Dialog 等相邻模式的边界，帮助使用方在新增场景时选用正确组件，而不是把不合适的需求塞进 Popover。

### 无障碍与 UI 契约（UI 工作必填）

- **用户角色 / 参与方**：消费该组件的应用开发者、评审交互规则的设计师、鼠标与触屏用户、仅键盘用户、屏幕阅读器用户、维护组件文档与契约的维护者
- **交互模型**：通过触发元素执行打开与关闭；默认使用 `click` 触发，并支持 `hover`、`context-menu` 与 `focus` 作为显式配置触发方式；支持键盘激活、顺序导航、外部区域关闭和关闭后返回原始上下文；面板内部点击默认保持打开；Popover 不做焦点陷阱，但键盘用户必须可以进入和离开内容区域；在轻量信息展示与轻量操作面板两类场景中复用同一组件契约
- **需要覆盖的状态**：关闭、打开、触发元素 `hover`、触发元素 `focus-visible`、不同展示位置、长内容、含可交互内容、多触发方式、显式状态管理、`disabled`、边界位置回退、自定义挂载容器、主题切换下的层级与可读性
- **主题 / token 影响**：优先复用现有表面、文字、边框、圆角、阴影、间距和层级 token，并支持箭头与面板保持一致的视觉语言；若 Popover 需要额外视觉元素或层级语义，必须先证明现有 token 无法满足

### 关键实体（如功能涉及数据则填写）

- **Popover 触发元素**：用户打开或关闭浮层的入口，负责承载可见状态反馈，并与浮层内容保持语义关联。
- **Popover 内容面板**：出现在触发元素附近的补充信息或轻量操作区域，可能包含纯说明内容，也可能包含少量可交互控件，并通过 `mode` 控制内容内边距风格。
- **展示位置偏好**：描述 Popover 相对触发元素应优先出现的位置、箭头指向关系、`offset` 间距、`boundaryPadding` 安全边距，以及在空间不足时应如何回退以保持内容可见的规则。
- **挂载容器契约**：定义浮层默认挂载位置以及 `popupPortalContainer` 指定自定义父节点时必须保持的层级、边界检测和关闭行为规则。
- **开关状态契约**：定义 Popover 何时打开、何时关闭、由谁负责管理状态，以及 `defaultVisible`、`visible`、`onVisibleChange` 如何与各类触发方式协同工作的规则集合。

## 假设

- 本次功能的首个版本聚焦于“单个触发元素对应单个非阻断式浮层”的基础模式，不覆盖级联菜单、嵌套 Popover 或复杂多步流程。
- 公开能力覆盖 `click`、`hover`、`context-menu` 与 `focus`，默认使用 `click`，但它们仍需遵循统一且文档化的开关状态规则。
- Popover 适用于补充说明、快捷操作和轻量配置，不承担需要强制打断用户流程的确认、审批或高风险操作。
- Popover 保持非模态浮层定位，不承担 Dialog 式焦点陷阱职责。
- 公开预览面应足以让维护者和消费者评审主要状态与边界状态，而不需要阅读实现细节来推断组件行为。

## 成功标准（必填）

### 可度量结果

- **SC-001**：组件消费者能够在 15 分钟内，仅依据公开文档和预览示例，完成一个标准信息型 Popover 的接入，而不需要额外封装私有辅助逻辑。
- **SC-002**：规格中定义的主要状态与边界状态有 100% 可评审预览覆盖，并在功能进入实现阶段前被维护者确认齐备。
- **SC-003**：键盘验证流程中的关键步骤有 100% 可完成，包括打开 Popover、进入内容、执行关闭以及把交互上下文返回到原始触发元素。
- **SC-004**：在预先定义的边缘位置评审样例中，Popover 有 100% 能保持内容可见或按文档化规则回退，而不会出现无法理解的截断或脱锚。
- **SC-005**：至少两类代表性使用场景能够直接复用该组件契约完成交付，分别是补充说明型 Popover 与轻量操作型 Popover，而无需扩张组件边界。
