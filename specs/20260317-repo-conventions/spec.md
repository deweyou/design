# 功能规格：仓库规范统一

**功能分支**： `20260317-repo-conventions`  
**创建时间**： 2026-03-17  
**状态**： 草稿  
**输入**： 用户描述： "我想先来约束一下整个仓库的一些规范: 函数默认使用箭头函数; react 组件统一使用 tsx,而不要使用 createElement; 文件夹和文件命名都用小写+横线; utils 和 hooks 的项目结构是在 src 下,每个工具函数或者 hook单独一个目录, 里面存放 index 和 index.test, 就不需要跟src并行有一个 test了,单测放在对应目录里比较好维护; components 也是如此."

## 用户场景与测试（必填）

### 用户故事 1 - 统一共享代码的编写规则（优先级：P1）

作为贡献者，我希望仓库层面对函数写法、组件源码格式和命名方式有清晰一致的规则，从而新增代码不需要依赖团队口头约定也能保持统一风格。

**为什么是这个优先级**：如果没有统一基线，贡献者会持续引入混杂模式，导致评审成本上升、代码扫描效率下降。

**独立测试**：通过阅读规范文档，验证贡献者在不额外询问维护者的前提下，能否判断新代码应使用哪种函数风格、组件文件格式和命名方式。

**验收场景**：

1. **假如** 贡献者要创建一个新的非类函数，**当** 其查阅规范时，**那么** 规范会明确箭头函数是默认写法。
2. **假如** 贡献者要创建一个新的 React 组件，**当** 其查阅规范时，**那么** 规范会明确组件源码使用 TSX，而不是 `createElement` 风格。
3. **假如** 贡献者要给新文件或目录命名，**当** 其查阅规范时，**那么** 规范会明确名称必须为小写加连字符形式。

---

### 用户故事 2 - 统一共享源码目录结构（优先级：P2）

作为维护者，我希望共享 utilities、hooks 和 components 都遵循统一目录结构，从而仓库结构可预测、代码位置易查找。

**为什么是这个优先级**：结构不一致会降低维护效率，提高评审负担，并让跨 package 重构更容易出错。

**独立测试**：检查维护者是否只靠规格文档，就能推断出一个新的 utility、hook 或 component 应该放在哪里、目录应长什么样。

**验收场景**：

1. **假如** 贡献者新增一个 utility 或 hook，**当** 其创建源码结构时，**那么** 该单元会位于 `src` 下自己的独立目录中。
2. **假如** 贡献者新增一个 component，**当** 其创建源码结构时，**那么** 该组件会位于 `src` 下自己的独立目录中。
3. **假如** 贡献者查看某个 utility、hook 或 component 的期望目录结构，**当** 其查阅规格文档时，**那么** 该结构会明确包含一个 `index` 入口文件和同目录下的 `index.test` 测试文件。

---

### 用户故事 3 - 测试与源码共置（优先级：P3）

作为维护者，我希望共享源码单元的测试文件和源码放在一起，从而后续修改、评审与清理都能集中在一个位置完成。

**为什么是这个优先级**：测试共置可以降低测试文件游离的概率，也更容易让人理解某个源码单元与其覆盖范围之间的关系。

**独立测试**：验证规格是否明确禁止在 utilities、hooks 和 components 已经拥有独立目录的情况下，再维护并行的顶层测试目录。

**验收场景**：

1. **假如** 贡献者新增或修改了一个 utility、hook 或 component，**当** 其为该单元编写测试时，**那么** 测试文件会放在该单元自己的目录下，而不是并行的顶层测试树中。
2. **假如** 评审者查看一个新的共享源码单元，**当** 其检查目录结构时，**那么** 可以不进入单独测试层级就同时看到源码与测试。

### 边界情况

- 当已有文件或目录不符合新命名规则时，规范必须明确：新增或迁移的产物应采用新标准，而不是继续扩散第二套命名模式。
- 当某个源码单元暂时还没有自动化测试时，目录结构也必须预留该单元未来放置 `index.test` 的位置。
- 当贡献者新增带有键盘或屏幕阅读器敏感交互的组件时，共置测试与预览要求仍需覆盖相应交互状态。
- 当组件出现在多个 theme 或 token 场景下时，预览和测试仍应跟随同一个源码单元目录，而不是被拆散到不同结构中。

## 需求（必填）

### 功能需求

- **FR-001**：仓库规范必须将箭头函数定义为该策略作用范围内函数的默认编写形式。
- **FR-002**：仓库规范必须将 TSX 组件源码定义为该策略作用范围内 React 组件的默认格式。
- **FR-003**：仓库规范必须禁止将 `createElement` 风格作为新增或修改 React 组件实现时的默认模式。
- **FR-004**：仓库规范必须要求受治理区域中新建或迁移的文件和目录采用小写加连字符命名。
- **FR-005**：仓库规范必须明确共享 utility 位于 `src` 下，并且每个 utility 占用自己的独立目录。
- **FR-006**：仓库规范必须明确共享 hook 位于 `src` 下，并且每个 hook 占用自己的独立目录。
- **FR-007**：仓库规范必须明确共享 component 位于 `src` 下，并且每个 component 占用自己的独立目录。
- **FR-008**：仓库规范必须要求每个受治理的 utility、hook 和 component 目录都包含 `index` 入口文件和同目录 `index.test` 测试文件。
- **FR-009**：仓库规范必须明确：受治理的 utilities、hooks 和 components 的测试与源码放在同一目录中，而不是顶层并行测试目录。
- **FR-010**：仓库规范必须说明在增量治理过程中，贡献者如何判断已有产物应重命名、重定位，还是保持不动。
- **FR-011**：仓库规范必须明确任何受影响共享 package 的公开 API 面预期，避免内部目录重组后用户导入方式产生歧义。
- **FR-012**：仓库规范必须保留组件的无障碍评审要求，包括语义结构、焦点行为、键盘交互和辅助技术标签，即便源码与测试被重组也不能丢失。
- **FR-013**：仓库规范必须说明本次策略调整是否引入新的设计 token、主题 hook 或视觉 primitives，或者明确现有机制已足够。
- **FR-014**：仓库规范必须要求 `website` 对受治理组件提供预览覆盖，包括主要状态、边界状态以及禁用或错误状态，避免结构调整后失去可评审行为。
- **FR-015**：仓库规范必须定义贡献者和 agent 使用的 commit message 约定，包括必需的 subject 格式和 scope 使用方式。

### 无障碍与 UI 契约（UI 工作必填）

- **用户角色 / 参与方**：新增代码的贡献者、评审结构的维护者、仅键盘用户、屏幕阅读器用户、验证导入面的 package 消费者、验证预览和 token 的设计师
- **交互模型**：通过文档指导编写代码、按目录发现源码、通过键盘评审组件行为、源码与测试共置、通过预览检查组件状态
- **需要覆盖的状态**：默认、hover、focus-visible、disabled、error、empty、loading，以及受治理组件相关的键盘导航状态
- **主题 / token 影响**：默认假设不需要新增 token；除非后续实现阶段发现缺口，否则应明确继续复用现有 theme 与 token 模式

### 关键实体

- **规范规则集**：覆盖函数风格、组件源码格式、命名模式、目录结构、测试共置和采纳方式的仓库级规则集合。
- **提交规范**：贡献者和 agent 在仓库历史中记录变更时必须使用的 commit subject 模式。
- **受治理源码单元**：受该规则集约束的 utility、hook 或 component，预期位于自身 `src` 子目录中，并与入口和测试文件共置。
- **采纳决策**：在保持新代码遵循标准的前提下，贡献者或维护者针对已有产物选择重命名、迁移或延后处理的决定。

## 假设

- 该策略优先作用于新增或有实质性改动的 utilities、hooks 和 components，从而支持仓库逐步采纳。
- 即使内部目录结构会随着治理逐步变化，对外公开导入方式仍应保持清晰可理解。
- 对受治理源码单元而言，共置单测仍是默认要求；更广泛的集成或端到端覆盖在必要时仍可放在其他位置。

## 成功标准（必填）

### 可度量结果

- **SC-001**：评审时，受治理区域内 100% 新建或有实质更新的 utilities、hooks 和 components 都遵循已文档化的命名与目录规则。
- **SC-002**：贡献者只看规格文档即可在 2 分钟内判断一个新受治理源码单元的正确目录结构和文件命名模式。
- **SC-003**：规范落地后，抽样 PR 中至少 90% 的变更，评审者无需额外澄清即可判断其函数风格、组件文件格式与测试位置是否符合规范。
- **SC-004**：至少 90% 新增的受治理源码单元在合并时都包含位于预期目录中的共置测试覆盖。
- **SC-005**：规范采纳后，所有受治理组件仍能在项目预览面中展示可评审的主要状态和边界状态。
