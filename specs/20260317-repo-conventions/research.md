# 研究：仓库规范统一

## 决策 1：将约束拆分到 lint、agent 指导和评审门禁中执行

**决策**：对可机械识别的规则使用仓库级 lint；对贡献者与 agent 的行为要求使用根级和 package 级 `AGENTS.md`；对结构性和无障碍敏感规则，通过评审与预览要求补充执行。

**理由**：不是每条仓库规范都适合用同一种方式强制。禁止在常规组件开发中使用 `React.createElement`、限制命名方式，这些规则可以自动化；但目录拓扑和渐进迁移决策则依赖上下文。分层治理既避免高估 lint 能力，也能把人类可读的规则放在最贴近代码的位置。

**备选方案**：

- 全部通过 lint 解决：拒绝，原因是目录结构和迁移规则很难仅通过内建 lint 配置可靠表达。
- 全部只写在文档里：拒绝，原因是纯描述型规则很容易漂移，且不能有效防回归。
- 只在 code review 中执行：拒绝，原因是评审者不应把时间浪费在抓取简单机械错误上。

## 决策 2：将 `createElement` 限制和箭头函数默认值作为自动化候选规则

**决策**：对受治理的 React 组件源码加入禁止 `createElement` 风格的自动化检查；在可以安全表达的范围内，对函数默认写法加入自动化约束；对于自动化难以准确识别的合理例外，保留文档化评审规则。

**理由**：这些是用户提出的高信号规则，而且仓库当前已有相应指导背景。它们直接影响贡献者日常写代码的方式，因此优先做部分自动化可以最快产生价值。

**备选方案**：

- 完全不做自动化，只依赖 `AGENTS.md`：拒绝，原因是仓库已有 lint 基础设施，这类规则足够明显，值得做自动化。
- 不区分例外地强制所有函数都用箭头函数：拒绝，原因是某些文件可能需要例外，例如框架入口点、类方法或生成代码，因此策略必须先明确作用范围和例外。

## 决策 3：将目录命名和源码/测试共置作为仓库策略，并辅以选择性自动化

**决策**：先将小写连字符命名和 `index` / `index.test` 共置结构定义为仓库策略，再只对那些 package 边界清晰、路径模式稳定的区域增加自动化检查，尽量减少误报。

**理由**：当前共享 packages 中仍有 `src` 与 `tests` 并行目录等历史结构，因此这个功能需要渐进迁移路径，而不是一次性重写。先用策略定义方向，再对新增受治理单元加有选择性的路径校验，可以在不制造大规模低价值迁移噪音的前提下逐步收敛。

**备选方案**：

- 立即让所有不符合规范的历史路径全部失败：拒绝，原因是当前仓库仍存在历史模式，这会制造一波噪音极大的迁移工作。
- 完全不对路径做自动化：拒绝，原因是当迁移范围明确后，对新增受治理源码单元做定向检查仍然很有价值。

## 决策 4：更新根级和 package 级 `AGENTS.md`，让规则在编写时即可被看到

**决策**：在根级 `AGENTS.md` 和相关 package 的 `AGENTS.md` 中增加或完善规范说明，使贡献者和 agents 在编辑 package 代码前就能看到这些规则。

**理由**：仓库已经把 `AGENTS.md` 用作日常操作指导面。沿用这一机制的成本更低，而且它可以与 lint 形成互补，解释规则为什么存在、哪些情况允许例外。

**备选方案**：

- 只把策略写在 feature spec 中：拒绝，原因是 spec 不是日常编写时最直接的阅读面。
- 只写进 package README：拒绝，原因是当前仓库已经依赖 `AGENTS.md` 作为主动编码约束入口。

## 决策 5：将 demo app 作为合规示例，而不是规则定义本身

**决策**：像 `apps/website` 这样的 demo app，如果当前还在示范被禁止模式，则需要同步对齐；但规则的事实来源仍应是仓库治理文件和共享 package 指导，而不是示例代码本身。

**理由**：示例代码会直接影响贡献者的行为。如果仓库禁止在常规组件开发中使用 `createElement`，而网站示例仍在这么做，那么策略本身就会失去说服力。更新示例是必要的，但示例应该是规则的体现，而不是规则定义。

**备选方案**：

- 先不动现有示例，留到以后再改：拒绝，原因是相互矛盾的示例会从一开始就削弱规则采纳。
- 只通过示例代码表达策略：拒绝，原因是示例无法充分表达例外、迁移规则和执行边界。
