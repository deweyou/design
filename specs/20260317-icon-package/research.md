# 研究：为 UI 组件库新增图标包

## 决策 1：使用 `tdesign-icons-svg` 作为初始 SVG 资产来源，但不把它暴露为公开消费者 API

- **决策**：基于从 `tdesign-icons-svg` 导入并筛选后的图标资产构建 `@deweyou-ui/icons`，同时将公开契约限制在 Deweyou UI 自己定义的图标名称、props 和命名导出上。
- **理由**：这样可以保留对命名稳定性、无障碍默认值、失败行为以及未来替换图标来源的控制力，同时也符合用户“未来可逐步转向自有 SVG 资产”的偏好，而无需一开始就手工绘制全部图标。
- **备选方案**：
  - 直接暴露 `tdesign-icons-react`：拒绝，原因是会把上游 React API 泄漏到 Deweyou UI 的公开契约中，并削弱后续迁移到自有 SVG 资产的可控性。
  - 从第一天起就在仓库里手工维护原始 SVG 文件：拒绝，原因是当初始目录尚未稳定时，这种方式维护成本更高、贡献体验更差。
  - 使用 sprite 或 icon font：拒绝，原因是会削弱按图标 tree-shaking、无障碍清晰度和 package-first API 体验。

## 决策 2：初始基础目录只使用同一套 TDesign 图标家族

- **决策**：初始基础图标集全部来自 `tdesign-icons-svg`，而不是混用多个视觉家族。
- **理由**：规格要求图标能够在同一设计系统中并排使用时保持足够一致。单一来源家族能更稳定地保持线宽、圆角处理和视觉重心一致。
- **备选方案**：
  - 立即混用多个第三方图标家族：拒绝，原因是会在基础 package 契约尚未稳定前就引入视觉漂移和额外治理成本。
  - 一开始就自定义完整图标集：拒绝，原因是会在目录治理和 API 还未验证前大幅增加前期设计与资产产生成本。

## 决策 3：发布一个 package 自有的 registry，并同时支持两种消费模式

- **决策**：实现一个由 package 自己维护的图标 registry，同时驱动泛型 `Icon` 组件与基于同一 canonical map 生成的官方命名导出。
- **理由**：这与澄清后的要求一致，即同时支持两种使用方式并保证行为一致。它也为 package 提供了一个集中处理不支持图标名错误和目录元数据的地方。
- **备选方案**：
  - 只支持命名导出：拒绝，原因是目录驱动文档和动态选择图标会变得笨重。
  - 只支持泛型查找入口：拒绝，原因是命名导出更有利于发现、IDE 自动补全和组件代码中的使用体验。

## 决策 4：将不支持的图标名视为契约错误

- **决策**：当泛型入口收到不支持的图标名时，在运行时抛出描述性错误；同时通过 TypeScript 类型在开发期拦截绝大多数非法输入。
- **理由**：澄清后的失败策略就是显式失败。将类型收窄与运行时错误结合，既能覆盖常规误用，也能处理 escape hatch 场景，而不会制造含糊不清的视觉输出。
- **备选方案**：
  - 回退为占位图标：拒绝，原因是会掩盖目录治理问题，并让 UI 表现变得模糊。
  - 直接不渲染：拒绝，原因是静默缺失比显式错误更难发现和排查。

## 决策 5：在新增图标专属全局 token 之前，优先复用现有主题语义

- **决策**：默认图标颜色通过 `currentColor` 继承周围文本颜色；尺寸优先映射到 package 契约中的标准尺寸；除非评审暴露出明确缺口，否则暂不新增全局图标 token。
- **理由**：styles package 已经暴露了语义化文本色与链接色 token。复用它们可以让图标自然与周围内容对齐，也能避免过早扩大主题面。
- **备选方案**：
  - 立即新增专门的全局图标颜色 token：拒绝，原因是当前需求尚不足以证明必须单独开辟一套 token 家族。
  - 在 package 内部写死图标颜色：拒绝，原因是这违背 token 驱动主题系统，也会削弱复用能力。

## 决策 6：同时覆盖内部评审面与公开指导面

- **决策**：使用 `apps/storybook` 进行内部目录评审和状态覆盖；使用 `apps/website` 提供官方 package 使用说明与精选示例。
- **理由**：仓库宪章要求存在评审可见的预览表面，而 website 规则又明确官方使用文档应属于 `apps/website`。
- **备选方案**：
  - 只用 Storybook：拒绝，原因是会让 package 官方指导脱离项目既定公开文档表面。
  - 只用 website：拒绝，原因是对于大规模目录浏览而言，Storybook 更适合作为内部状态评审工具。
