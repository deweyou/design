# 功能规格：为 UI 组件库新增图标包

**功能分支**： `20260317-icon-package`  
**创建时间**： 2026-03-17  
**状态**： 草稿  
**输入**： 用户描述： "我先想把 icon 的问题解决了,就是新开个 icon 的packages, 来实现这个 ui 组件库的 icon 集合. 我的想法是能不能直接使用 iconify 来实现"

## 需求澄清

### 会话 2026-03-17

- 问题：图标包是否只暴露官方维护的图标目录、允许任意第三方图标名，还是两者都支持？  
  回答：只暴露官方维护的受支持图标目录。
- 问题：消费者应使用单图标导出、统一泛型入口，还是两者都支持？  
  回答：同时支持泛型 `Icon` 入口和官方命名导出，两者都基于同一套目录。
- 问题：当消费者请求一个不受支持的图标名时，包应显式失败、显示占位符，还是直接不渲染？  
  回答：显式失败，并提供清晰且文档化的错误，而不是静默回退。
- 问题：初始图标目录应尽量做大、仅覆盖当前组件所需，还是先提供一组通用基础集？  
  回答：先提供一组共享的基础图标集，覆盖常见动作、状态、导航与反馈图标。
- 问题：共享图标 props 契约应包含什么？  
  回答：保留 `name`、`className`、`style`、`label` 和 `size`；不单独暴露 `decorative`。
- 问题：图标尺寸如何表达？  
  回答：同时支持数字尺寸和 `extra-small`、`small`、`medium`、`large`、`extra-large` 这些标准尺寸，其中默认值为 `medium`。
- 问题：命名图标导出应如何命名？  
  回答：统一使用 `XxxIcon` 命名约定。

## 用户场景与测试（必填）

### 用户故事 1 - 使用共享图标集（优先级：P1）

作为使用该 UI 库的应用开发者，我希望有一个拥有稳定目录和一致使用规则的专用图标包，从而可以在组件和产品界面中使用图标，而不必重复维护资产或发明临时模式。

**为什么是这个优先级**：这是该功能的核心价值。如果没有一套可复用的共享基础图标集，组件库仍然缺少一个标准化的视觉表达方式来表示动作、状态、导航和反馈。

**独立测试**：从共享图标目录中选取多个命名图标，在示例 UI 中渲染，并确认它们都遵循同一套文档化使用契约。

**验收场景**：

1. **假如** 共享图标包已可用，**当** 开发者从文档化目录中选择一个图标时，**那么** 该图标可以通过图标包的公开 API 被渲染，而无需直接导入原始图稿文件。
2. **假如** 同一屏幕中使用了多个图标，**当** 它们以相同尺寸和视觉角色渲染时，**那么** 它们在视觉上应足够一致，能够并排用于同一设计系统中。

---

### 用户故事 2 - 快速发现可用图标（优先级：P2）

作为组件开发者或设计师，我希望有一个可浏览并展示全部可用图标的预览面，从而在使用前就能快速找到目标图标并确认其外观。

**为什么是这个优先级**：只有当贡献者能快速发现已存在的图标并确认预期效果时，共享图标集才真正降低协作摩擦。

**独立测试**：打开预览面，浏览图标目录，并在一次评审中找到目标图标及其使用说明。

**验收场景**：

1. **假如** 贡献者需要一个状态或动作图标，**当** 其打开图标预览面时，**那么** 能浏览到可用图标、稳定名称和代表性视觉示例。
2. **假如** 贡献者正在评估图标使用方式，**当** 其查看图标包文档时，**那么** 能看到支持的使用模式、无障碍要求和已声明的限制。

---

### 用户故事 3 - 安全处理无障碍与缺失图标场景（优先级：P3）

作为维护者，我希望图标使用规则能够覆盖无标签与有标签图标，以及不受支持图标名的处理方式，从而保证该库在无障碍和错误场景下都是可预测的。

**为什么是这个优先级**：图标包经常成为无声无息引入无障碍回归或运行时惊喜的来源，尤其是在图标缺失或有意义图标未提供标签时。

**独立测试**：评审有标签和无标签图标示例，然后尝试使用一个不在支持目录中的图标引用，并确认系统行为是文档化且确定的。

**验收场景**：

1. **假如** 某个图标表达了超出周围文本本身的含义，**当** 开发者在库中使用它时，**那么** 使用契约会要求提供可访问名称或等价文本替代。
2. **假如** 开发者引用了一个不在受支持目录中的图标，**当** 图标包收到该请求时，**那么** 系统会给出清晰且文档化的失败行为，而不是含糊不清的静默结果。

### 边界情况

- 当两个被请求的图标语义接近但视觉风格不同，应如何处理？
- 当请求的图标名拼写错误、已移除或根本不在受支持目录中时，系统应如何处理？
- 当同一个图标在某个场景中无标签、另一个场景中有标签时，键盘用户和屏幕阅读器用户会遇到什么？
- 在支持的 themes、颜色角色和常见尺寸范围下，该功能如何保证图标仍清晰可辨且视觉对齐？

## 需求（必填）

### 功能需求

- **FR-001**：UI 组件库必须在共享 package 工作区中提供一个专门的图标 package。
- **FR-002**：图标 package 必须暴露一套稳定的官方受支持图标目录，每个图标都具有唯一的面向消费者名称。
- **FR-002a**：初始官方图标目录必须优先覆盖一组共享基础图标，包括常见动作、状态、导航与反馈图标，再视情况扩展到功能特定图标。
- **FR-003**：消费者必须能够通过图标包文档化的公开 API 渲染任何受支持图标，而无需直接导入或管理原始图标资产。
- **FR-004**：图标 package 必须定义图标使用时支持的视觉自定义契约，包括尺寸、颜色角色以及附加样式钩子的使用方式。
- **FR-005**：图标 package 必须保证同一共享目录中的图标在视觉上足够一致，能够在同一设计系统中并排使用。
- **FR-006**：图标 package 必须定义并文档化不受支持、缺失或已废弃图标引用的处理行为，使消费者得到可预测结果。
- **FR-007**：系统必须定义图标 package 的公开 API 面，包括命名、variants、用于目录查找的泛型图标入口，以及与同一目录对应的官方命名导出。
- **FR-007a**：泛型 `Icon` 组件必须将 `name`、`className`、`style`、`label` 和 `size` 作为其核心公开 props。
- **FR-007b**：泛型 `Icon` 组件不得额外暴露单独的 `decorative` prop；图标的无障碍行为应由是否提供 `label` 决定。
- **FR-007c**：`size` prop 必须同时支持数字自定义尺寸和标准尺寸 `extra-small`、`small`、`medium`、`large` 与 `extra-large`，其中 `medium` 为默认尺寸。
- **FR-007d**：所有公开命名图标导出都必须遵循 `XxxIcon` 命名规范。
- **FR-008**：系统必须满足明确的无障碍要求，包括图标出现在交互控件中时的语义结构、键盘交互以及有意义图标的辅助技术标签。
- **FR-009**：系统必须说明图标尺寸、颜色角色和对齐所需的新设计 token、主题 hook 或视觉 primitives 是新增、复用，还是明确不需要。
- **FR-010**：系统必须为图标 package 提供预览覆盖，展示支持目录、代表性使用示例，以及无标签使用、有标签使用和缺失图标处理等边界情况。
- **FR-011**：系统必须提供面向贡献者的图标新增、评审和命名指导，以保证目录长期可维护。
- **FR-012**：系统必须保留已发布图标的稳定消费者契约；若对已发布图标进行移除或重命名，必须在发布前明确记录。
- **FR-013**：系统必须对官方受支持目录之外的图标引用给出清晰且文档化的错误，而不是静默替换为占位图标或空输出。
- **FR-014**：系统必须保证泛型图标入口与官方命名导出的行为保持一致，即同一个图标名无论通过哪种方式消费，都应解析到相同视觉资产和相同无障碍契约。

### 无障碍与 UI 契约（必填）

- **用户角色 / 参与方**：消费该 package 的应用开发者、在控件中组合图标的组件开发者、评审图标选择的设计师、仅键盘用户、屏幕阅读器用户、维护目录的 package 维护者
- **交互模型**：从官方目录选择图标，通过泛型 `Icon` 入口或官方命名导出渲染；通过 `label` 派生装饰性与有意义图标行为；在预览面评审使用示例；对不支持图标名做可预测处理
- **需要覆盖的状态**：目录视图、无标签单图示例、有标签有意义图标示例、标准尺寸示例、数字自定义尺寸示例、theme 变化、不受支持或已废弃图标请求及其明确失败行为
- **主题 / token 影响**：图标应尽可能复用现有语义尺寸与颜色角色；任何新 token 或角色都必须证明其是目录级需求，而非单个图标例外

### 关键实体

- **图标目录项**：一个受支持的图标定义，包含稳定名称、视觉语义以及用于发现和治理的贡献者侧元数据。
- **图标使用契约**：定义消费者如何选择、定尺寸、着色、添加标签以及为图标附加样式的规则集合。
- **图标尺寸 Token**：package 暴露的标准图标尺寸名称，同时保留数字自定义尺寸能力。
- **预览示例**：一个可评审的示例，用于展示单个图标或图标组在代表性上下文中的表现，包括无障碍要求和边界行为。

## 假设

- 图标 package 的第一个版本重点在于建立可复用的共享目录与使用契约，而不是覆盖未来产品功能可能需要的全部图标。
- 第一个版本优先建立共享基础图标集，而非一开始就做到全面或按具体业务功能覆盖。
- 图标图稿可以来自外部整理好的来源，只要其许可、视觉一致性和贡献者工作流要求都满足。
- 本功能只覆盖 UI 库内部的独立图标交付与预览，不包括为接入共享图标而重设计无关组件 API。
- 任意第三方图标名不属于该 package 对外公开契约的范围。
- package 同时支持泛型图标渲染器与官方命名导出，但两者都必须由同一份规范目录支撑。
- 不受支持的图标引用被视作契约错误，而不是可恢复的视觉回退。

## 成功标准（必填）

### 可度量结果

- **SC-001**：贡献者可在 5 分钟内通过库的预览和 package 文档找到目标图标及其使用规则。
- **SC-002**：初始共享目录中的 100% 图标都具备唯一稳定名称、至少一个预览示例，以及文档化的无障碍指导。
- **SC-003**：被选入评审范围的代表性图标使用场景中，100% 都在功能完成前同时覆盖了无标签和有标签使用方式。
- **SC-004**：消费者能在代表性的组件和应用示例中使用共享图标 package，而无需导入原始图标文件或自行封装一次性 wrapper。
- **SC-005**：维护者可以在一次评审中，仅通过已发布指导而非阅读实现代码，就判断出不支持、缺失或废弃图标引用会得到什么结果。
