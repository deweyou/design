# 研究：建立统一颜色 token 体系

## 决策 1：把当前色卡从 `Text` 专用能力提升为 `styles` 的共享基础色卡

- **决策**：现有 26 色族 x 11 色阶的能力不再被定义为“只服务 `Text` 的专用色卡”，而是升级为 `packages/styles` 中的共享基础色卡，供 `Text`、`Button` 和后续组件共同复用。
- **理由**：用户的目标是让所有依赖颜色 token 的组件都回到同一来源，而不是继续围绕单个组件扩张专用 token。把色卡放回 `styles` 的基础层，才能符合宪章中的 token-first 原则和 package-first 边界。
- **备选方案**：
  - 继续保持 `Text` 专用色卡，只让其他组件按需复制：拒绝，原因是会延续颜色孤岛。
  - 把共享色卡只放在 Storybook 或 website 示例里：拒绝，原因是会让展示层变成事实来源。

## 决策 2：共享颜色体系拆分为“基础色卡 + 纯黑白 + 语义主题色”三层

- **决策**：颜色体系采用三层模型：基础层是 26 个颜色家族与 11 个色阶；基础补充层是纯黑与纯白；消费层是品牌色、危险色、链接色、焦点色以及 `Text` 高亮变量这类语义主题色。
- **理由**：共享基础色卡解决“颜色从哪里来”，纯黑白解决“不属于任意色族但必须统一维护的基础色”，语义主题色解决“组件实际如何稳定消费”。三层模型能同时满足系统性和组件接入便利性。
- **备选方案**：
  - 只暴露基础色卡，不维护语义主题色：拒绝，原因是组件会重复做映射判断。
  - 只保留语义主题色，不公开基础色卡：拒绝，原因是无法满足统一色卡治理和后续扩展需求。

## 决策 3：新增通用导出命名，但保留现有 text-specific 命名作为兼容 alias

- **决策**：`packages/styles` 的 TypeScript 导出与内部命名将以通用颜色词汇为主，同时保留当前 `textColorFamilyNames`、`textPaletteStepNames` 等 text-specific 命名作为兼容 alias，避免不必要 breaking rename。
- **理由**：当前能力已经被 `Text` 使用，直接重命名会制造对消费方和已有测试的不必要破坏；但如果完全保留 text-specific 命名，又无法表达“这是全局共享色卡”。additive alias 是当前最稳妥的过渡策略。
- **备选方案**：
  - 立即删除 text-specific 命名并全量改名：拒绝，原因是收益低于迁移成本。
  - 永久只保留 text-specific 命名：拒绝，原因是会继续误导后续组件把共享色卡理解为 `Text` 私有能力。

## 决策 4：现有语义变量保持稳定，但底层必须能追溯到共享色卡或纯黑白

- **决策**：现有 `--ui-color-brand-*`、`--ui-color-danger-*`、`--ui-color-link`、`--ui-color-focus-ring`、`--ui-text-color-*`、`--ui-text-background-*` 等语义变量继续保留为稳定消费面，但它们的值必须统一来自共享基础色卡或纯黑白。
- **理由**：`Button`、`Text` 和现有示例已经基于这组语义变量工作。保留命名稳定性可以降低回归风险；同时把来源统一收敛，可以满足新功能的治理目标。
- **备选方案**：
  - 强制所有组件直接消费 `red-500` 一类基础色阶：拒绝，原因是会把语义映射责任推回组件层。
  - 继续允许语义变量独立维护、不要求追溯来源：拒绝，原因是无法真正完成统一治理。

## 决策 5：非必要不得新增特化 token，例外申请必须文档化

- **决策**：本期明确建立治理规则：如果共享基础色卡和现有语义主题色已经能表达需求，就禁止新增特化 token；只有在现有体系明确不足时，才允许提出例外，并记录必要性、适用范围和为什么现有体系失败。
- **理由**：这是用户明确提出的规则，也是避免 token 面无序膨胀的关键。把规则写进规格、计划和 README，后续组件才能在评审时有统一判断标准。
- **备选方案**：
  - 保持“按需新增，事后整理”：拒绝，原因是会继续累积历史债务。
  - 完全禁止任何新增 token：拒绝，原因是会让确有必要的扩展失去入口。

## 决策 6：`Button` 与 `Text` 继续优先消费语义层，而不是直接依赖基础色阶编号

- **决策**：`Button` 继续优先消费品牌色、危险色、链接色、焦点色等语义主题色；`Text` 继续消费 `--ui-text-color-*` 与 `--ui-text-background-*` 这类语义变量；基础色卡主要作为统一来源、文档展示与必要映射输入存在。
- **理由**：组件关心的是“语义稳定”和“接入简单”，不是每次自己选择某个具体色阶。保持组件通过语义层接入，既能统一来源，又能避免组件实现直接耦合 `50`-`950` 编号。
- **备选方案**：
  - 让 `Button` / `Text` 直接消费基础色卡编号：拒绝，原因是会泄露主题映射细节并增加维护成本。
  - 继续让每个组件维护自己的内部颜色常量：拒绝，原因是与统一来源目标冲突。

## 决策 7：Storybook 新增专门的色卡 story，website 提供精简公开指导

- **决策**：`apps/storybook` 新增独立的 `Color.stories.tsx` 作为内部评审矩阵，集中展示完整色卡、纯黑白、语义主题色与代表性组件消费关系；`apps/website` 仅补充面向消费方的精简颜色使用指导，不重复完整矩阵。
- **理由**：这与仓库既有“Storybook 负责内部评审、website 负责公开指导”的分工一致，也满足宪章对测试与预览双门禁的要求。用户明确要求 Storybook 色卡 story，因此 Storybook 是本期主评审面。
- **备选方案**：
  - 只更新 Storybook：拒绝，原因是缺少 website 侧公开指导更新，不符合宪章。
  - 只更新 website：拒绝，原因是 Storybook 更适合做完整矩阵和主题切换评审。

## 决策 8：验证以 `styles` 输出完整性、组件消费回归和双 app 构建为门禁

- **决策**：最小验证集为 `vp check`、`vp test`、`vp run styles#build`、`vp run components#build`、`vp run storybook#build`、`vp run website#build`，并补齐 `packages/styles/tests/theme-outputs.test.ts`、`Button` / `Text` 相关测试与 Storybook / website 示例更新。
- **理由**：本期变更横跨基础 token 层、组件消费层和两个展示面。只做 `styles` 测试无法证明组件没有跑偏，只做 app 预览又无法证明 token 输出完整。
- **备选方案**：
  - 只跑 `vp test`：拒绝，原因是无法覆盖所有 build / preview 入口。
  - 只验证 Storybook：拒绝，原因是 website 也承担公开指导职责。
