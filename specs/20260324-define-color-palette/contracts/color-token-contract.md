# 契约：统一颜色 token 体系

## Package 目的

`@deweyou-ui/styles` 中的颜色体系负责为 Deweyou UI 提供统一的颜色事实来源，并为 `@deweyou-ui/components`、Storybook 和 website 提供稳定的消费 surface。该契约定义共享基础色卡、基础黑白、语义主题色、组件消费边界，以及预览与治理要求。

## 公开 API 变化

- `@deweyou-ui/styles` 必须新增面向通用颜色体系的公开导出与文档说明，使共享色卡不再只以 `Text` 专用语义被理解。
- 现有品牌色、危险色、链接色、焦点色以及 `Text` 相关语义变量必须继续保留为稳定消费面，但都要能够追溯到共享基础色卡或基础黑白。
- 对 `@deweyou-ui/styles` 来说，这属于 additive public API change，预期为 minor。
- `@deweyou-ui/components` 中的 `Button` 与 `Text` 本期不新增颜色相关公开 props；其变化是消费来源统一收敛，而不是 API 扩张。

## 共享基础色卡契约

- 共享基础色卡必须覆盖 26 个颜色家族。
- 每个颜色家族必须完整提供 11 个色阶，覆盖 `50` 到 `950`。
- 色卡命名和排序必须保持稳定，以便设计师、开发者、Storybook 和 website 共享同一认知模型。
- 共享基础色卡是颜色相关能力的唯一标准来源，不允许在组件或 app 中维护平行色卡。

## 基础黑白契约

- 系统必须提供纯黑与纯白两个独立基础颜色 token。
- 纯黑与纯白不能被视为任意颜色家族的变体，也不能在组件局部重新写死。
- 当语义色或组件场景需要纯黑 / 纯白时，必须优先复用这两个基础 token。

## 语义主题色契约

- 语义主题色必须包括但不限于：
  - 品牌背景、品牌 hover、品牌 active、品牌背景上的文字
  - 危险背景、危险 hover、危险 active、危险文字、危险背景上的文字
  - 链接强调
  - 焦点强调
  - `Text` 高亮前景与高亮背景语义变量
- 每个语义主题色都必须明确追溯到共享基础色卡或基础黑白。
- 浅色 / 深色主题切换后，语义主题色必须保持语义连续，不能漂移到无关颜色家族。

## 组件消费契约

- `Button` 必须继续优先消费品牌色、危险色、链接色、焦点色等语义主题色，而不是直接依赖原始色阶编号。
- `Text` 必须继续通过稳定的前景 / 背景语义变量消费共享色卡映射结果，而不是把色阶编号暴露到公开 API。
- 后续新增组件默认必须先从现有共享基础色卡和语义主题色中选择来源。
- 当现有体系足以满足需求时，组件不得新增专属颜色 token 或来源不明的私有颜色常量。

## 特化 token 治理契约

- 非必要不得新增特化 token。
- 若要申请新增特化 token，必须同时记录：
  - 为什么共享基础色卡不足
  - 为什么现有语义主题色不足
  - 预期适用范围
  - 为什么这是长期而非一次性需求
- 无法证明必要性的申请必须被驳回。

## 预览与文档契约

- `apps/storybook` 必须提供专门的颜色 story，集中展示：
  - 完整共享基础色卡
  - 纯黑与纯白
  - 现有语义主题色
  - `Button`、`Text` 等代表性消费关系
  - 浅色 / 深色主题切换结果
- `apps/website` 必须提供精简的公开颜色指导，说明推荐复用路径与“非必要不新增特化 token”的边界。
- website 不能成为颜色行为和治理规则的唯一事实来源；共享规则必须先定义在 package 与规格文档中。

## 无障碍与主题切换契约

- 主题切换后，颜色相关的正文、背景、链接和焦点状态必须继续可读、可辨识、可聚焦。
- 颜色体系收敛不得破坏现有组件的键盘焦点可见性和内容可读性。
- Storybook 颜色 story 必须支持在浅色 / 深色主题下评审相同语义角色的连续性。
