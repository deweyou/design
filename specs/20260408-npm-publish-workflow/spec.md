## 背景与现状

项目目前没有发包工作流：所有包版本固定在 `0.0.0`，无 changelog 工具，无版本管理机制，无 CI 发布流水线。`packages/infra/scripts/write-published-manifest.mjs` 已实现构建产物的 `package.json` 重写（workspace 依赖解析、dev 依赖裁剪），发包基础设施部分就绪，但需要完整的版本与发布流程来配合。vite-plus 本身不提供 publish 或 release 命令，需要引入专门的发版工具。

## 用户场景与测试（必填）

### 用户故事 1 — 在开发分支发布 beta 版本（优先级：P1）

维护者在 feature 分支上完成一个阶段性开发后，希望将其发布为 beta 预发布包（如 `1.0.0-beta.1`），供外部测试或内部预览使用，但不影响正式 npm `latest` tag。

**为什么是这个优先级**：这是整个发包流程的起点。只有先跑通预发布流程，才能安全地建立正式发布流程；同时也是最常用的发布场景（开发中持续发布可测试版本）。

**独立测试**：可以在 feature 分支上手动触发发布，验证包出现在 npm 的 `beta` dist-tag 下，版本号符合 prerelease 格式，正式 `latest` tag 不受影响。

**验收场景**：

1. **假如**维护者处于非 `main` 分支，**当**触发发布动作，**那么**发布的包版本号包含 prerelease 标识（如 `-beta.N`），且发布到 `beta` dist-tag。
2. **假如**维护者处于 `main` 分支，**当**触发 beta 发布动作，**那么**系统拒绝发布并提示错误（main 分支只能发正式版）。
3. **假如**同一 beta 基础版本已存在（如 `1.0.0-beta.1`），**当**再次触发发布，**那么** prerelease 序号自动递增（如 `1.0.0-beta.2`）。

---

### 用户故事 2 — 在 main 分支发布正式版本（优先级：P1）

维护者将 feature 分支合入 `main` 后，希望发布一个语义化版本的正式包（如 `1.0.0`），自动更新 `latest` dist-tag，供下游消费者使用。

**为什么是这个优先级**：与用户故事 1 同等重要，构成完整的双轨发布模型（beta → stable）。

**独立测试**：在 `main` 分支上触发发布，验证包出现在 npm `latest` tag 下，版本号为纯语义化版本，无 prerelease 后缀。

**验收场景**：

1. **假如**维护者处于 `main` 分支，**当**触发正式发布，**那么**发布的版本号为纯 semver（如 `1.2.0`），并更新 `latest` tag。
2. **假如**维护者处于非 `main` 分支，**当**触发正式发布，**那么**系统拒绝并提示"正式版只能从 main 分支发布"。
3. **假如**多个包有变更，**当**发布时，**那么**只有真正有变更的包版本被升级，其余包保持不变。

---

### 用户故事 3 — 自动生成 changelog（优先级：P2）

每次发布时，工作流自动根据 commit 历史生成或更新 `CHANGELOG.md`，记录本次版本的变更内容（新功能、修复、破坏性变更等）。

**为什么是这个优先级**：changelog 提升了版本可追溯性，但不阻塞发包核心流程，可在基础流程就绪后迭代引入。

**独立测试**：发布一个版本后，检查根目录或各包目录下的 `CHANGELOG.md` 是否新增了该版本的条目，且内容与 commit 记录一致。

**验收场景**：

1. **假如**发布了新版本，**当**查看 `CHANGELOG.md`，**那么**包含该版本号对应的变更条目，按 `feat` / `fix` / `BREAKING CHANGE` 分组。
2. **假如** commit message 不符合约定式提交格式，**当**生成 changelog，**那么**该 commit 被归入"其他变更"或跳过，不中断发布流程。

---

### 用户故事 4 — 版本号由 commit 内容自动推断（优先级：P2）

发版时，工作流根据自上次发布以来的 commit 类型（`feat` → minor、`fix` → patch、`BREAKING CHANGE` → major）自动推断版本号增量，无需维护者手动指定。

**为什么是这个优先级**：自动推断减少人为失误，但可在人工指定版本的基础版本流程就绪后再引入。

**独立测试**：在只有 `fix` commit 的场景触发发布，验证版本号只升 patch；在包含 `feat` commit 的场景触发，验证升 minor。

**验收场景**：

1. **假如**仅有 `fix` 类型的新 commit，**当**触发发布，**那么**版本号 patch 位 +1。
2. **假如**包含至少一个 `feat` 类型的新 commit，**当**触发发布，**那么**版本号 minor 位 +1，patch 位归零。
3. **假如**包含 `BREAKING CHANGE` 标记，**当**触发发布，**那么**版本号 major 位 +1。

---

### 边界情况

- 当 npm 鉴权 token 未配置或已过期时，发布失败应有明确提示，不静默失败。
- 当构建产物不存在时（忘记先 build），发布前应校验并中止，给出清晰错误信息。
- monorepo 中部分包发布失败时，其余包的发布状态需明确（已发布的不回滚，失败的需有重试路径）。
- prerelease 版本不得覆盖已发布的正式版本（版本号防倒退）。

## 需求（必填）

### 功能需求

- **FR-001**：系统必须支持从非 `main` 分支发布 prerelease 包，版本格式为 `X.Y.Z-beta.N`，发布到 npm `beta` dist-tag，不更新 `latest`。
- **FR-002**：系统必须仅允许从 `main` 分支发布正式版本，并更新 npm `latest` dist-tag。
- **FR-003**：系统必须在发布前校验当前分支，防止在错误分支发布错误类型的包。
- **FR-004**：系统必须在发布前完成全量构建，确保 `dist/` 产物与源码同步。
- **FR-005**：系统必须支持 monorepo 中多个包的联动版本发布，只升级有实际变更的包。
- **FR-006**：系统必须提供可在本地手动触发的发布命令，同时支持通过 CI 自动触发。
- **FR-007**：系统必须自动生成或更新 `CHANGELOG.md`，内容基于约定式提交（Conventional Commits）格式。
- **FR-008**：系统必须根据 commit 类型自动推断版本号增量（`fix` → patch、`feat` → minor、`BREAKING CHANGE` → major）。
- **FR-009**：发布过程必须校验 npm 鉴权状态，鉴权失败时给出明确错误提示。
- **FR-010**：系统必须在发布完成后输出本次发布的包名、版本号和 npm dist-tag 的摘要。

### 关键实体

- **发布通道（Channel）**：区分 `beta`（预发布，非 main 分支）和 `stable`（正式，main 分支）两种通道，决定 dist-tag 和版本格式。
- **版本增量（Bump）**：根据 commit 类型推断的语义化版本升量（patch / minor / major），适用于正式发布；beta 发布额外附加 `-beta.N` 序号。
- **变更集（Changeset）**：自上次发布以来有代码变更的包集合，决定哪些包需要升级版本并重新发布。
- **发布产物（Release Artifact）**：构建后的 `dist/` 目录，包含经 `write-published-manifest.mjs` 处理的 `package.json`。

## 成功标准（必填）

### 可度量结果

- **SC-001**：维护者可在 5 分钟内完成一次 beta 包的发布，从触发命令到 npm 上可查询到包。
- **SC-002**：维护者可在 10 分钟内完成一次正式包的发布（含全量构建），从触发到 `latest` tag 更新。
- **SC-003**：发布流程中不存在需要手动编辑版本号的步骤（版本由工具自动推断或自动递增）。
- **SC-004**：每次发布后均可在 `CHANGELOG.md` 中找到对应版本条目，准确率 100%。
- **SC-005**：beta 包与正式包的分支保护规则经过验证，无法通过单一命令绕过（分支校验有效）。

## 假设与约束

- 项目已使用约定式提交格式（`.vite-hooks/commit-msg` 强制校验），changelog 生成可直接依赖此基础。
- 发包工具的选型需与 pnpm workspace + vite-plus 构建流程兼容，不引入与现有基础设施冲突的工具。
- `packages/infra` 不发布到 npm，发包流程只覆盖 `packages/react`、`packages/react-hooks`、`packages/react-icons`、`packages/styles`、`packages/utils`。
- 版本号初始值需要在实现时确定（当前均为 `0.0.0`），本 spec 不约束具体起始版本。
